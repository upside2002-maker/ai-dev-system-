## 1. Что строим

Машинный реестр знаний системы — три дописываемых файла + карантин + единый
вход `aida` + валидатор, подключённый к `make check`. Это средний ярус памяти
из стратегии §6: между «рабочими досками» и «конституцией». Доски (DISPATCHER,
STATUS_RU) НЕ трогаем — рендер из реестра будет отдельным шагом (v0.5).

## 2. Формат хранения

Каталог `ledger/`, формат JSONL (одна запись = одна строка JSON, кодировка
UTF-8, русский в значениях полей — норма):

| Файл | Что лежит | Кто пишет |
|---|---|---|
| `ledger/inbox.jsonl` | карантин: сырые заметки/сигналы без источника (И-4) | любая роль, свободно |
| `ledger/facts.jsonl` | факты СО ИСТОЧНИКОМ, статус verified/accepted/stale/superseded (И-5) | только через `aida` |
| `ledger/decisions.jsonl` | решения Owner'а/ролей: выбор, не факт о мире | только через `aida` |
| `ledger/contradictions.jsonl` | противоречия: обе стороны + рабочая версия (И-14) | только через `aida` |

**Дописываемость (И-14):** записи никогда не редактируются и не удаляются.
Изменение состояния = новая запись со ссылкой `supersedes` (для фактов) или
`status`-запись (см. схемы). История — в git.

## 3. Схемы записей (`schemas/*.schema.json`)

Хранить как честные JSON-схемы (для документации и будущих валидаторов);
валидатор v0 проверяет их подмножество кодом (см. §5). Все обязательные поля
ниже — enum-значения дословно.

**fact** (`schemas/fact.schema.json`):
- `id` — `F-YYYYMMDD-NNN` (NNN — порядковый за день, уникален в файле)
- `statement` — одно предложение, русский
- `scope` — `system | global | sitka-office | astro | aida-voice | crypto`
- `status` — `verified | accepted | stale | superseded`
- `confidence` — `low | medium | high`
- `source` — объект, ОБЯЗАТЕЛЕН: `{kind: command|file|git|url|human|handoff,
  ref: строка (команда/путь/адрес/кто), quote: строка-выжимка (опционально)}`
- `checked_at` — `YYYY-MM-DD`
- `expires` — `YYYY-MM-DD` | `event:<описание>` | `null` (И-13)
- `supersedes` — id факта | `null`
- `origin` — id записи inbox | `null` (если факт родился из карантина)
- `recorded_by`, `recorded_at`

**decision** (`schemas/decision.schema.json`):
- `id` — `D-YYYYMMDD-NNN`
- `decision` — что решено, одним-двумя предложениями
- `decided_by` — `owner | admin | tl-sitka | tl-astro`
- `decided_at` — `YYYY-MM-DD`
- `scope` — как у факта
- `basis` — массив строк (id фактов и/или свободный текст «надиктовка 11.06»)
- `review_when` — условие пересмотра | `null`
- `status` — `active | superseded | revoked`
- `supersedes` — id решения | `null`
- `recorded_by`, `recorded_at`

**contradiction** (`schemas/contradiction.schema.json`):
- `id` — `C-YYYYMMDD-NNN`
- `a`, `b` — id конфликтующих фактов (оба обязаны существовать)
- `working` — id рабочей версии (равен `a` или `b`)
- `resolves_when` — что разрешит спор
- `status` — `open | resolved`; `resolved_by` — id факта/решения | `null`

**inbox** (`schemas/inbox.schema.json`):
- `id` — `I-YYYYMMDD-NNN`
- `note` — свободный текст
- `context` — откуда взялось (опционально)
- `status` — `raw | promoted | dropped`; `promoted_to` — id факта | `null`
- `recorded_by`, `recorded_at`

## 4. Вход `aida` (`scripts/aida`)

Bash, стиль соседних скриптов (`set -euo pipefail`, русские сообщения с «как
исправить»), подкоманды:

- `aida inbox "<note>" [--context "..."]` — запись в карантин.
- `aida fact "<statement>" --scope X --source-kind K --source-ref R
  [--quote "..."] [--confidence high] [--expires ...] [--origin I-...]
  [--supersedes F-...]` — новый факт со статусом `verified`.
  **Без `--source-kind`+`--source-ref` — отказ** с текстом «факт без
  источника — не факт (И-5); сырое клади в карантин: aida inbox ...».
- `aida promote I-NNN ... (те же флаги источника)` — факт из карантина:
  пишет факт с `origin`, дописывает inbox-запись со `status: promoted`
  (новой строкой, старую не трогает).
- `aida accept F-NNN` — повышение verified→accepted (новая запись-копия со
  `supersedes` старого id и `status: accepted`).
- `aida stale F-NNN --reason "..."` — пометка протухшим (аналогично).
- `aida decision "<text>" --by owner --scope X [--basis "..."]...
  [--review-when "..."] [--supersedes D-...]` — новое решение.
- `aida contradict F-a F-b --working F-a --resolves-when "..."` — противоречие.
- `aida show [facts|decisions|inbox|contradictions] [--scope X] [--status S]`
  — человекочитаемая выдача (таблица/список, не сырой JSON).
- `aida check` — валидатор (§5).

Генерация id: по дате + max существующий номер за день + 1 (читать файл).
Часть логики удобнее на python3 — разрешено: `scripts/aida` (bash-обёртка) +
`scripts/aida_ledger.py` (вся работа с JSONL; только стандартная библиотека,
никаких pip-зависимостей).

## 5. Валидатор (`aida check`, и цель в Makefile)

Проверяет все четыре файла:
1. Каждая строка — валидный JSON; обязательные поля присутствуют; enum-поля —
   из допустимых значений; id соответствует шаблону и уникален в своём файле.
2. У каждого факта есть `source.kind` и `source.ref` (И-5).
3. Ссылки существуют: `supersedes`, `origin`, `a/b/working/resolved_by`,
   `promoted_to` указывают на реальные id (в т.ч. поперёк файлов, где уместно).
4. `working` противоречия равен `a` или `b`.
5. Протухшие факты (`expires` — дата в прошлом, статус не stale/superseded) —
   ПРЕДУПРЕЖДЕНИЕ списком (И-13), не отказ.
6. Открытые противоречия — информационный список.
Выход: 0 если нет ошибок (предупреждения допустимы), 1 с перечнем ошибок.
Подключить в `Makefile`: цель `ledger-check`, включить её в `check`.
Пустые/отсутствующие файлы реестра — норма (система только заводится).

## 8. Приёмка

1. `aida fact` без источника → отказ с подсказкой про карантин.
2. `aida fact` с источником → строка в facts.jsonl, `make check` зелёный.
3. `aida inbox` → запись; `aida promote` с источником → факт с origin, inbox
   дописан статусом promoted (старая строка не изменена — проверить git diff).
4. Руками испортить копию строки (битый JSON / неверный enum / дубль id /
   supersedes в никуда) → `aida check` ловит каждое, с номером строки.
5. `aida accept`, `aida stale` — новая запись, старая не тронута.
6. `aida contradict` с несуществующим id → отказ; с working ≠ a/b → отказ.
7. `aida show decisions --scope crypto` — читаемо.
8. Образцы §6 занесены, `make check` зелёный, протухших ошибок нет.
9. shellcheck -S info чисто; python3 -m py_compile чисто.
