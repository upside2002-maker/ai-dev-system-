# Макро-режим как шестая семья решения долгого спота (CPDS) — дизайн + критик

**Дата:** 2026-06-14. Источник: прогон `macro-family-longspot-design`.
**Основание:** решение Owner 2026-06-14 (D — «макро обязательно в долгом споте»).
**Вердикт критика:** дизайн ГОДЕН С ПРАВКАМИ (архитектура верная, к деньгам — только после блокеров).

## Суть дизайна (рекомендованный Вариант А)

Макро входит в денежное решение как **шестая детерминированная семья** снимка,
структурно как пять существующих (зона cheap..extreme, уверенность, сырьё, источники).
- **Сигналы** — только числовые ряды FRED (реальная ставка, чистая ликвидность ФРС,
  спред SOFR−IORB, кредитные спреды BBB/HY, правило Сама, индекс доллара, керри-спред,
  глобальная М2). Каждый → перцентиль СВОЕЙ истории → флаг risk-off / risk-on.
- **Зона режима** = доля сработавших risk-off среди живых сигналов (каскадный
  мажоритарный счёт, НЕ одиночный переключатель) + гистерезис против дребезга.
- **Связь с решением:** risk-off → раньше/крупнее фиксация; risk-on на просадке →
  охотнее накопление; risk-off на просадке → осторожнее (зеркальные множители зон,
  новой денежной логики не добавляем).
- **Некритична:** нет данных/протухло → нейтрально, деньги НЕ блокируются (критична
  только он-чейн оценка). LLM в денежном пути нет (И-8 цела); вся арифметика на целых.

## Честные ограничения (критик — обязательно знать)

1. **Доказать выгоду на самом CPDS на истории СЕЙЧАС нельзя.** На исторических днях
   он-чейн оценка молчит → ядро в наблюдении → «с макро / без макро» даёт одинаковый
   ноль. Поэтому макро сперва валидируется как **отдельная стратегия-фактор без
   он-чейн** на полной истории FRED, и лишь потом вес переносится в CPDS — когда
   накопятся живые снимки с одновременно живыми оценкой и макро. Это проксированное
   доказательство, не прямое.
2. **Два «заглядывания вперёд» закрыть ДО денег:** (а) ревизии FRED — сейчас сборщик
   берёт последнюю ревизию, нужен режим «значение, известное на дату» (ALFRED/vintage);
   (б) база перцентиля и калибруемые пороги — только данные ≤ даты снимка, через
   калибратор на калибровочном срезе.
3. **Переобучение — риск ВЫСОКИЙ:** макро-циклов в истории крипты мало (2-3),
   свободных параметров ~12 → лёгкая подгонка. Сам корпус противоречив (RU#33:
   собственный 4-летний цикл BTC может перебивать ликвидность). Якорь Damodaran:
   одиночный макро-таймер за 100 лет даёт минус. Вывод: **макро входит как осторожный
   МОДИФИКАТОР с НИЗКИМ стартовым весом**, не как сильный предиктор. Стартовый вес 1.0
   (как у плеча) — слишком много; начать заметно ниже.
4. Штормовые окна валидации НЕ должны совпадать с окнами калибровки (иначе проверка
   идёт на обучающей выборке).

## Блокеры до внесения веса в деньги
- vintage/ALFRED в `collectors/fred.py` + тест «значение на дату T не знает поздних ревизий».
- база перцентиля/калибровка только ≤ T, через калибратор, тест изоляции.
- штормовые окна вне калибровки; накопление истории снимков (сейчас на диске ~2).
- A/B честно помечать: на CPDS ноль; выгода — на факторе-прокси; перенос веса — через
  отчёт о повышении (И-3), пошагово.

## Развилки Owner
1. **Модуляция:** А (6-я семья в сумме, рекомендую) / +В (узкое усиление: макро-экстрим →
   подтверждение на накопление) / Б (режим двигает сам порог — сложнее, риск двойного учёта).
2. **Где валидировать сперва:** отдельный макро-фактор без он-чейн на истории FRED
   (рекомендую — обходит он-чейн дыру) / ждать накопления живых снимков.
3. **Стартовый вес и пороги:** рекомендую низкий осторожный старт, поднимать только бэктестом.
4. Японский ряд для керри-спреда — включать сейчас или отложить (на старте молчит).
5. Тяжёлый прогон бэктеста — под окно лимита, один на окно (дисциплина).

## Путь (последовательность, НЕ один заход)
Контракт (схемы +6-я семья, синхронно Python↔Haskell) → сборщик FRED с vintage →
макро-семья (детерминированно, некритична) → фактор-прокси на истории FRED → бэктест
с защитой от заглядывания/переобучения → отчёт о повышении → пошаговый ввод низкого
веса в CPDS на живых снимках. Денежное ядро — tier A strict, независимое ревью.
