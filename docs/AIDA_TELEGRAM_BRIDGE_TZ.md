## 0. Главный принцип (он же экономика)

Телефон — это **новый транспорт к существующему мозгу чтения хозяйства**
(Phase 2.1, `aida_status.py`), а НЕ второй секретарь. Переиспользовать
`aida_status.read_project_status()` и `read_dispatcher_summary()` напрямую —
они уже защищены белым списком, обрезаны, со словарём синонимов.

**Деньги (решение Owner'а 12.06 — жёсткое).** Чтение статусов = чтение готовых
файлов-снимков, модель тут почти не нужна. Голосовой движок OpenAI Realtime для
текстового телефона НЕ используется (привязан к речи и дорог). Мост обязан
стоять на собственной экономике и **НЕ опираться ни на лимиты подписки, ни на
месячный кредит программного режима** (продление Макс20 отключено; остаток
кредита Owner бережёт под Фейбла после 22.06). Дефолт — бесплатный шаблон из
снимков; опция — дешёвая модель (Хайку) через СВОЙ API-ключ, копейки за вызов.
Бот НЕ должен подсовывать OAuth-профиль подписки (запрет условий — см.
`ledger/facts.jsonl`, факты о биллинге 15.06 и OAuth).

## 1. Что строит v1

Новый модуль `telegram_bridge.py` (+ при нужде тонкий `aida_processor.py`):
1. **Транспорт:** Телеграм-бот (`python-telegram-bot`, async — стек проекта уже
   asyncio). Токен — из окружения `TELEGRAM_BOT_TOKEN` или `config.py`
   (gitignored), НИКОГДА в код и логи.
2. **Доступ только Owner'у:** бот отвечает только на сообщения с
   `TELEGRAM_OWNER_ID` (числовой id Owner'а в Телеграме); чужим — молчание или
   «не для вас». Это замок: секретарь хозяйства не публичный.
3. **Текстовый вход:** сообщение → распознать намерение → вызвать нужное умение
   чтения → вернуть ответ операторским языком, кратко (2–4 фразы), по-русски.
4. **Голосовой вход (приём):** голосовое Телеграма → распознать в текст
   (OpenAI-транскрипция тем же ключом, что голос; дёшево) → дальше как текст.
   Ответ — текстом (озвучку ответа в v1 НЕ делаем — экономия и простота).
5. **Умения v1 — РОВНО те же два (белый список, без записи/запуска):**
   - «что по Ситке / Астро / голосу» → `read_project_status(project)`;
   - «что по хозяйству / что вообще / общий статус» → `read_dispatcher_summary()`.
   Никаких других инструментов. Неизвестный проект / запрос вне умений →
   честный отказ, не импровизация.

## 2. Маршрутизация намерения (где экономия)

Сначала — **детерминированно**, без модели: нормализовать текст, прогнать через
тот же `_SYNONYMS`/`resolve_project` из `aida_status.py`; если в запросе узнаётся
проект — звать `read_project_status`; если общие слова («хозяйство», «все»,
«что происходит») — `read_dispatcher_summary`. Это покрывает большинство и стоит
ноль.

**Формулировка ответа из снимка** — переключатель `TELEGRAM_REPLY_STYLE`:
- `template` (дефолт на старте, бесплатно): отдать приоритетные секции снимка
  как есть, слегка причесав (заголовок + выжимка). Снимки уже на операторском
  языке и обрезаны.
- `model` (опция): прогнать снимок через ДЕШЁВУЮ модель (Haiku через свой
  API-ключ) для живой краткой формулировки. Включается, когда Owner захочет
  «поживее»; копейки за вызов.
Не зашивать дорогую модель. Не использовать OAuth подписки.

## 5. Приёмка

1. Бот стартует с токеном из окружения; без токена — внятный отказ, не падение.
2. Текст «что по Ситке» с Owner-id → ответ из снимка Ситки, операторским
   языком, кратко; в логе — `[инструменты] get_project_status(sitka)`.
3. «что по хозяйству» → выжимка доски (активные роли + открытые блоки).
4. Запрос вне умений / неизвестный проект → честный отказ, не выдумка.
5. Сообщение НЕ от Owner-id → бот не отвечает по существу.
6. Голосовое Телеграма → распознано в текст → тот же путь (если §1.4 в объёме
   v1; если Owner отложит голосовые — этот пункт переносится).
7. Произвольный путь/инъекция в запросе (`../`, имя чужого файла) → та же защита
   отказывает (прогнать 2–3 атаки из набора Phase 2.1).
8. Ключи/токены не светятся в логах и ответах.
9. `python3 -m py_compile` чисто; офлайн-тест маршрутизации без сети (как в
   проекте — без pytest, свой тест-скрипт) проходит.

Отчёт: файлы, решения по месту, выводы приёмки, что осталось на 2.2. Ничего не
коммитить в git ai-dev-system (мост живёт в aida-voice-bridge, его бэкап — по
