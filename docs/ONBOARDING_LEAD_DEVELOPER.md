# Онбординг — ведущий разработчик

Документ для второго разработчика (Виталий, `bondvit@gmail.com`), когда он первый раз заходит в проекты. Читается за 5-10 минут. Если что-то непонятно — напиши Илье.

## 1. Что должно быть на машине

- **git** — система контроля версий. Проверить: `git --version` в терминале должно показать версию.
- **make** — запуск стандартных команд проекта. Проверить: `make --version`. На macOS обычно ставится вместе с инструментами разработчика (`xcode-select --install`).
- **perl** — встроенный язык, нужен скриптам для безопасной замены строк. Проверить: `perl --version`. На macOS и Linux идёт из коробки.
- **bash** — командная оболочка. Проверить: `bash --version` (нужна версия 4 или новее; на macOS системная — старая, поэтому нужен `brew install bash` чтобы получить свежую).
- **Аккаунт GitHub с доступом к двум приватным репозиториям:**
  - `https://github.com/upside2002-maker/ai-dev-system-.git` (с дефисом в конце — так создан владельцем) — система разработки;
  - репозиторий продуктового проекта (`sitka-office`) — Илья добавит тебя соавтором через настройки GitHub отдельно.

## 2. Что склонировать

Создай у себя папку для проектов и склонируй туда оба репозитория:

```
mkdir -p ~/Projects
cd ~/Projects
git clone https://github.com/upside2002-maker/ai-dev-system-.git ai-dev-system
git clone <url репозитория sitka-office>  # Илья пришлёт точный адрес
```

Имя локальной папки `ai-dev-system` — без дефиса в конце. Дефис только в имени удалённого репозитория на GitHub.

**Раскладка папок обязательная — ровно эти имена и места:**

- `~/Projects/sitka-office` — продуктовый репозиторий CRM
- `~/Projects/ai-dev-system` — методология и скрипты ведения

Это не совет, а требование. Slash-команды (`/sitka-status`, `/sitka-worker`, `/sitka-ba`, `/sitka-review`) и подагенты Claude Code (`sitka-worker`, `sitka-reviewer`, `sitka-business-analyst`) ссылаются на эти пути жёстко — `~/Projects/sitka-office/...` и `~/Projects/ai-dev-system/...`. Без точно такой раскладки они не найдут нужные файлы и упадут с «нет такого файла».

Если у тебя на машине другая раскладка проектов и переезд не вариант — настрой символические ссылки:

```
ln -s /твой/реальный/путь/sitka-office     ~/Projects/sitka-office
ln -s /твой/реальный/путь/ai-dev-system    ~/Projects/ai-dev-system
```

Тогда команды будут видеть нужные пути, а сами репозитории физически останутся там, где у тебя.

## 3. Настроить почтовый адрес для своих коммитов

Открой терминал в каждой склонированной папке и выполни две короткие команды для подписи твоих коммитов:

```
cd ~/Projects/ai-dev-system
git config user.email bondvit@gmail.com
git config user.name "Виталий"
```

Затем то же самое в `sitka-office`:

```
cd ~/Projects/sitka-office
git config user.email bondvit@gmail.com
git config user.name "Виталий"
```

**Почему это важно:** система определяет твою роль по почтовому адресу из коммитов. Если этого не сделать — твои правки будут приходить от чужого имени, защиты не сработают, а в `git log` будет неразбериха. Это локальная настройка для каждой папки, на других проектах твоей машины ничего не меняется.

## 4. Что прочитать первым

В папке `ai-dev-system` запусти команду свежего состояния проекта:

```
cd ~/Projects/ai-dev-system
make context SLUG=sitka-office
```

Команда покажет блок «Кто сейчас ведёт проект», статус, активные задачи, последние изменения. Если ведение проекта свободно — там же будет подсказка как его взять.

Затем прочитай файл `project-overlays/sitka-office/starts/TECH_LEAD.md` — это стартовый промпт для технического ведущего проекта. Там описаны роль, правила, что разрешено и что нельзя.

Для общей картины:
- `ROLE_MODEL.md` — какие роли есть в команде и кто чем владеет.
- `policies/SHIFTS.md` — правила ведения проектов (взять, продлить, сдать, экстренное прерывание).
- `policies/USERS.md` — список разработчиков и их прав.
- `policies/REMOTES.md` — как устроены две удалённые копии (GitHub и локальная резервная).

## 5. Как взять ведение проекта

Когда хочешь начать работать над проектом — берёшь ведение. По умолчанию бессрочно (держишь пока сам не сдашь):

```
make take-shift SLUG=sitka-office SCOPE="что ты планируешь делать одной фразой"
```

Если знаешь что сессия короткая и есть смысл указать срок — добавь `HOURS=N`:

```
make take-shift SLUG=sitka-office SCOPE="доделать UI к вечеру" HOURS=4
```

После взятия ведение записано на тебя. Файл `project-overlays/sitka-office/TL_SHIFT.md` обновится, изменение уйдёт в удалённое хранилище — Илья увидит что ты сейчас работаешь.

Если ты уже ведёшь проект и хочешь обновить описание зоны работы (переключился на другое) — просто вызови `make take-shift` снова с новым `SCOPE`. Скрипт поймёт что это ты же, и обновит поле без отказа.

## 6. Как сдать ведение

Когда закончил период работы — сдаёшь ведение:

```
make release-shift SLUG=sitka-office NOTES="что сделано за этот период, что осталось"
```

`NOTES` обязательно непустое. Это пара фраз для Ильи (или для тебя самого через неделю, когда вернёшься): что было сделано, что осталось открытым, на чём стоит. Лень сдавать с пустым `NOTES` — потом сложно поднять контекст обратно.

Если забудешь сдать — ничего страшного не случится. Бессрочное ведение остаётся за тобой до явной сдачи или экстренного прерывания. Просто Илья будет видеть что ты «ведёшь» проект пока не сдашь.

## 7. Что ты НЕ можешь делать

У тебя в правах (см. `policies/USERS.md`) стоят три `no`:

- **`can_override: no`** — ты не можешь забирать чужое активное ведение силой. Если проект ведёт Илья и ты хочешь начать — договорись с ним или подожди когда сдаст. Команда `OVERRIDE=yes` от твоего адреса будет отвергнута. Это право есть только у владельца.
- **`can_approve_critical: no`** — ты не можешь сам закрывать задачи, которые трогают файлы методологии (правила, политики в `policies/`, шаблоны в `templates/`, скрипты в `scripts/`, поправки в `corrections/`, опоры архитектуры). Если такая задача нужна — ты её создаёшь как обычно через `make new-task`, потом отправляешь Илье на подпись командой `make approve-critical FILE=<путь к задаче>`. Илья запустит подпись от своего адреса, после этого задачу можно принимать. Полный список критичных путей — `policies/CRITICAL_PATHS.md`.
- **`can_deploy: no`** — выкаткой в продакшен ты не занимаешься.

Это не недоверие, а разделение зон. Архитектура и правила остаются за владельцем, повседневная разработка — за тобой.

## 8. Куда смотреть когда непонятно

- `make context SLUG=<проект>` — свежий снимок состояния. Первая команда когда заходишь после паузы.
- `policies/SHIFTS.md` — подробное описание процесса ведения (взять, продлить, сдать, что значит срок истёк).
- `policies/USERS.md` — кто есть кто в команде и какие права у каждого.
- `policies/CRITICAL_PATHS.md` — что считается серьёзной правкой методологии, требующей подписи владельца.
- `policies/REMOTES.md` — как устроены удалённые копии репозитория и почему отправка идёт в обе.
- `policies/OPERATOR_LANGUAGE.md` — список слов, которых не должно быть в твоих сообщениях пользователю продукта (если пишешь текст, который пойдёт оператору-Марине напрямую — например, в передачи между ролями).
- `policies/AUDIT.md` — описание периодической сверки (её запускает Илья раз в 2-3 недели, тебе не нужно делать самому).
- `CLAUDE_GLOBAL.md` — общие правила работы AI-агентов в системе (защита от выдумывания фактов, обращение к пользователю продукта на простом русском, и тому подобное).
