## 1. Суть в одном абзаце

Это не торговый бот и не предсказатель рынка. Это дисциплинированный
распределитель: на подтверждённом перегреве цикла отщипывает малые транши
прибыли из ростовых активов и уводит их по корзинам с растущей
защищённостью; на подтверждённой капитуляции — частично возвращает буфер
в рынок; в терминальную корзину поток односторонний — оттуда возврата нет.
Цель — не максимум доходности, а чтобы заработанное не сгорало в следующем
цикле. Решения редкие (несколько раз в год); в обычный день система молчит.

**Главная рамка (ревью Owner'а 12.06, принято):** это **учётная система
фиксации и распределения чистой прибыли**, а не «умный ребалансировщик».
Журналы и себестоимость — центр ядра, не вспомогательная часть. Логика
ядра всегда идёт цепочкой: есть ли подтверждённая прибыль → можно ли её
частично зафиксировать → куда распределить чистую часть → можно ли позже
вернуть часть буфера в рынок.

## 1-бис. Место CPDS в большой системе (поправка Owner'а 12.06)

CPDS — не вся система, а **первый безопасный боевой модуль** большой
многоуровневой платформы анализа, тестирования и отбора стратегий.
Полная рамка — семь контуров (мандат §4-бис): данные (включая макро,
геополитику, новости; расширяемо вплоть до солнечной активности и фаз
Луны), аналитика 24/7 (LLM классифицирует и сводит, но не торгует),
лаборатория стратегий (любые гипотезы, включая фьючерсные), бэктест-движок,
бумажный движок (у каждой стратегии своё виртуальное депо, автомат на
бумаге разрешён — денег нет), боевое риск-ядро (в MVP — CPDS) и панель.

Принцип платформы: **исследовать можно всё; исполнять — только прошедшее
фильтр** «идея → формализация → бэктест → бумага → сравнение с базой →
стабильность → ручное одобрение → ограниченный боевой вес».

Для первого этапа это значит: строим не «наблюдателя-советчика» в одиночку,
а сразу три вещи — крипто-данные, бумажный движок с виртуальными депо
и CPDS как первого жильца боевого ядра. Гипотезы Г-1 (кэрри), Г-2 (сетка
с опционным прикрытием), Г-3 (тренд) встают в очередь лаборатории
и бумаги, не дожидаясь боевого статуса.

## 2. Рамочные решения Owner'а (12.06)

1. **Универсальность по капиталу.** Капитал любой — сегодня один, завтра
   другой. Всё в долях и параметрах. Из этого следствие для всех чужих
   черновиков: абсолютные пороги в валюте (вида «100–500 тысяч») в ядро
   не попадают — только доли капитала в конфиге профиля.
2. **Риск выбирается профилем:** консервативный / сбалансированный /
   агрессивный. Профиль задаёт размеры траншей, пороги подтверждения,
   **карту распределения прибыли** (консервативный — больше в буфер
   и терминал, меньше обратно в рынок; агрессивный — больше в буфер
   и активнее обратный ход на капитуляции) и доли корзин — в том числе
   долю защитной корзины: не хочешь токен-золото — ставишь ноль.
3. **Перспектива продукта:** оформить как сервис и продавать. Следствие:
   ядро с самого начала без глобального состояния — «конфиг + профиль +
   журналы» на каждого пользователя, один и тот же движок.
4. **Первый этап — только крипта.** Пять активов. Остальные классы
   (акции и т.п.) — заложить в универсальность типов, в работу не брать.
5. **Вырезано прямым словом Owner'а:** налоги, юридические вопросы,
   банковские шлюзы и страновые ограничения — не рассматриваем. Из чужих
   диалогов НЕ переносим: банковское законодательство, страхование
   вкладов, налоговые льготы, локальные облигации и акции, модель
   шлюза вывода в фиат. Терминальная корзина в ядре — просто «выведено
   из системы»; чем её наполнять — личное решение владельца капитала
   за пределами контура.

## 3. Принятые принципы (из диалогов; согласуются с исследованием)

1. **Малые транши.** 1–3% позиции за раз при обычном перегреве, 3–5%
   при сильном; крупнее — только ручное решение. Плавная фиксация
   оставляет позиции «дышать» при параболическом росте.
2. **Запрет ФОМО-перезахода.** Проданное не откупается дороже без
   «сброса оценки». Сброс формален (любое из): просадка ≥25% от пика
   после продажи / возврат он-чейн оценки в нижнюю зону / зафиксированная
   капитуляция после продажи / тайм-аут (порядка 18 месяцев). Параметры —
   в профиле, калибровка — на стенде.
3. **Чистая прибыль, не валовая — с формальной бухгалтерией.** Продаётся
   часть позиции, а не «прибыль»; журнал делит выручку на возврат
   себестоимости (пропорционально лотам) и чистую реализованную прибыль:
   выручка − пропорциональная себестоимость − комиссии/спред = чистая
   прибыль. По корзинам распределяется ТОЛЬКО чистая прибыль; возврат
   себестоимости прибылью не считается. Фиксация — только если чистый
   эффект выше порога профиля. (Налоговую часть не считаем — вырезана;
   для будущего сервиса вернётся опциональным модулем.)
4. **Пять семей скоров вместо одного индикатора:** цена/растяжение,
   он-чейн оценка, импульс, плечо рынка (фандинг, открытый интерес),
   реализация прибыли. Триггер — взвешенная сумма семей выше порога
   (веса в конфиге: оценка — тяжёлая, реализация прибыли — лёгкая).
   Зоны и квантили вместо абсолютных констант — защита от подгонки под
   прошлые циклы. Отсутствие данных формализовано: молчит критичная
   семья (он-чейн оценка) — крупные действия запрещены, только наблюдение;
   молчит некритичная — скор считается с предупреждением и пониженной
   уверенностью. Веса оставшихся семей при пропуске молча не раздуваются.
5. **Журналы — центр ядра (четыре):** журнал лотов (себестоимость),
   журнал зафиксированной прибыли, журнал распределений и журнал
   блокировок перезахода — что продали, по какой цене, когда, какие
   условия сброса уже выполнены; на нём держится запрет ФОМО. Без
   журналов система не отличает «продаю прибыль» от «режу основной
   капитал». Ядро — чистая функция: снимок рынка + четыре журнала +
   профиль-конфиг → решение. Запись журналов и вся внешняя работа —
   снаружи ядра.
6. **Темп.** Решения редкие; в ~90% дней — «наблюдаем». Любое действие
   крупнее ~5% капитала — ручное подтверждение Owner'а.

## 4. Корзины — универсальная форма

| Корзина | Что это | Возврат в рынок |
|---|---|---|
| К0 «жизнь» | резерв на жизнь | вне системы, ядро о ней НЕ ЗНАЕТ |
| К1 ликвидный буфер | стейблы | да, быстро — топливо капитуляционных докупок |
| К1.5 крипто-защита | токен-золото — в составе этапа, доля по профилю (может быть 0) | да; временная разгрузка, НЕ терминал |
| К2 рабочий капитал | в первом этапе — абстрактная внешняя запись в журнале: без доходности, шлюзов и конкретных инструментов | да, медленно/с издержками |
| К3 терминал | выведено из системы | НИКОГДА |

Поток направленный: рост → К1 → К2 → К3. Обратный ход только из К1/К2
в рост и только на подтверждённой капитуляции. К0 ядро не видит вовсе.
По К3 точная формула: ядро МОЖЕТ рекомендовать вывод в терминал, но
не моделирует содержимое К3 и никогда не возвращает его в рынок.

## 5. Первый этап — крипто-наблюдатель

**Активы:** биткойн, эфир, два долларовых стейблкоина, токен-золото
(в составе этапа как защитный; профиль может задать его долю нулём).

**Контур дня:** утренний сбор данных → «снимок дня» одним документом →
ядро считает семьи скоров → решение (наблюдаем / зафиксировать X% /
докупить Y% из буфера / данные плохие — семья не голосует) → значимое
действие = уведомление с ручным подтверждением → всё в журнал.

**Бумажный режим с первого дня** (красная линия мандата §3.4): все
решения и «сделки» виртуальные, статистика копится для приёмки.

### Для исполнителя — источники и метрики

- Бесплатно: Binance API (цены, funding, open interest, объёмы),
  Coinbase (кросс-чек цен), Deribit (опционные данные — копим, в решениях
  не используем), FRED (золото), Farside (потоки спот-ETF), DeFiLlama
  (капитализация стейблкоинов).
- Платно: CryptoQuant Advanced ~$29/мес. Уточнение по тарифу (ревью
  12.06): API на Advanced ЕСТЬ, но для нашей задачи бесполезен — отдаёт
  цены (OHLCV) и короткую историю, он-чейн метрик не отдаёт. Поэтому
  он-чейн читаем через браузер: ~13 графиков раз в день через Claude in
  Chrome, классификация значений в зоны (дёшево / нейтрально / разогрето /
  экстремум). Известная
  хрупкость: смена вёрстки сайта ломает чтение → обязателен детектор
  пустых/кривых чтений и деградация «семья без данных не голосует».
- Метрики: EMA200d, RSI14w, drawdown от пика; MVRV-Z, NUPL, SOPR,
  Realized Price, Exchange Reserve/Netflow, Stablecoin Reserve; funding,
  OI, базис фьючерс-спот; потоки ETF; премия PAXG к золоту и корреляция
  с BTC.
- Хранилище локально (DuckDB / Parquet). Снимок дня — один JSON: зоны +
  сырые значения + источники + время; версионируется, хэшируется
  (воспроизводимость решений).
- Бюджет: ~$30–50/мес.

## 6. Стыковка с исследованием (раунды 1–2)

- «Оборот и комиссии — главный убийца» (подтверждено дважды) → счёт
  чистой прибыли и редкие решения зашиты в самое ядро концепта. ✔
- «Простое бьёт сложное после издержек» → детерминированные правила
  распределения; никакого машинного обучения в контуре решений. ✔
- «Сигналы не доказаны литературой» → семьи скоров — ГИПОТЕЗА, не факт.
  Проверяется бумагой и бэктестом; цена ошибки снижена ролью советника,
  малыми траншами и ручным подтверждением. Полосы он-чейн оценки — ровно
  та гипотеза, что отправлена на стенд исследованием (§8.1).
- Модули заработка поверх базы — Г-1 кэрри, Г-2 сетка с опционным
  прикрытием (уточнение Owner'а, §9 исследования), Г-3 тренд/моментум —
  отдельные гипотезы стенда, в первый этап НЕ входят.
- Плечо в первом этапе — ноль (мандат §3.5 + исследование §8.2). ✔

## 6-к. Контракт ядра — после ревью Owner'а 12.06 (для исполнителя)

Вход (всё неизменяемое, ядро только читает):
```
MarketSnapshot          — снимок дня: зоны + сырые значения + источники + время
LotJournal              — лоты с себестоимостью
RealizedProfitJournal   — зафиксированная чистая прибыль
DistributionJournal     — куда распределялась прибыль
SaleLockJournal         — блокировки перезахода: цена/дата продажи,
                          статус условий сброса
ProfileConfig           — профиль риска: транши, веса семей,
                          карта распределения, доли корзин
```

Словарь решений ядра (не «купить/продать», а):
```
Observe                 — наблюдаем, действий нет
TakeProfit              — зафиксировать X% позиции
DistributeProfit        — распределить чистую прибыль по корзинам
BuyDipFromBuffer        — докупить из буфера на подтверждённой капитуляции
MoveBufferToTerminal    — рекомендация вывода в терминал (решение ручное)
ManualReview            — нужен человек: порог превышен / конфликт правил
NoDecisionBadData       — критичные данные отсутствуют, действия запрещены
```

Минимальный выход решения:
```
action, asset, position_pct,
expected_net_profit, distribution_plan,
reset_conditions_created, warnings,
human_review_required,
input_hash, config_hash
```

Хэши входа и конфига — в каждом решении: любое решение воспроизводится
с тем же входом байт в байт (аудит, золотые тесты).

## 7. Мелкие развилки — закрыть в детальной архитектуре (старт не блокируют)

1. Себестоимость лотов: рекомендация — средневзвешенная (универсально,
   налоговая привязка вырезана); FIFO оставить опцией конфига для
   будущего сервиса.
2. Инициализация существующих позиций: если себестоимость неизвестна —
   консервативно взять текущую цену (занижает «прибыль», система
   осторожнее на старте).
3. Терминальный порог: уведомление-триггер + ручное решение
   (рекомендация); автоматических переводов нет по красным линиям.
4. Веса семей и пороги: дефолты из диалогов — стартовые, калибровка
   только стендом; в ядре — как конфиг, не константы.

   (б) чтение платных графиков через браузер + зоны + детектор сбоев;
   (в) каркас типов ядра + журнальная триада + золотые тесты;
   (г) бумажный журнал + дневной отчёт секретарю/в Телеграм.
3. **Бумага с первого дня.** Живые деньги не участвуют нигде
   (мандат §3.4, §5).
