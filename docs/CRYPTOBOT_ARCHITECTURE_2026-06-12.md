Граница И-8 проходит между контурами 1–5 (исследование, модель может всё)
и контуром 6 (деньги, только детерминированная программа). LLM живёт
в контуре 2 и в докладах 7; в контуре 6 её нет.

## 2. Частоты контуров (требование формулы утверждения)

| Контур | Частота | Замечание |
|---|---|---|
| Данные | непрерывно: цены — поток/минуты; он-чейн — раз в день; макро-календарь — по событиям | снимок дня собирается раз в сутки, 06:00 |
| Аналитика 24/7 | непрерывно, дайджест — раз в день + срочные события | не порождает торговых решений |
| Лаборатория/бэктест | по запросу | тяжёлые прогоны — по светофору лимитов §12 |
| Бумажный полигон | у каждой стратегии своя частота из её паспорта — хоть ежечасно | автомат разрешён |
| Боевое ядро CPDS | раз в день читает снимок; ДЕЙСТВИЯ — редкие, несколько раз в год | «молчит» = свойство этого контура |
| Панель/доклады | дневной отчёт + событийные уведомления | подтверждения — только здесь |

## 3. Контур данных

### 3.1. Карта возможностей поставщика (DataVendorCapabilities)

Каждый источник описан декларацией; ядро и сборщики зависят от деклараций,
не от конкретного поставщика. Смена «браузер → API» меняет декларацию
и адаптер, больше ничего.

```
DataVendor:
  id, metrics[], cadence, access (api | browser | file),
  history_depth, reliability (critical | normal | flaky),
  cost, notes
```

Стартовый набор (тарифы — текущие гипотезы, проверяются при подключении):
Binance (api: цены, фандинг, открытый интерес, объёмы; история глубокая),
Coinbase (api: кросс-чек цен), Deribit (api: опционы — только копим),
FRED (api: золото, макро-ставки), Farside (browser/file: потоки ETF),
DeFiLlama (api: стейблкоины), CryptoQuant (browser: он-чейн зоны, flaky).

### 3.2. Снимок дня — единственный вход боевого ядра

```json
{
  "meta": { "date": "...", "schema_version": 1, "snapshot_hash": "..." },
  "assets": { "BTC": {"price": ..., "ema200d": ..., "ath_drawdown": ...},
              "ETH": {...}, "PAXG": {...} },
  "families": {
    "price":     {"raw": {...}, "zone": "...", "confidence": "...", "sources": [...]},
    "valuation": {...}, "momentum": {...},
    "leverage":  {...}, "profit_taking": {...}
  },
  "data_quality": { "missing": [], "stale": [], "critical_ok": true }
}
```

Зоны: `cheap | neutral | warm | hot | extreme` (и зеркало вниз для
капитуляции). Зона + уверенность на семью; сырые значения хранятся рядом
для аудита. Снимок неизменяем, хэшируется, складывается в историю.

### 3.3. Качество данных

- свежесть: каждая метрика несёт срок годности; протухла → `stale`;
- браузерные чтения: детектор пустых/кривых значений (вне диапазона,
  скачок без рыночного события, нечитаемый график) → метрика выпадает;
- правило критичности (концепт §3.4): нет he-чейн оценки → `critical_ok =
  false` → ядро вернёт `NoDecisionBadData`; нет некритичной семьи →
  предупреждение и пониженная уверенность.

### 3.4. Хранилище

Локально: Parquet (сырьё и история) + DuckDB (запросы), каталог `data/`
проекта. Журналы боевого ядра — отдельно (§10), с резервной копией.
Снимки и журналы версионируются схемой (`schema_version`).

### 3.5. Известная дыра данных (честно, риск Д-1)

Для бэктеста семей нужна ИСТОРИЯ он-чейн метрик, а браузерное чтение даёт
текущие значения. Закрытие: (а) бесплатные исторические ряды (Coin Metrics
community и аналоги) для прошлого; (б) собственное накопление с первого дня;
(в) если дыра останется — бэктест ядра на ценах+деривативах, он-чейн
семья валидируется на накопленном вперёд. Решение по платному экспорту —
отдельно, когда упрёмся.

## 4. Контур аналитики 24/7

Вход: новостные ленты, макро-календарь (ставки, инфляция), регуляторные
события, геополитика. LLM классифицирует (событие → домен, важность,
ожидаемое влияние), сводит в дневной дайджест и срочные пометки.

Выход — информационные артефакты: `EventDigest` (дайджест дня),
`RegimeNote` (оценка режима рынка словами). Они идут в доклад и Owner'у;
в снимок дня и в решения ядра на первом этапе НЕ входят. Путь в решения
один: если захотим «семью сентимента» — она оформляется как StrategySpec-
фактор и проходит трубу §6 (бэктест → бумага → повышение).

Расширяемость по слову Owner'а: любые источники — вплоть до солнечной
активности и фаз Луны — подключаются как поставщики данных (§3.1)
и проверяются как факторы через ту же трубу. Платформе всё равно,
что тестировать; статус «работает» выдаёт только статистика.

## 5. Лаборатория стратегий

Место, где живут гипотезы. Очередь на старте: **Г-1** дельта-нейтральный
кэрри (фандинг), **Г-2** сетка с опционным прикрытием (план теста —
исследование §9.1), **Г-3** тренд/моментум, **Г-0** сами семьи скоров CPDS
(калибровка весов и зон). Фьючерсы и плечо здесь разрешены без ограничений.

## 6. Контракт стратегии — одна труба для всех

```
StrategySpec:
  id, version, author, hypothesis (откуда деньги — обязательное поле),
  universe[], data_requirements[] (ссылки на декларации §3.1),
  decision_cadence, rules (формальные, исполнимые без модели),
  params{}, risk_class (spot | derivative | leveraged)

BacktestResult:
  spec_ref, period, walk_forward_windows[],
  metrics: net_return, max_drawdown, turnover, fees_paid,
           vs_benchmarks (BTC, ETH, 50/50, CPDS-база),
  storm_behavior (март-2020, 2022, крах FTX — обязательные окна),
  data_gaps[], result_hash

PaperAccount:
  spec_ref, virtual_depo, открытие/история сделок, текущие позиции,
  ежедневная переоценка, журнал решений стратегии

PromotionReport:
  spec_ref, paper_period, стабильность по под-периодам,
  сравнение с бенчмарками и действующей базой,
  рекомендация (promote | extend | kill) + предлагаемый боевой вес
```

Жёсткие правила трубы:
1. `hypothesis` («кто и за что нам платит») — без него спецификация
   не принимается в лабораторию.
2. Правила стратегии формальны и исполняются программой; модель может
   ПРИДУМАТЬ стратегию, но не может БЫТЬ стратегией (И-8).
3. Бэктест обязан включать издержки и штормовые окна; результат без
   комиссий не существует (урок исследования: оборот — главный убийца).
4. Повышение в бой — только PromotionReport + записанное слово Owner'а +
   ограниченный начальный вес.

## 7. Бэктест-движок

- событийный прогон по историческим снимкам/свечам; запрещено заглядывание
  вперёд (point-in-time: стратегия видит только то, что было известно
  на дату решения);
- модель издержек: комиссии мейкер/тейкер, спред, проскальзывание
  (консервативная константа на старте), фандинг по историческим ставкам,
  для плечевых — маржа и ликвидации;
- walk-forward: калибровка на одном окне, проверка на следующем; запрет
  подгонки одним периодом (урок исследования §2.1);
- выход — `BacktestResult` с хэшем; повторный прогон того же спека на тех
  же данных обязан дать тот же хэш.

## 8. Бумажный полигон (требование формулы утверждения)

- у КАЖДОЙ стратегии — собственное виртуальное депо (`PaperAccount`);
  стартовый размер — условная единица, всё в долях;
- исполнение виртуальных сделок — детерминированный движок по правилам
  спека, по реальным котировкам с моделью издержек §7; автомат разрешён —
  живых денег и ключей здесь нет по построению;
- бенчмарки полигона: купи-и-держи биткойн, эфир, 50/50, действующая
  CPDS-база;
- ежедневная переоценка всех депо + сравнение — в дневной отчёт;
- CPDS на полигоне живёт так же, как остальные: её бумажное депо —
  «боевой кандидат», статистика для приёмки этапа «бумага» (мандат §5.3).

## 9. Боевое ядро CPDS — контракт (требование формулы утверждения)

Технология: Haskell — чистая функция, плотные типы, цена ошибки
максимальная (федерация §7). Персистентность и вся внешняя работа —
Python-обвязка. Ядро не ходит в сеть, не пишет на диск, не знает времени
кроме как из снимка.

### 9.1. Вход / выход

```
decide :: DecisionInput -> DecisionOutput

DecisionInput:
  snapshot       — снимок дня (§3.2), включая data_quality
  lots           — LotJournal
  profits        — RealizedProfitJournal
  distributions  — DistributionJournal
  saleLocks      — SaleLockJournal
  profile        — ProfileConfig

DecisionOutput:
  action            — Observe | TakeProfit | DistributeProfit
                      | BuyDipFromBuffer | MoveBufferToTerminal
                      | ManualReview | NoDecisionBadData
  asset, position_pct
  expected_net_profit         — по формуле §9.3
  distribution_plan           — доли по корзинам из профиля
  reset_conditions_created    — новые блокировки перезахода
  warnings[], human_review_required
  input_hash, config_hash, core_version
```

### 9.2. Семьи скоров (стартовые определения, калибрует Г-0)

| Семья | Метрики | Вес | Критичность |
|---|---|---|---|
| Цена/растяжение | отклонение от EMA200d, просадка от пика | 1.0 | нет |
| Он-чейн оценка | MVRV-Z, NUPL, отношение к реализованной цене | 1.5 | ДА |
| Импульс | недельный RSI, дивергенции | 1.0 | нет |
| Плечо рынка | фандинг (среднее 7д), открытый интерес/капитализация, базис | 1.0 | нет |
| Реализация прибыли | SOPR (7д), чистый поток на биржи | 0.5 | нет |

Перегрев подтверждён: взвешенная сумма «горячих» зон ≥ порога профиля.
Капитуляция — зеркально по «холодным» зонам. Зоны считаются по квантилям
скользящей истории, не по абсолютным константам.

### 9.3. Денежная арифметика (центр системы)

```
чистая_прибыль = выручка
               − пропорциональная_себестоимость (по лотам, метод из конфига)
               − комиссии − спред
TakeProfit разрешён, если чистая_прибыль ≥ minNetProfit профиля.
Распределяется ТОЛЬКО чистая_прибыль; возврат себестоимости — в К1
как тело, прибылью не считается.
```

Блокировка перезахода создаётся каждой продажей; сброс — любое из:
просадка ≥ X% от пика после продажи / он-чейн оценка в нижней зоне /
зафиксированная капитуляция / тайм-аут T месяцев (X, T — из профиля).

### 9.4. Профили риска (стартовые значения, калибрует Г-0 + бумага)

| Параметр | Консервативный | Сбалансированный | Агрессивный |
|---|---|---|---|
| Транш обычный / сильный перегрев | 1% / 3% | 2% / 4% | 3% / 5% |
| Порог взвешенной суммы (перегрев) | 3.5 | 3.0 | 2.5 |
| Распределение прибыли К1/К2/К3 | 40/40/20 | 50/35/15 | 60/30/10 |
| Доля защитной корзины (PAXG), макс | 0–10% | 0–15% | 0–20% |
| Обратный ход за раз (от буфера) | 15% | 25% | 35% |
| Мин. чистая прибыль для фиксации | выше | средняя | ниже |
| Действие без подтверждения, макс | 0% (всё руками) | ≤2% | ≤5% |

Транш >5% капитала — всегда `ManualReview`, в любом профиле (красная линия).

### 9.5. Инварианты ядра (проверяются типами и тестами на свойствах)

1. Сумма распределения = чистой прибыли (ни монеты мимо журнала).
2. К0 не существует для ядра; К3 — только входящий поток.
3. Ни одно действие при `critical_ok = false`, кроме `NoDecisionBadData`.
4. Перезаход дороже цены продажи при активной блокировке — невыразим.
5. Одинаковый вход → одинаковый выход (золотые тесты на хэшах).
6. Лимиты профиля — верхние границы; превышение невыразимо в типах.

## 10. Журналы и учёт

```
LotJournal:            lot_id, asset, qty, price, ts, origin, status
RealizedProfitJournal: sale_id, asset, qty, sale_price, proceeds,
                       cost_basis, fees, net_profit, ts, decision_id
DistributionJournal:   dist_id, sale_ref, bucket, amount, ts,
                       status (recommended | confirmed | done)
SaleLockJournal:       lock_id, asset, sale_price, qty, ts,
                       reset_rules{}, status (active | reset), reset_ts
```

Пишет журналы Python-обвязка ПОСЛЕ подтверждения действий; ядро только
читает. Каждое решение хранится с `input_hash`/`config_hash` — решение
воспроизводимо байт в байт. Сверка: на бумаге — журнал ↔ полигон;
на живых деньгах (потом) — журнал ↔ выписка биржи, расхождение = стоп.
Когда реестр фактов ядра федерации будет готов — журналы встают на него;
до тех пор локальная реализация с тем же контрактом (точка стыковки Ф-1).

## 11. Панель и доклады

- статус-снимок по контракту федерации §3: позиции (бумажные), результат
  за период, текущие зоны семей, последние решения, где нужен человек;
- дневной отчёт: одна страница — зоны, депо полигона против бенчмарков,
  события аналитики; «ничего не делаем» — тоже отчёт одной строкой;
- уведомления с подтверждением (Телеграм): только значимые действия;
- голос: секретарь читает статус-снимок — «что по крипте» в две фразы.
- Панель-интерфейс (веб) — НЕ первый этап; сначала текстовый отчёт.

## 13. Нарезка этапа «данные и стенд» (вход для ТЗ исполнителям)

| № | Задача | Зависит от | Приёмка (суть) |
|---|---|---|---|

| Т-5 | бэктест-движок v1 (цены + деривативы) | Т-4 | walk-forward на истории 2019–2025, хэш воспроизводим |
| Т-6 | дневной отчёт + Телеграм + статус-снимок федерации | Т-1, Т-4 | Owner получает утренний отчёт; «что по крипте» отвечает |
| Т-7 | спеки Г-0 (калибровка семей) и Г-1 (кэрри) в трубу | Т-5 | два BacktestResult с штормовыми окнами |

Порядок: Т-0 → Т-1 → (Т-2 ∥ Т-3) → Т-4 → (Т-5 ∥ Т-6) → Т-7.
Каждая задача — Worker (Опус) → независимый Reviewer → приёмка Admin.
Живые деньги не появляются нигде в этом этапе.

## 14. Чего нет в первом этапе

Живой торговли и биржевых ключей; живого плеча; веб-панели; сентимента
и социальных метрик; whale-трекинга; DEX; полных опционных стратегий
(данные Deribit копим); мультипользовательского сервиса (заложен типами:
профиль+конфиг+журналы на пользователя, ядро без глобального состояния).
