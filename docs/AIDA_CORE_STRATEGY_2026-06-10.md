# Aida Core — стратегия строгого ядра

**Дата:** 2026-06-10
**Автор:** Admin (архитектор системы)
**Основания:** обсуждение Owner'а 9–10 июня 2026 (идея «строгой части мозга» + ТЗ на план), архитектурный обзор `docs/ARCHITECT_REVIEW_2026-06-05.md`, прямое чтение текущей системы.
**Статус:** стратегия v1. Направление задано Owner'ом; исполнение первых шагов — после его «поехали». Первые правки затрагивают `scripts/**` (критичный путь — нужна подпись Owner'а).

## 1. Резюме для Owner'а — простыми словами

**Что строим.** Нейросети — это воображение и руки: они быстро придумывают и быстро
делают, но уверенно ошибаются, путают догадку с фактом и забывают историю. Мы
строим строгую часть мозга — программное ядро, через которое проходят слова и
действия всех моделей. Ядро не думает вместо них. Оно делает другое: **не
пропускает наружу** то, что не прошло проверку формы, источника, прав и риска.

**Почему это не устареет.** Ядро владеет тем, чего у чужих моделей никогда не
будет: нашей памятью, нашими правами, нашим журналом решений, нашими правилами
приёмки. Модели — сменные исполнители: сегодня одна, завтра другая, послезавтра
локальная. Чем сильнее становятся модели, тем больше они производят — и тем
ценнее фильтр, который отделяет проверенное от шума. Сильная модель, подключённая
к ядру, становится сильнее, а не опаснее.

**Что уже есть.** Наша система — уже зародыш ядра: роли, задачи с воротами
приёмки, политики, 22 поправки на повторные ошибки, журнал через git. Обзор от
5 июня показал на фактах, где «дисциплины без замков» не хватило: обязательная
независимая проверка на самом рискованном уровне один раз была пропущена, доска
показывала «рассинхрон 0» при реальном отставании, ворота обходятся ручной
правкой статуса. Это не повод для паники — это готовый список первых замков.

**Первый шаг.** Не строить новый сервис. Врезать замки в уже работающие ворота
(три правки, описаны в §8) и завести «чистый склад знаний» — реестр фактов и
решений с источниками. Дальше ядро растёт слоями, и каждый слой полезен сам по
себе, даже если следующий не построим.

**Риски.** Главных три: превратить систему в бюрократию (противоядие: каждый
новый замок обязан снимать ручную обязанность); завести «вторую правду» рядом с
досками (противоядие: доски рендерятся из реестра, а не ведутся параллельно);
недострой (противоядие: этапы самоценны). Подробно — §10.

**Через 3 месяца при правильном движении:** действия агентов идут через единый
вход с журналом; на рискованных уровнях изоляция проверяющего обеспечена
конструкцией, а не памятью модели; факты и решения лежат в реестре с источниками
и статусами; доски собираются из реестра; есть набор проверок, который ловит
обход ворот и загрязнение памяти; голосовой секретарь читает состояние хозяйства
из реестра, а не парсит тексты.

## 2. Главная формула и конституция ядра

> **Модель порождает возможности. Ядро превращает часть возможностей в
> проверенное знание, планы и действия.**

Нельзя программно заставить модель «не врать внутри головы». Можно программно
сделать так, что её слова и действия **не проходят в мир** без формы,
источников, прав и оценки риска. Ядро управляет не мышлением модели, а каналом
между моделью и миром.

Семь принципов (конституция — меняется редко и только Owner'ом):

1. **Модель предлагает — ядро решает.** Любое действие с последствиями — это
   запрос, который ядро разрешает, отклоняет или отправляет на подтверждение.
2. **Строгость пропорциональна цене ошибки.** Деньги, данные, память, права,
   прод — строго. Черновики, исследования, фантазия — свободно. Ядро, душащее
   мозговой штурм, — мёртвое ядро.
3. **Каждый новый замок снимает ручную обязанность.** Если проверку нельзя
   автоматизировать — она остаётся поправкой (correction), а не воротами. Число
   ручных ритуалов после каждого этапа должно уменьшаться.
4. **Человеку — доска, машине — реестр.** Человекочитаемые доски (DISPATCHER,
   STATUS_RU) остаются фасадом. Под ними появляется машинный реестр, из которого
   фасады собираются. Одна правда, два представления. (Примечание в конце
   DISPATCHER «не формализовать доску» остаётся в силе — формализуется слой ПОД
   доской, не сама доска.)
5. **Файлы и git — дом данных.** Ничего не переезжает в базу, пока файлов
   хватает. Git уже даёт журнал, авторство, историю и откат бесплатно. База —
   когда запросы станут медленными, не раньше.
6. **Ядро не думает.** Оно не заменяет техлида, не спорит с моделью о
   содержании, не выбирает архитектуру. Оно проверяет форму, источник, право,
   риск — и ведёт журнал.
7. **Знание отделено от шума.** Факт ≠ гипотеза ≠ решение ≠ мнение. У факта есть
   источник, или это не факт. Противоречия хранятся, а не замалчиваются.

## 4. Целевая архитектура: v0.1 → v1.0

Ядро растёт тремя слоями. Каждый слой самоценен.

### v0.1 «Узкое горло» (июнь) — замки + реестр

```
Owner / роли
   ↓
aida CLI  ←  единый вход: под капотом существующие скрипты
   ├─ ворота (ужесточённые: §7)
   ├─ ledger/  — машинный журнал: действия, факты, решения (JSONL, append-only, в git)
   └─ schemas/ — схемы шапок TASK/HANDOFF/SHIFT + записей реестра, валидатор в make check
```

Никакого сервера, никакой базы. Всё — файлы в git. Словарь действий уже
наполовину существует в Makefile (new-task, submit-task, accept-task,
take-shift…) — он облагораживается и получает журнал.

### v0.5 «Шлюз» (лето) — агенты ходят через ядро

- Запуск Worker/Reviewer — программно через ядро (Agent SDK): на Tier A ядро
  **само** поднимает проверяющего в холодной изолированной сессии. Изоляция
  обеспечена конструкцией, а не запретом — класс нарушений X-1 умирает.
- Память с типами и продвижением: сырое → гипотеза → проверено → принято (§6).
- Доски рендерятся: блок «Открытые блоки» DISPATCHER и шапка STATUS_RU
  собираются из реестра; снимок состояния — из `status.sh`, не из памяти модели.
- Голос (Phase 2 мандата) подключается к ядру как первый внешний клиент: дайджест
  хозяйства читает из реестра, команды отдаёт через словарь действий.

### v1.0 «Мозговой ствол» (осень) — сменные модели и долгие процессы

- Адаптеры моделей: Claude / Codex / локальные — сменные исполнители за общим
  контрактом (структурный ответ, запрос действий, обязательные источники).
- Долгие процессы: пауза, возобновление, контрольные точки человека, повтор
  упавшего шага — без потери смысла между сессиями.
- Лаборатория проверки в полный рост (§ниже) + метрика повторных ошибок.
- Узкие модели-органы там, где правил не хватает: роутер задач, классификатор
  риска, «факт или мнение». Каждый орган — как Vosk-ключ: маленький, локальный,
  с программными порогами, **никогда не источник истины, только фильтр-помощник**.

### Что где хранится

| Слой | Дом | Почему |
|---|---|---|
| Конституция (роли, принципы, критпути) | markdown + git, подпись Owner'а | Уже работает: критпути + approve |
| Рабочие доки (доски, overlay, ТЗ) | markdown + git | Человекочитаемость — ценность |
| Реестр (действия, факты, решения) | JSONL в git, append-only | Машиночитаемо, диффуемо, журнал бесплатно |
| Схемы и валидаторы | `schemas/` + `make check` | Форма проверяется программно |
| База (SQLite) | **не сейчас** | Только когда запросы по реестру станут медленными |

### Чем ядро НЕ является

Не оркестратор, заменяющий роли. Не вторая методология рядом с первой. Не
облачный сервис. Не своя большая нейросеть. Не «умная бюрократия»: строгость —
только в точках из принципа 2.

## 5. Онтология — типы, которые не плавают

«Огонь — это огонь»: у каждого понятия есть структура, права и жизненный цикл.

**Уже живые типы** (формализуются схемами, смысл не меняется): Project, Role,
User (права: can_override, can_approve_critical, can_deploy), Task (12 полей
шапки), Handoff, Shift, MailboxNote, Correction (=Lesson, формат BAD/GOOD/WHY
готов), CriticalPath, RiskTier (A/B/C), Mode (light/normal/strict/preview),
Approval.

**Новые типы** (эпистемический слой):

| Тип | Ключевые поля | Жизненный цикл | Создаёт | Подтверждает | Устаревает |
|---|---|---|---|---|---|
| **Fact** (факт) | утверждение (1 предложение), источник, дата проверки, область, уверенность, статус, связи | raw → verified → accepted → stale/superseded | любая роль, **источник обязателен** | повышение до accepted: Admin/Owner или повторная независимая проверка | по классу: снимки инфраструктуры ~30 дней; договорённости — при событии; провал проверки → stale |
| **Hypothesis** (гипотеза) | формулировка, что даст если верна, как проверить, статус | предложена → в проверке → подтверждена (рождает Fact/Decision) / отклонена (хранится с причиной) | любая модель, свободно — это зона фантазии | проверка по заявленному способу | не гниёт: либо проверена, либо честно лежит как гипотеза |
| **Decision** (решение) | что решено, кем, дата, основания (ссылки), область, условия пересмотра | принято → действует → пересмотрено/отменено новым решением | Owner или TL в своей зоне | уровень решающего = уровень отмены | только новым решением; история не стирается |
| **Source / Evidence** | вид (файл+строка / команда+вывод / git-объект / адрес+дата / слово человека: кто+когда), содержимое | — | автор записи | — | вместе с фактом |
| **ActionRequest** (запрос действия) | действие из словаря, параметры, инициатор, обоснование, риск-класс, статус, результат | запрошено → разрешено/отклонено/на подтверждение → исполнено → записано | любая модель | ядро по политике; критичное — Owner | журнал вечен |
| **Contradiction** (противоречие) | утверждение А, утверждение Б, источники, рабочая версия, что разрешит | открыто → разрешено (одна сторона победила/обе уточнены) | ядро при записи конфликтующего факта | тот, кто разрешает конфликт | пока не разрешено — живёт и видно |

Правила, которые делают типы «нерезиновыми»:
- Пересказ — не источник. «Модель сказала» — не источник.
- Решение — не факт о мире, а выбор; его нельзя «опровергнуть», только пересмотреть.
- Текущие примеры для миграции: «Owner выбрал минимальную страховку по Avito
  28.05» — сегодня строчка в DISPATCHER, должна стать записью Decision; «прод
  на таком-то снимке» — должен быть Fact с источником-командой, а не фразой в
  таблице (именно тут обзор поймал «рассинхрон 0» при реальном отставании).

## 6. Эпистемический контур — двигатель знания

Отвечает не на вопрос «что сказала модель», а на вопрос **«имеем ли мы право
считать это знанием»**.

**Карантин.** Новая информация не попадает в чистую память напрямую:
`сырое → гипотеза → проверено → принято`, отдельно `устарело / опровергнуто`.
Модель может принести сто идей — в принятое пройдут три, с источниками и
областью применения.

**Пять правил пропуска:**
1. Факт без источника не публикуется как факт — только как гипотеза.
2. Наблюдение («тесты упали») отделено от вывода («падает из-за X») — у вывода
   свой статус и своя проверка.
3. Устаревший факт не используется без перепроверки (у классов фактов — срок
   годности).
4. Конфликт фактов не затирается — рождается Contradiction с рабочей версией.
5. В долгую память — только через продвижение (promotion), не автоматически.

**Уровни памяти** (медленнее = чище):

| Уровень | Сегодня | Становится |
|---|---|---|
| Быстрая (разговор) | контекст чата | без изменений, нигде не хранится |
| Рабочая (неделя/проект) | DISPATCHER, MAILBOX, NEXT_ACTIONS | то же, но утверждения со ссылками на реестр |
| Проверенная (месяцы) | corrections (есть!), остальное размазано по доскам | реестр фактов и решений со статусами |
| Конституция (годы) | ROLE_MODEL, CLAUDE_GLOBAL, критпути + подпись | уже строго; добавляются принципы ядра (§2) |

Важно: ярус «конституция» **уже** работает строго (критпути + подпись Owner'а) —
иерархия памяти наполовину построена. Недостаёт среднего яруса: реестра фактов
и решений.

**Что это чинит из обзора:** дрейф досок (снимки состояния — из команды, не из
памяти) и шум evidence-проверки (правило сужается до дат рядом с PR/commit —
астрологические даты прогнозов перестают краснеть). Распухание STATUS_RU уже
починено после обзора ротацией и лимитом в make check; рендер из реестра
закрепит это конструкцией, а не дисциплиной.

## 7. Контур действий и политик

Маршрут любого действия с последствиями:

```
модель формирует ActionRequest
   → ядро: словарь (такое действие существует?)
   → политика (роль может? риск-класс? нужна подпись?)
   → исполнение (или отказ, или «жду Owner'а»)
   → запись в реестр (кто, что, когда, чем кончилось)
```

**Риск-классы действий** (стыкуются с Tier A/B/C задач):

| Класс | Примеры | Политика |
|---|---|---|
| read | чтение файлов, поиск, статус | свободно, в журнал |
| write-local | правка в рабочей копии, черновики | свободно, в журнал |
| commit | фиксация в git | по роли и зоне (Files-список задачи) |
| publish | push, PR, отправка наружу | ворота + журнал |
| critical | прод, деньги, секреты, удаление, критпути | подпись Owner'а; часть — запрещена всегда |

**Три первых замка** — ровно те места, где обзор показал течь (и все три уже
имеют дешёвую механику):

1. **Изоляция Reviewer на Tier A** — приёмка отказывает, если в HANDOFF не
   зафиксирована отдельная сессия проверяющего (копия существующего Tier-гейта;
   закрывает X-1).
2. **Дрейф досок** — `make check` на pre-push уже подключён после обзора
   (проверено живой отправкой 10.06). Остаётся: поднять проверку «грязного»
   продукт-репозитория с предупреждения до отказа и генерировать снимок
   состояния из `status.sh` (дозакрывает ADS-DRIFT-1).
3. **Подпись по git-автору** — проверка критпутей сверяет подпись с автором
   коммита, защита `USERS.md` локальным pre-commit (сужает ADS-SEC-1).

Честная оговорка: пока система живёт на машине Owner'а с полными правами,
абсолютной защиты «от себя» нет — замки защищают от **дрейфа дисциплины модели
и человека**, не от злонамеренного владельца. Этого достаточно: наша угроза —
энтропия, не злоумышленник.

## 8. Дорожная карта

### Сейчас (1–2 дня) — v0.1 старт

Три правки-замка из §7 (PR-1, PR-2, PR-3). Все трогают `scripts/**` → критпуть →
подпись Owner'а, исполняет Worker по ТЗ (Correction 020: Admin готовит ТЗ, не
кодит). Плюс одна задача без кода: **исполнить просроченную сверку AUDIT (a–e)**
— она ни разу не выполнялась (X-5) и сама вытащит хвосты.

### 1–2 недели — реестр и вход

- `aida` CLI v0: единый вход, под капотом существующие скрипты; каждое действие
  пишет строку в `ledger/actions-YYYY-MM.jsonl`.
- `schemas/` + валидатор шапок (TASK/HANDOFF/SHIFT) в `make check` — формат
  перестаёт быть джентльменским соглашением.
- `ledger/facts.jsonl` + `ledger/decisions.jsonl` и команды `aida fact add /
  promote / contradict`, `aida decision add`. Перенос 10–15 живых решений Owner'а
  из DISPATCHER в реестр как образец.

### 1–2 месяца — шлюз и фасады

- Запуск Worker/Reviewer через ядро (Agent SDK), автоматический холодный
  Reviewer на Tier A — изоляция по конструкции.
- Рендер блока «Открытые блоки» DISPATCHER и шапки STATUS_RU из реестра; ротация
  досок (живой статус ≤ 1 экрана, остальное — журнал).
- Лаборатория проверки v0: contract-тесты методологии (каждый путь из
  ROLE_MODEL/поправок существует; ворота не обходятся: Tier A без Reviewer —
  отказ, критпуть без подписи — отказ, факт без источника — отказ валидатора).
- Голос Phase 2 — секретарь читает дайджест хозяйства из реестра и запускает
  агентов через словарь действий (после живого теста этапа 1).

### 3–6 месяцев — сменные модели и долгие процессы

- Адаптеры: Claude / Codex / локальная модель за одним контрактом; модели
  становятся расходным материалом.
- Долгие процессы: пауза/возобновление/контрольные точки (своя тонкая
  state-machine поверх реестра; тяжёлые движки — только если своя перестанет
  тянуть).
- Узкие модели-органы по мере боли: роутер задач, классификатор риска,
  «факт/мнение» — сперва правилами, модель — если правил не хватит.
- Метрика качества: повторные инциденты по поправкам (Correction 018 уже
  повторялась — считаем такие).

### Чего НЕ делать (по слову Owner'а и здравому смыслу)

- Не переписывать систему с нуля — только наращивать замки на живое.
- Не писать свою большую нейросеть.
- Не тащить тяжёлую инфраструктуру (policy-движки, оркестраторы процессов,
  брокеры, облачный сервер ядра) — пока своя тонкая прослойка справляется.
- Не переносить overlay-доки в базу.
- Не формализовать человекочитаемые доски (формализуется слой под ними).
- Не строить ворота там, где цена ошибки низкая.

### Приоритетная оговорка

Пункты P0 обзора (открытый HTTP с деньгами, бэкапы на том же диске, счёт CI,
PII в фикстурах) — **впереди ядра по срочности**, но они не конкурируют за то же
ресурс: это задачи Owner'а (карта, аккаунты) и Worker'ов, сильная модель им не
нужна. Ядро — не оправдание их отложить.

## 10. Риски и противоядия

| Риск | Противоядие |
|---|---|
| Умная бюрократия — всё тормозит | Принцип 3: замок обязан снимать ручную обязанность; после каждого этапа считаем ручные ритуалы — их должно становиться меньше |
| Вторая правда (реестр против досок) | Фасады рендерятся из реестра поэтапно; пока блок не рендерится — он живёт по-старому; никогда не «ведём в двух местах» |
| Недострой / мёртвый сервис | Каждый этап самоценен: замки полезны без CLI, CLI — без шлюза, шлюз — без адаптеров |
| Один оператор (X-2 из обзора) | Ядро снижает зависимость от внимания человека (ворота не устают), но вопрос второго человека остаётся открытым — отдельное решение Owner'а |
| Загрязнение чистой памяти | Карантин + продвижение только через подтверждение; валидатор отказывает факту без источника |
| Ядро отстаёт от роста моделей | Ядро намеренно не зависит от способа мышления моделей — только от контракта на входе/выходе; контракт версионируется |
