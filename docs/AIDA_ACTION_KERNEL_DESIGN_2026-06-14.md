## Что это
Тонкий слой ВОРОТ поверх уже работающего журнала (`scripts/aida_ledger.py` + `ledger/*.jsonl`).
Без сервера, без базы, без второго движка. Четыре недостающих звена И-2:
1. **Словарь действий** — `PROJECTS/actions.json` (данные): имя, домен, класс риска, кто запрашивает, кто одобряет, нужно ли подтверждение, исполнитель (путь к УЖЕ существующему скрипту).
2. **Журнал действий** — `ledger/actions.jsonl` + схема: строки одного A-id `requested → approved|rejected → executed|failed` (открыть до исполнения, закрыть результатом).
3. **Вход** — `aida action request|approve|reject|execute|show` (над тем же журналом, не новый инструмент).
4. **Ворота** — проверка ПОЛЕЙ программой (миллисекунды, без модели): действие в словаре? роль вправе? класс риска? опасное → есть подтверждение по ЭТОМУ запросу?

Делает И-7 (опасное только с подтверждением) и И-10 (всё в журнал) **проверяемыми тестом**.
Первые 4 действия: снимок проекта (свободно), запуск работника, приёмка работы, отправка наружу (с подтверждением). Денежный контур И-8 НЕ трогаем. Текст конституции НЕ трогаем (И-11).

## ГЛАВНАЯ ЧЕСТНАЯ ПРАВДА (критик, и это про саму идею ядра)
«Мир меняется ТОЛЬКО через ядро» нельзя сделать **абсолютным** на машине с полными правами.
Исполнители (`new-task.sh`, `accept-task.sh`, `snapshot.sh`) — обычные скрипты; у кого есть
терминал, тот может вызвать их **напрямую, мимо ядра и журнала**. Ядро не перекрывает прямой вход —
оно добавляет рядом дисциплинированный путь (ворота + журнал) и **ловит обход постфактум сверкой**.

Значит «только через ядро» держится не одной стеной, а **тремя слоями вместе**:
1. ядро — гейтит и журналирует одобренный путь (принуждение на своём входе);
2. **ограничение инструментов модели** — подчинённая модель физически не должна иметь Bash/запись
   (так уже сделано в голосе: brain.py режет до поиска/чтения) → она в принципе не может вызвать
   исполнитель мимо ядра;
3. **сверка-с-миром** — рекон ловит любой артефакт (TASK-файл, коммит), появившийся без записи ядра,
   и помечает инцидентом.

Это не дефект, это структурный предел: ядро защищает от ДРЕЙФА дисциплины модели/человека, не от
злоумышленника с правами root. Настоящая «клетка» для модели — пункт 2 (урезанные инструменты),
ядро — пункты 1 и 3.

## ОБЯЗАТЕЛЬНЫЕ правки до сборки (критик)
1. **Сверка-с-миром, а не журнала с собой.** Добавить красный тест: прямой запуск `new-task.sh`
   без A-id → `aida check` помечает TASK-файл мимо журнала как ИНЦИДЕНТ. Без него несущая часть И-2
   не принуждена (рекон сейчас сверяет журнал сам с собой).
2. **Якорь личности на critical.** `--by owner` сейчас самоназначаем аргументом — модель/скрипт
   поставит `--by owner` и пройдёт. На critical-исполнении сверять `git config user.email` с USERS.md
   из HEAD (ровно как `.githooks/commit-msg`), иначе «защита от самоодобрения» мнимая.
3. **Честная приписка охвата.** Снять из `run.sh` honor_system ТОЛЬКО покрытое: И-7/И-10 — полно;
   И-2 — лишь «через ядро без одобрения нельзя» (прямой обход ловится постфактум, не блокируется);
   И-1 — лишь «исполнение отказывает голосу», манифест «4 инструмента» принуждается в голосе, не ядре.
   Не преувеличивать охват.
