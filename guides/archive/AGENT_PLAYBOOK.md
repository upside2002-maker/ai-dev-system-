# Agent Playbook — обучение и общение с агентами

Все рекомендации, которые я давал по работе с AI-агентами в проекте Sitka Office.
Собрано в одном файле для быстрого доступа.

---

## Часть 1. Базовые принципы

### 1.1 Агент — это смышлёный коллега, который только что вошёл в комнату

Он не видел вашу переписку, не знает, что вы уже пробовали, не понимает, почему задача важна.

**Плохой промпт:**
> Исправь баг в pricing

**Хороший промпт:**
> В sitka-core/src/Engine/Pricing.hs функция calculateMargin возвращает RUB вместо Percent.
> Маржа должна быть в процентах от quotedPrice.
> Нужно: переписать сигнатуру `calculateMargin :: RUB -> RUB -> Percent`,
> формула `(revenue - cost) / revenue * 100`.
> Не трогай PricingParams и calculateCost.
> Проверь: stack test должен пройти все 20 тестов pricing.

### 1.2 Никогда не делегируй понимание

**Плохо:** "На основе твоего анализа исправь проблему"
**Плохо:** "После того как разберёшься, реализуй"

Эти фразы перекладывают синтез на агента вместо того, чтобы сделать его самому. Промпт должен доказывать, что вы поняли: указать конкретные файлы, строки, что именно поменять.

### 1.3 Самодостаточность промпта

Промпт должен работать без контекста переписки. Включайте:
- Цель задачи и зачем она нужна
- Конкретные пути файлов
- Что вы уже выяснили или исключили
- Контекст вокруг проблемы для самостоятельных решений
- Если нужен короткий ответ — скажите явно ("отчёт до 200 слов")

### 1.4 Терпение с context window

Сложные задачи требуют большого контекста. Не торопите агента с краткими ответами, когда задача объёмная. Пользователь знает о limit-ах и ждёт автономной работы до завершения.

## Часть 2. Структура self-learning системы

Система состоит из 5 слоёв, каждый имеет свою роль.

### 2.1 CLAUDE.md (правила проекта)

**Что:** Инструкции для агента в конкретном проекте.
**Где:** В корне репозитория (`/Users/ilya/Projects/sitka-office/CLAUDE.md`).
**Когда обновлять:** При смене архитектурного решения, добавлении нового слоя, появлении нового запрета.

**Что должно быть:**
- Архитектура коротко (3-5 строк)
- MANDATORY правила (type safety, паттерны)
- Anti-patterns с примерами кода (BAD → GOOD)
- Reading order для агента (что читать перед задачей)
- Что уже существует (чтобы агент не дублировал)
- Что НЕ делать (запреты)

**Чего избегать:**
- Stale prohibitions ("не делай X", если X уже сделано)
- Маркетинговый текст
- Теория без примеров
- Дублирование того, что есть в reference-snippets

### 2.2 corrections.md (журнал ошибок)

**Что:** Лог повторяемых ошибок агента в формате BAD → GOOD → WHY.
**Где:** `.claude/corrections.md` в проекте + `corrections/global-corrections.md` в ai-dev-system.
**Когда обновлять:** ВСЕГДА после ревью или коррекции, если ошибка может повториться.

**Формат записи:**
```markdown
## #N — Краткое название

**Когда:** (контекст, в каком слое/файле/паттерне)

**BAD:**
```haskell
-- что агент сгенерировал
let rawPrice = toRealFloat (unUSD price) :: Double
```

**GOOD:**
```haskell
-- как должно быть
instance PersistField USD where
  toPersistValue (USD x) = PersistDouble (toRealFloat x)
```

**WHY:** Newtype не должен сниматься на границе с БД. PersistField сохраняет инвариант.
```

**Правила:**
- Записывать сразу, а не "потом"
- Конкретный код, не абстрактные описания
- WHY обязательно — без него правило непонятно
- Локальные правила в проекте, кросс-проектные в ai-dev-system

### 2.3 review-checklist.md (pre-commit чеклист)

**Что:** Список проверок перед коммитом.
**Где:** `.claude/review-checklist.md`.
**Длина:** 20-30 пунктов максимум.

**Структура:**
- Type safety checks
- Тесты прошли
- Линтеры зелёные
- Нет hardcoded values
- Документация обновлена если API изменилось
- Corrections обновлены если нашли новый anti-pattern

### 2.4 reference-snippets/ (эталонный код)

**Что:** Образцы правильной реализации типовых паттернов.
**Где:** `.claude/reference-snippets/` или `templates/reference-snippets/`.

**Пример наполнения для Haskell:**
- `haskell-handler-patterns.md` — как писать AppM handler
- `haskell-state-machine.md` — паттерн SM с pure transition
- `haskell-pricing-engine.md` — pure функции без IO

**Правило:** Если паттерн используется в 3+ местах — он должен быть в snippets.

### 2.5 prompts/task-splitting.md (шаблоны декомпозиции)

**Что:** Шаблоны для разбиения больших задач на агентские.
**Где:** `.claude/prompts/task-splitting.md`.

**Что должно быть:**
- Шаблон задачи (формат: Слой/Файлы/Задача/Не трогать/Проверка)
- Примеры разбиения (одна большая задача → 3-4 агентских)
- Когда задачу нельзя дробить (атомарные изменения API)

## Часть 3. Operating Model для агентов

### 3.1 Роли

Один проект = несколько ролевых агентов с чёткими границами:

| Роль | Владеет | Не владеет |
|------|---------|-----------|
| **Core Agent** | Domain types, state machine, pricing, API контракты, DB schema | Парсеры, UI, browser automation |
| **Services Agent** | FastAPI, парсеры, Telegram bot, notifications, external APIs | Бизнес-решения по деньгам и статусам |
| **Frontend Agent** | Операторские сценарии, UX, API client | Расчёт цен, бизнес-логика статусов |

**Принцип:** Один агент = один слой. Не давайте агенту задачу, требующую изменений в двух слоях одновременно.

### 3.2 Цикл работы агента

1. Прочитать `CLAUDE_GLOBAL.md` → corrections → overlay (текущее состояние проекта)
2. Определить, в каком слое работаем
3. Прочитать playbook для слоя + relevant reference snippets
4. Выполнить задачу в рамках одного слоя
5. Build / test / lint
6. Обновить corrections если ошибка повторяемая
7. Обновить overlay если изменилось состояние проекта

### 3.3 Формат задачи для агента

```
Слой: [core | services | web]
Файлы: [конкретные пути]
Задача: [что сделать]
Не трогать: [что не менять]
Проверка: [stack build | pytest | tsc | npm test]
Контекст: [что уже сделано, какие смежные изменения]
```

### 3.4 Координация между агентами

- **Один файл = один агент в момент времени.** Параллельная работа над одним файлом = merge conflicts и потерянные изменения.
- **API контракты фиксируются ДО параллельной работы.** Core определяет тип, services и web берут его как данность.
- **Core мержится первым.** Он определяет контракт, остальные подстраиваются.
- **Если задача затрагивает 2 слоя** — разбейте на две задачи, core сначала.

## Часть 4. Запуск задач в Claude Code

### 4.1 Когда использовать Agent (sub-agent)

**Используйте sub-agent, когда:**
- Задача требует много чтения файлов, но результат — короткий отчёт
- Параллельная работа (3 агента ищут разные вещи одновременно)
- Изоляция контекста (главный поток не должен загрязняться)

**Не используйте, когда:**
- Цель уже известна (используйте Read напрямую)
- Конкретный символ нужно найти (используйте Grep)
- Простая задача без множественных шагов

### 4.2 Параллельный запуск

Если есть 2-3 независимых задачи — запустите их в одном сообщении параллельно. Например:

```
1. Agent читает domain types
2. Agent читает API endpoints  
3. Agent читает DB schema

→ Все три в одном сообщении, не последовательно.
```

Это экономит время и cache hits.

### 4.3 Worktree для рискованных изменений

Если задача требует экспериментов или может сломать кодовую базу:
- Создайте worktree (`EnterWorktree`)
- Агент работает в изолированной копии
- После проверки — мержите или удаляете

## Часть 5. Что НЕ делать с агентами

### 5.1 Не давайте задачи без контекста

**Плохо:** "Улучши pricing"
**Хорошо:** "В calculateCost нет учёта customs duty для товаров > $200. Добавь параметр highValueCustoms :: Maybe Percent в PricingParams. Если задан и itemPriceUsd > 200, добавь к total."

### 5.2 Не позволяйте агенту самому решать архитектуру

Агент хорош в реализации, плох в стратегии. Архитектурные решения принимаете вы:
- Где живёт каталог (services, не core)
- Какой state machine
- Где границы слоёв

Агенту даёте: "Сделай X в файле Y по образцу Z".

### 5.3 Не доверяйте безоговорочно

Каждое изменение проверяйте. Особенно:
- Удаление файлов
- Изменения в DB schema
- Изменения публичных API контрактов
- Touched файлы вне заявленного слоя

### 5.4 Не игнорируйте повторяемые ошибки

Если агент второй раз делает ту же ошибку — это сигнал обновить corrections. Без этого ошибки будут множиться.

### 5.5 Не ведите длинные диалоги в одной сессии

Контекст имеет предел. Лучше:
- Закрыть текущую задачу
- Зафиксировать результат в commit или документе
- Начать новую сессию для новой задачи

## Часть 6. Multi-agent работа (несколько агентов на проекте)

### 6.1 Когда это полезно

- У вас есть параллельные задачи в разных слоях
- Один агент пилит core, второй services
- Нет пересечений по файлам

### 6.2 Когда вредно

- Маленькая команда (вы один с агентом)
- Задачи последовательные (А блокирует Б)
- Мало кода (overhead координации > выгода)

### 6.3 Координационные артефакты

Если работаете с несколькими агентами параллельно:

**`HANDOFF.md`** — что один агент передаёт другому. Удаляется после того, как принят.

**`.claude/agent-locks/`** — какие файлы у какого агента. Простой текстовый файл.

**Daily sync через CURRENT_STATE.md** — каждый агент в начале сессии читает, в конце обновляет.

### 6.4 Communication discipline

- Не общайтесь между агентами через свой мозг ("я скажу второму агенту что первый сделал X"). Записывайте.
- Если решение принято — оно в файле, а не в чате.
- Сессии непостоянны, файлы постоянны.

## Часть 7. Развитие самого себя как тех-лида с агентами

### 7.1 Ежедневная практика (30-45 минут)

**Понедельник:** читайте код, который агент написал на прошлой неделе. Найдите 1 место, где сделали бы по-другому.

**Вторник:** обновите corrections если нашли новый anti-pattern.

**Среда:** прочитайте одну часть Haskell core full understanding (не только просмотр).

**Четверг:** разберите один Servant pattern или Persistent feature до уровня "могу объяснить".

**Пятница:** ревью одного PR агента максимально жёстко. Если бы это писал джуниор — что бы вы сказали?

### 7.2 Еженедельный ритуал

В пятницу перед закрытием:
1. Что я понял в коде, чего не понимал в понедельник?
2. Какие 2-3 решения принял агент, которые я бы пересмотрел?
3. Что записать в corrections?
4. Какая часть системы для меня всё ещё чёрный ящик? Запланировать на следующую неделю.

### 7.3 Контрольные точки роста (12 недель)

**Неделя 1-4:** понимаю весь Haskell core, могу объяснить каждый файл.
**Неделя 5-8:** уверенно ревьюю PR-ы, ловлю anti-patterns раньше агента.
**Неделя 9-12:** могу сформулировать архитектурное решение и обосновать выбор.

К 12 неделе вы должны быть способны: написать ADR (Architecture Decision Record) на 1 страницу, защитить его перед другим разработчиком.

## Часть 8. Отношение к "vision documents"

Когда другой агент (или ChatGPT) пишет вам грандиозный план — фильтруйте по 4 критериям:

### 8.1 Учитывает ли текущий масштаб?

Если у вас 182 контакта, а план описывает систему для 10,000 — это не план, это фантазия.

### 8.2 Учитывает ли что уже построено?

Хороший план опирается на существующее, расширяет. Плохой — предлагает переписать.

### 8.3 Различает ли правила и данные?

Правила (state machine, pricing, validation) — стабильные, идут в core.
Данные (catalog, prices, listings) — изменчивые, идут в services.

Если план путает их — он будет дорог в реализации и поддержке.

### 8.4 Реалистичен ли по технологии?

"Агент автономно ведёт переписку с клиентами на Avito" — нереалистично (Avito банит ботов, юр. риски, нет API чата).
"Агент готовит черновик ответа, менеджер отправляет" — реалистично.

Грандиозные vision documents — это inspiration, не roadmap. Берите идеи, отбрасывайте сроки и автоматизацию выше реалистичной.

## Часть 9. Чеклист: правильно ли я использую агентов

Раз в 2 недели проверяйте себя:

- [ ] Каждый промпт самодостаточен (агент может работать без переписки)
- [ ] Указываю конкретные файлы и строки
- [ ] Говорю что НЕ трогать, не только что трогать
- [ ] Прохожу через corrections перед каждой Haskell задачей
- [ ] Обновляю corrections после каждой ревью
- [ ] Не позволяю агенту менять архитектуру без обсуждения
- [ ] Каждый PR агента ревью лично, не на доверии
- [ ] CLAUDE.md актуален (нет stale prohibitions)
- [ ] Reference snippets отражают реальные паттерны кода
- [ ] Multi-file задачи разбиты на single-file агентские

Если 8+ из 10 — вы используете агентов правильно. Если меньше — есть конкретные точки роста.

## Часть 10. Быстрая шпаргалка

**Перед задачей:**
- Какой слой? (core / services / web)
- Какие файлы трогаем? Какие точно НЕ трогаем?
- Какая проверка покажет, что готово?

**Промпт агенту:**
- Цель + контекст в первых 2 предложениях
- Конкретные пути файлов
- Что НЕ менять
- Команда проверки
- Длина ответа если важно

**После работы агента:**
- Diff читаем глазами, не на доверии
- Если ошибка — в corrections.md
- Если новый паттерн — в reference-snippets/
- Если изменилось состояние — в overlay/CURRENT_STATE.md

**Не делайте:**
- Не давайте агенту "придумай как лучше"
- Не позволяйте трогать несколько слоёв в одной задаче
- Не игнорируйте повторяемые ошибки
- Не доверяйте архитектурным решениям агента
