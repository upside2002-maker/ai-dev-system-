# ТЗ: VPN Reality на основном сервере для 2 непубличных пользователей

Документ — спецификация инфраструктурной задачи. Адресован VPN-агенту
(отдельная Claude-сессия с SSH-доступом к серверу).

Дата: 2026-05-24.
Owner: Илья.
Сервер: 94.72.112.106 (Hetzner-style, Ubuntu 24.04, root).

## Контекст

На сервере уже работает прод-стек Sitka (postgres / core / services /
web / backup) и nginx FounderOS (Виталий, наш совладелец) на 80/443.
Это один физический сервер, мы — единственная команда (Илья + Виталий).
Никаких «соседей по shared VPS» нет, это была устаревшая модель в
журнале до 2026-05-24.

Под VPN требуется развернуть **отдельный изолированный контейнер**
который не трогает ни Sitka, ни FounderOS. Два пользователя — внешние
(не мы), не публичная служба, без подписки/регистрации/панели.

Главная цель — **устойчивость к блокировкам в РФ** при максимальной
скрытности от DPI.

## Технические требования

### Протокол

VLESS + Reality + xtls-rprx-vision (Xray-core).

Это сегодня самый устойчивый к фильтрам в РФ протокол:
- маскируется под TLS handshake к реальному внешнему сайту;
- внешне не отличается от обычного HTTPS-запроса;
- не требует своего TLS-сертификата (использует чужой);
- активный probing не определит наш сервер как VPN.

OpenVPN, IKEv2, WireGuard без обфускации — **не рассматриваются**,
они стабильно блокируются в РФ.

### Транспорт и порт

- **TCP**, нестандартный высокий порт (по умолчанию `39847`, можно
  заменить любым другим из диапазона 30000-65000 по согласованию).
- НЕ занимать 443 — там работает nginx FounderOS, не трогаем.
- НЕ занимать 8088 — там работает sitka-web.
- НЕ открывать UDP 443 — Reality TCP-only, UDP не нужен и привлекает
  внимание сканеров QUIC.

### Маскирующий target

`www.cloudflare.com:443` — дефолтный выбор. Альтернативы при желании
Owner'а: `www.microsoft.com:443`, `dl.google.com:443`, `www.apple.com:443`.
Все одинаково безопасны, различие только косметическое.

### Изоляция

- Каталог `/opt/vpn/` (новый, не пересекается с `/opt/sitka-office/`).
- Внутри: `docker-compose.yml`, `config/config.json` (Xray), `README.md`.
- Контейнер запускается отдельной compose-сессией, не примыкает к
  Sitka-compose.
- Image — официальный `teddysun/xray` или `ghcr.io/xtls/xray-core`
  (без сторонних форков с панелями).

### Пользователи

- Два разных UUID, генерируются командой `xray uuid` (или эквивалент).
- В конфиге `clients` массив с двумя записями.
- Метки `user1` / `user2` — нейтральные, без раскрытия личности.
- Один потеряет / сольёт ключ — отзываем только его, второй продолжает
  работать.

### Безопасность

- **No-logs:** Xray access-log отключён (`"log": {"loglevel": "none"}`
  или путь `/dev/null`).
- **DNS:** через DNS-over-HTTPS внутри Xray (рекомендую `1.1.1.1` или
  `https+local://cloudflare-dns.com/dns-query`), не через системный
  resolver сервера.
- **IPv6:** проверить отключён ли на сервере; если включён — отключить
  для VPN-контейнера или явно роутить через IPv4, чтобы исключить
  утечки.
- **iptables `DOCKER-USER`:** добавить разрешение `ACCEPT` для TCP на
  выбранный порт. Остальной DROP / закрытие соседних портов не трогать.
- **fail2ban на SSH:** если ещё не настроен — настроить (отдельно от
  VPN, как общая гигиена сервера).
- **Регулярные обновления:** автоматическое `docker compose pull &&
  up -d` через cron или systemd timer раз в неделю (для security
  patches Xray).
- **SSH-доступ к VPN-конфигу:** только root на сервере. Никакой
  публичной панели управления.

## Категорически НЕ делать

- Не трогать `/opt/sitka-office/` (compose, конфиги, контейнеры).
- Не трогать nginx FounderOS на 80/443.
- Не трогать Sitka-контейнеры (postgres / core / services / web / backup).
- Не трогать `docker-compose.override.yml` Sitka.
- Не устанавливать панели управления VPN: 3X-UI, X-UI, Marzban,
  Hiddify. Они избыточны для 2 пользователей и добавляют attack
  surface (своя веб-морда, JWT, БД).
- Не публиковать ничего автоматически в Telegram-боты для выдачи
  ключей.
- Не логировать активность пользователей (IP, посещённые домены,
  объёмы трафика).
- Не публиковать ссылку для регистрации.
- Не открывать UDP 443.
- Не модифицировать системный nginx (отдельный от FounderOS, если
  такой обнаружится).

## Deliverables

1. **Каталог `/opt/vpn/`** на сервере с:
   - `docker-compose.yml` — определение Xray-сервиса.
   - `config/config.json` — конфиг Xray.
   - `README.md` — что это, как запустить/остановить, как ротировать
     ключи, как читать логи (если потребуется debug-режим).

2. **Открытый порт в `iptables DOCKER-USER`:** одно правило ACCEPT для
   TCP на выбранный порт. С комментарием в формате
   `iptables-save | grep` понятно «vpn xray vless reality». Сохранить
   правило через `iptables-persistent` или эквивалент чтобы выживало
   reboot.

3. **Две VLESS-ссылки** (URI-формат `vless://...`) в виде текстового
   файла или в выводе агента. Один — `user1`, второй — `user2`.

4. **Два QR-кода** — сгенерированные из ссылок выше, как PNG-файлы
   или ASCII в терминале. Для удобства передачи пользователям через
   мессенджер.

5. **Краткая инструкция для конечного пользователя** одним абзацем
   на русском: какое приложение скачать (по платформам), как
   импортировать ключ через QR или ссылку, что делать если не
   подключается. Без технического жаргона.

6. **Smoke-тест** прохождения через VPN:
   - подключиться с одного устройства (любого);
   - открыть `https://2ip.ru` или `https://ipinfo.io`;
   - убедиться что показывается IP `94.72.112.106`, а не клиентский;
   - проверить что трафик на `https://yandex.ru` идёт нормально
     (страна провайдера в whois — соответствует серверу).

7. **Короткий доклад в `MAILBOX/to-admin.md`** по итогам:
   - где лежит конфиг;
   - какой порт открыт;
   - что было сделано в системных файлах (только iptables);
   - две ссылки (можно прислать отдельным сообщением через защищённый
     канал, не в git);
   - результат smoke-теста.

## Open questions перед стартом

1. **Порт** — default `39847`, или генерируется случайный из
   30000-50000.
2. **Маскирующий target** — default `www.cloudflare.com:443`.
3. **Метки пользователей** — default `user1` / `user2`, никаких имён.
4. **DNS внутри Xray** — default `1.1.1.1`.
5. **Авто-обновления контейнера** — default включено, раз в неделю.

Если по какому-то пункту default не подходит — Owner возражает
явно ДО запуска.

## Что вернуть Admin'у в MAILBOX

Короткая записка по формату `MAILBOX/README.md`:

- что: VPN развёрнут (или не развёрнут с пояснением почему);
- где: путь на сервере, открытый порт;
- ссылки для двух пользователей — **передать отдельным каналом, не в
  git и не в обычной переписке** (например, голосом / в зашифрованном
  мессенджере);
- результат smoke-теста (одна строка);
- срочность: done.
