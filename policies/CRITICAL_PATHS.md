# Файлы и зоны в обязательной подписи владельца

Любая задача, которая трогает один из путей ниже, требует подписи владельца — поле `Critical approved by: <email>` в шапке задачи + commit от email с `can_approve_critical: yes` в `policies/USERS.md`.

## Зачем это нужно

Эти файлы определяют как работает сама система разработки — описание ролей агентов, правила, шаблоны задач, скрипты, поправки. Случайное изменение здесь ломает координацию между разработчиками и между сессиями. Поэтому правки идут только через явное подтверждение владельца.

Защита работает на стороне `scripts/accept-task.sh`: если задача в секции `## Files` указывает один из путей ниже — приёмка задачи откажет без подписи. Прямые коммиты в эти файлы технически не запрещены (git не блокирует), но владелец видит их в журнале и в периодической сверке (`policies/AUDIT.md` появится в коммите 5).

## Когда подпись НЕ нужна

Если автор задачи (`Created by:`) — сам владелец с правом `can_approve_critical: yes`, отдельная подпись не требуется. Логика: создатель и подписант — один человек, отдельная строка `Critical approved by:` была бы избыточной. Commit-автор в `git log` остаётся как аудит-след.

Это означает, что владелец работает в обычном режиме: создаёт задачу через `make new-task` (поле `Created by:` заполняется его email автоматически), правит файлы, принимает. Скрипт пропускает.

Второму разработчику этот short-circuit не работает: его email не имеет `can_approve_critical: yes`, поэтому он обязан получить подпись владельца перед приёмкой такой задачи.

## Список критичных путей

Скрипт `scripts/accept-task.sh` читает блок ниже как список паттернов. Один паттерн на строку. Комментарии (`# ...`) и пустые строки игнорируются.

Форматы:

- `путь/к/файлу.md` — точное совпадение.
- `каталог/**` — любой путь, начинающийся с `каталог/`.
- `*суффикс.md` — любой путь, заканчивающийся на `суффикс.md` (для имён файлов независимо от каталога — например, `*architecture-invariants.md` ловит и продуктовый, и overlay-копию).

```
# Корневая методология системы разработки
ROLE_MODEL.md
CLAUDE_GLOBAL.md
START_HERE.md
README.md
Makefile

# Все политики, шаблоны, скрипты, поправки
policies/**
templates/**
scripts/**
corrections/**

# Карта слоёв
guides/LAYER_RESPONSIBILITIES.md

# Архитектурные инварианты и целевая архитектура — в overlay и в продуктовых repo
*architecture-invariants.md
*target-architecture.md
```

## Что НЕ в списке (намеренно)

- `OPERATING.md`, `CURRENT_STATE.md`, `NEXT_ACTIONS.md`, `KNOWN_ISSUES.md`, `STATUS_RU.md`, `PROJECT_MAP.md` — оперативные снимки состояния, обновляются часто, blocking signature была бы трением без пользы.
- `TASKS/`, `HANDOFFS/` — конкретные задачи и передачи, их сами по себе править — обычная работа.
- Продуктовый код `sitka-services/`, `astro/services/...` — это работа техлида соответствующего проекта, не методология системы.

## Будущая итерация

Сейчас защита — только на стороне `accept-task`. Прямые `git commit` в эти файлы технически не блокируются. Hard guard (`pre-commit` hook локально + защита на удалённом хранилище когда придёт время публиковать репо) — отдельная итерация. Для двоих разработчиков текущей защиты плюс периодической сверки (`policies/AUDIT.md`) достаточно.
