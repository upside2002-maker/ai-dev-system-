# Режимы выполнения задач

Режим (`Mode:` в шапке TASK) определяет **объём процесса** — сколько ceremony нужно вокруг задачи. Класс риска (`Risk tier:`) определяет **техническую опасность**. Это две независимые оси: tier описывает "что может сломаться", mode — "сколько проверок".

Оба поля обязательны в новой задаче. Mode выбирается явно при создании через `make new-task ... MODE=<mode>`.

## Соответствие класса риска и режима по умолчанию

| Risk tier | Default mode | Когда отклоняться |
|-----------|--------------|-------------------|
| C         | `light`      | `normal` — если задача мелкая по риску, но затрагивает несколько слоёв или модулей |
| B         | `normal`     | `strict` — если задача формально B, но трогает близкие к core invariants участки |
| A         | `strict`     | **никогда** (gate в `accept-task` откажет на A без strict) |
| —         | `preview`    | разведка / спайк / proof-of-concept без acceptance в продукт |

Отклонение от default'а фиксируется в самой задаче (раздел «Контекст») с обоснованием. Без обоснования — это либо подгонка ради скорости, либо перестраховка ради вида (см. `corrections/global-corrections.md` → Correction 012).

## Описание режимов

### `light`
- **Когда**: docs-правки, мелкие mechanical правки кода (rename, comment-only, formatting), Tier C по умолчанию.
- **Кто работает**: Tech Lead inline допустим, если в TASK явно стоит body-marker `Execution isolation: same-session TL inline accepted (Tier C mechanical)`. Иначе — отдельный Worker subagent.
- **Артефакты**: TASK + HANDOFF (короткие). Reviewer не нужен.
- **Цель**: не убивать темп на тривиальном.

### `normal`
- **Когда**: типичная фича, новый endpoint, изменение бизнес-логики без core/math/visual инвариантов. Tier B по умолчанию.
- **Кто работает**: отдельный Worker subagent (cold-start). Reviewer — по усмотрению TL (опционален; обязателен только если задача задевает invariants).
- **Артефакты**: TASK + HANDOFF + опциональный Reviewer report.
- **Цель**: стандартная разработка с одной независимой парой глаз — Worker отдельно от TL.

### `strict`
- **Когда**: миграции, money-flow, security boundaries, изменения схемы БД, ledger, auth, payment, любая правка с явным инвариантом в `architecture-invariants.md`. Tier A — только этот режим.
- **Кто работает**: отдельный Worker subagent + **отдельный** Reviewer subagent (две разные сессии, не одна и та же).
- **Артефакты**: TASK + HANDOFF Worker→TL + HANDOFF Reviewer→TL (файлом, не stdout). Evidence rule обязателен (git short hash для всех ссылок).
- **Цель**: никаких same-session shortcut'ов на коде, который может незаметно сломать продукт.

### `preview`
- **Когда**: разведка, спайк, proof-of-concept, ad-hoc эксперимент. Результат **не вливается** в основной код напрямую — он становится basis для последующего нормального TASK'а.
- **Кто работает**: TL inline или Worker — на усмотрение TL.
- **Артефакты**: TASK + короткий HANDOFF с findings. Тестов и acceptance может не быть; явно отмечается «preview, не для merge как есть».
- **Цель**: посмотреть, как лучше, без полного процесса; отделить «подумать» от «сделать».

## Запреты

1. **Tier A без `Mode: strict`** — `accept-task` откажет. Нет способа принять Tier A в light/normal/preview.
2. **Понижать mode ради скорости** — запрещено. Если задача формально B, но TL хочет `light` потому что «простая» — это маскировка риска. Либо она правда C (тогда обоснуй пере-tier'инг в Контексте), либо она B и нужен normal.
3. **Завышать mode из перестраховки** — нежелательно. `strict` на тривиальной docs-правке не делает её безопаснее, но удваивает время каждого участника. Mode имеет смысл только если выбран по таблице, не "на всякий случай".
4. **Менять mode после `Status: review`** — нельзя без явного rejection и пересоздания TASK. Mode — это контракт на объём процесса, изменение задним числом ломает аудит.
5. **Менять код динамическим сценарием (Dynamic Workflows / параллельные помощники) в обход процесса** — запрещено. Помощники внутри сценария авто-одобряют свои правки (`acceptEdits`) **независимо** от выбранного здесь mode'а, поэтому сценарии разрешены только на чтение/анализ; любой результат, меняющий код, проходит обычный жизненный цикл (TASK → Worker → Reviewer → одобрение). См. `corrections/global-corrections.md` → Correction 019.
