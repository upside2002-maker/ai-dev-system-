# Периодическая сверка системы разработки

Раз в две-три недели владелец проходит по чек-листу, чтобы убедиться, что система разработки не разъехалась с реальностью. Это страховка от ситуации «правила в файлах работают, а исполнение хромает» — даже если все скрипты-проверки зелёные, ручная сверка ловит drift, который не виден автоматике.

## Когда запускать

- По календарю — раз в две-три недели.
- После большого блока работ — когда закрыта серьёзная задача, завершилась фаза, прошёл крупный merge.

Что раньше наступит. Если оба триггера сработали одновременно — одна сверка покрывает оба.

## Кто запускает

Только владелец (email с `can_approve_critical: yes` в `policies/USERS.md`). Второй разработчик не может запустить `make new-audit-task` — скрипт откажет с понятным сообщением. Это не недоверие, а единообразие: сверка — операция контроля над методологией, симметрична подписи серьёзных задач. Тот же флаг прав.

## Как запускать

```
make new-audit-task SLUG=sitka-office
make new-audit-task SLUG=astro
```

Скрипт создаст задачу класса C режима light со стандартной шапкой и чек-листом пунктов (a)-(e). Дальше владелец проходит по чек-листу руками (с помощью `git log`, `make check`, `make context` — что нужно), записывает находки прямо в задачу в раздел Acceptance, потом запускает `make accept-task FILE=<task>` чтобы зафиксировать факт проведённой сверки.

## Чек-лист сверки

Готовый, не каждый раз с нуля. Пять пунктов.

### (a) Подписи на серьёзных задачах за период

Цель: проверить что все принятые за период задачи, затрагивающие критичные пути из `policies/CRITICAL_PATHS.md`, имели подпись владельца.

Как делать:

```
git log --since="3 weeks ago" --format='%h %ae %s' \
  -- 'project-overlays/*/TASKS/archive/'
```

Для каждого закрытого TASK (Status: done, ушёл в `archive/`) проверить:

1. Есть ли в шапке `Critical approved by:` с email из `policies/USERS.md` с `can_approve_critical: yes`.
2. Если поля нет — `Created by:` должен быть владельцем (тогда подпись не требуется по правилу из `policies/SHIFTS.md`).
3. Если ни то, ни другое — записать как находку.

### (b) Авторство прямых правок критичных путей

Цель: ловить ситуации, когда кто-то правил файл из `policies/CRITICAL_PATHS.md` напрямую коммитом, без задачи и без подписи. Технически это не блокируется, но видно в `git log`.

Как делать — для каждого паттерна:

```
git log --since="3 weeks ago" --format='%h %ae %s' \
  -- ROLE_MODEL.md CLAUDE_GLOBAL.md START_HERE.md README.md Makefile
git log --since="3 weeks ago" --format='%h %ae %s' \
  -- policies/ templates/ scripts/ corrections/
git log --since="3 weeks ago" --format='%h %ae %s' \
  -- guides/LAYER_RESPONSIBILITIES.md
```

Для каждого коммита убедиться, что автор — email с `can_approve_critical: yes`. Если нет — нарушение, записать в задачу.

### (c) Соответствие CURRENT_STATE.md фактическому HEAD продуктового repo

Цель: ловить drift между overlay-snapshot'ом и фактическим состоянием кода.

Как делать:

```
make check
```

`scripts/check-overlay-consistency.sh` уже сравнивает Snapshot commit в `CURRENT_STATE.md` с актуальным HEAD продуктового repo (для активных overlay'ев). Если расхождение — `make check` падает.

Дополнительно глазами:

```
make context SLUG=sitka-office
make context SLUG=astro
```

Прочитать выводы, особенно секции STATUS_RU и блок «Кто на смене». Если в STATUS_RU описана старая ситуация (старше двух недель) — drift, записать.

### (d) Запрещённые слова в OPERATING.md и STATUS_RU.md

Цель: проверить что оперативные файлы статуса не накопили жаргон. Self-check-handoff проверяет только передачи; OPERATING и STATUS_RU проверяются здесь руками.

Как делать:

```
grep -i -nE '\b(handoff|task|worker|reviewer|verdict|blocker|fixture|pytest|CI|Tier|Mode)\b' \
  project-overlays/*/OPERATING.md \
  project-overlays/*/STATUS_RU.md
```

(Полный список — `policies/OPERATOR_LANGUAGE.md`. Здесь сокращённый для быстрой сверки.)

Любое срабатывание — посмотреть глазами. Если слово в backticks или в техническом контексте (внутренний раздел для админа) — ок. Если в адрес пользователя — записать как находку.

### (e) Здравомыслие — глазами

Цель: финальный взгляд опытным глазом, не сводимый к скрипту.

Прочитать `OPERATING.md` и `STATUS_RU.md` обоих overlay'ев глазами. Задать три вопроса:

1. Если бы я сейчас был вторым разработчиком и зашёл в проект впервые, понял бы я что происходит и за что браться?
2. Есть ли в активных задачах висяки старше двух недель без движения? Если есть — нужны ли они вообще?
3. Есть ли противоречия между разными файлами? Например, STATUS_RU говорит «деплой готов», а OPERATING.md показывает блокер.

Записать что увидел.

## Что не делает сверка

- **Не правит** файлы методологии. Если в (a)-(e) нашлась проблема — заводится отдельная задача с обычной подписью. Сверка только описывает находку.
- **Не отменяет** скрипты-проверки. Это страховка поверх автоматики, не замена ей.
- **Не покрывает продуктовый код.** Это методология. Продуктовый drift и архитектура — отдельные задачи в overlay'ях.

## Что делать с находками

Каждая находка идёт одним из трёх путей:

1. **Тривиальная** (опечатка, явный устаревший факт) — fix в той же сверке после её закрытия, отдельной мелкой задачей.
2. **Системная** (паттерн нарушений, повторяется) — отдельная задача с подписью владельца, обновление `policies/CRITICAL_PATHS.md` или `corrections/global-corrections.md`.
3. **Архитектурная** (что-то в самой методологии не работает) — обсуждение с владельцем, изменение `ROLE_MODEL.md` / `CLAUDE_GLOBAL.md`.

## Связанные файлы

- `templates/AUDIT_TASK_TEMPLATE.md` — заготовка задачи на сверку.
- `scripts/new-audit-task.sh` — создаёт задачу по шаблону, проверяет право `can_approve_critical`.
- `policies/CRITICAL_PATHS.md` — список критичных путей (используется в (a), (b)).
- `policies/USERS.md` — права (используется в (a), (b) + проверка при запуске сверки).
- `policies/OPERATOR_LANGUAGE.md` — запрещённые слова (используется в (d)).
