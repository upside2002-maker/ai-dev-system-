# Пользователи системы разработки

Список людей, которые работают над проектами, и их права.

Файл читают скрипты `scripts/take-shift.sh`, `scripts/accept-task.sh`, `scripts/approve-critical.sh`, `scripts/new-audit-task.sh` — отсюда они узнают кто имеет право забрать чужое ведение, подписать изменение в зоне обязательной подписи и запустить периодическую сверку. Формат записи строгий: email на отдельной строке, поля `can_*` — каждое на своей строке.

## Поля

- `email` — рабочий email. По нему скрипты различают одного человека от другого. Email берётся из `git config user.email` в момент действия.
- `role` — описание роли в команде. **Только для людей**, скрипты это поле не читают — права считаются по флагам `can_*`. Помогает читать файл глазами.
- `can_override` — может ли забирать чужое активное ведение (экстренное прерывание). Сейчас только владелец.
- `can_approve_critical` — может ли подписывать изменения в зоне обязательной подписи (методологические файлы, описание ролей, политики, шаблоны, скрипты системы — полный список в `policies/CRITICAL_PATHS.md`). Это же право даёт возможность запустить периодическую сверку (`make new-audit-task`).
- `can_deploy` — может ли выкатывать в продакшен.
- `notes` — короткая заметка для человека, читающего файл. Скрипты не читают.

## Правила формата

- Email пишется без backticks, в одну строку после `- email:`. Скрипты ищут по подстроке после двоеточия.
- Поля `can_*` — `yes` или `no`, всё нижним регистром. Скрипты сравнивают строго эти значения.
- Поля `role` и `notes` — свободный текст, не парсятся скриптами. Их можно править как угодно, не сломав логику.
- Не вставляй пустые строки внутри блока одного пользователя — скрипты читают блок до следующего заголовка `###`.
- Новый пользователь — новый блок `### Имя` с тем же набором полей.
