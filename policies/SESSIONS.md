# Политика: модель постоянных чатов

Дата принятия: 2026-05-20. Owner: Илья (`upside2002-maker`).

## Что это и зачем

До этой политики Owner для каждой задачи открывал новый Claude-чат,
вручную переносил промпт в нужную роль, потом копировал результат
обратно. К 2026-05-20 накопилось ~10 параллельных чатов, из которых
реально использовалось 2-3. Диспетчеризация съедала внимание вместо
продукта.

Эта политика фиксирует переход на модель **«три постоянных чата»**.
Новые вкладки под каждую задачу больше не открываются. Контекст
синхронизируется через файлы в репозитории, агенты сами читают
свежее состояние при возврате Owner'а в чат.

## Три постоянных чата

| Чат | Роль (по `ROLE_MODEL.md`) | Запускается в каталоге | Чем владеет |
|-----|---------------------------|------------------------|-------------|
| 1. Admin | `AI Dev System Admin` (§5) | `~/Projects/ai-dev-system` | методология, диспетчеризация, координация TL'ов, аудиты |
| 2. TL Sitka | `Project Tech Lead` (§1) | `~/Projects/sitka-office` | продукт Sitka, lifecycle TASK/HANDOFF, deploy |
| 3. TL Astro | `Project Tech Lead` (§1) | `~/Projects/astro` | продукт Astro, lifecycle TASK/HANDOFF, цикл с Мариной |

Других постоянных чатов нет. Worker / Reviewer / BA — это
**subagent'ы внутри** одного из трёх постоянных чатов (запускаются
TL'ом или Admin'ом по необходимости), а не отдельные вкладки.

## Общая доска

`DISPATCHER.md` в корне `ai-dev-system` — единая страница которую
читают все три роли. Обновляется любой ролью когда меняется её
статус.

Что там лежит:
- кто из трёх ролей сейчас активен и над чем;
- план Owner'а на сегодня (если есть);
- открытые блоки (что-то ждёт от кого-то);
- 5 последних cross-role событий.

Доска — это **информация о происходящем**, не команды.

## Почтовые ящики

`MAILBOX/` в корне `ai-dev-system`. Три накопителя:

- `MAILBOX/to-admin.md` — TL'ы пишут сюда отчёты и эскалации.
- `MAILBOX/to-tl-sitka.md` — Admin пишет сюда задачи для TL Sitka.
- `MAILBOX/to-tl-astro.md` — Admin пишет сюда задачи для TL Astro.

Формат записок — append-only, с timestamp + автор. Точный шаблон —
в `MAILBOX/README.md`.

Прочитанные сообщения **не удаляются**: переезжают в
`MAILBOX/archive/<YYYY-MM>.md` той ролью которая их обработала. Audit
trail остаётся в git.

## Протокол «обнови контекст»

Owner запускает обновление **обычной фразой**, не командой:

- «обнови контекст»
- «что нового»
- «как дела»
- «давай по статусу»
- любое первое сообщение в чате после паузы (≥30 минут)

Что агент делает на эту фразу:

1. Читает `DISPATCHER.md` целиком.
2. Читает свой почтовый ящик (`to-<моя-роль>.md`) — собирает все
   неархивированные сообщения.
3. Делает `git -C <repo> log origin/master -5` своего проекта чтобы
   увидеть свежие коммиты.
4. Если в overlay есть active TASKS / HANDOFFS — проверяет их статус.
5. Сообщает Owner'у короткое summary: что нового на доске, что в
   ящике, что я предлагаю делать дальше.

Никакого `Read` всего overlay сверху донизу — только дельта с
прошлого раза.

## Как Admin ставит задачи TL

В чате Admin Owner пишет фразой: «передай TL Sitka задачу X». Admin:

1. Формулирует задачу в формате записки (см. `MAILBOX/README.md`).
2. Append'ит в `MAILBOX/to-tl-sitka.md`.
3. Обновляет соответствующую строку в `DISPATCHER.md` («TL Sitka:
   ждёт запуск задачи X»).
4. Сообщает Owner'у «записал, переключайся в TL Sitka».

Owner переключается в чат TL Sitka, пишет «обнови контекст» (или
любую другую фразу-триггер). TL Sitka подтягивает почту, видит
новую задачу, начинает работу по обычному TASK lifecycle.

## Как TL возвращает результат

TL закончил работу. В чате TL Sitka он:

1. Завершает TASK по обычному lifecycle (HANDOFF / merge / overlay
   refresh — как было до этой политики).
2. Append'ит короткую записку в `MAILBOX/to-admin.md`: «задача X
   закрыта, PR #Y merged, прод healthy».
3. Обновляет `DISPATCHER.md` (вычёркивает задачу из «ждёт запуск»,
   добавляет в «свежие события»).
4. Перемещает прочитанные сообщения из своего ящика в `archive/`.

Owner возвращается в чат Admin, пишет «обнови контекст». Admin
читает `to-admin.md`, видит отчёт, докладывает Owner'у. Если Owner
говорит «принимаю» — Admin архивирует сообщение.

## Защита от деградации long-lived чатов

В одной Claude-сессии context window ограничен. На длинных дистанциях
полезно периодически чистить short-term память агента без потери
state мира.

Три механизма:

- **Лёгкий рефреш** — на фразу «обнови контекст» агент делает
  pull-from-files (см. протокол выше). Старая переписка остаётся.
- **`/compact`** — раз в день в каждом чате (Owner запускает явной
  командой). Сжимает разговор, оставляя ключевые факты.
- **Чистый старт** — на фразу Owner'а «начнём с чистого листа»
  агент:
  1. Пишет короткое резюме текущей сессии в
     `MAILBOX/archive/<role>-<YYYY-MM-DD>-conversation-summary.md`
     (на случай если завтра придётся посмотреть).
  2. Просит Owner'а сделать `/clear` в чате.
  3. После `/clear` — Owner пишет любую фразу-триггер, агент делает
     полный refresh с нуля (как при первом запуске).

`/clear` — это единственная «команда» которую Owner вообще набирает.
Это встроенная команда Claude Code, не наша. Всё остальное —
обычные фразы.

## Что **не** меняется

- Lifecycle scripts (`accept-task.sh`, `submit-task.sh`,
  `accept-handoff.sh`, etc.) — те же самые, работают как было.
- Worker / Reviewer / BA — те же самые subagent'ы, запускаются TL'ом
  из своего чата.
- `policies/SHIFTS.md` — ведение проекта остаётся бессрочным, в
  чате TL Sitka только активный держатель ведения пишет правки.
- `policies/CRITICAL_PATHS.md` — owner signature остаётся требованием
  для критичных областей.
- Git как источник истины — все правки идут через коммиты, доска и
  почтовые ящики тоже коммитятся.

## Что **меняется**

- Owner больше не открывает новый чат под каждую задачу.
- Owner не копирует промпты между чатами руками.
- Admin расширен функцией «диспетчер по поручению Owner'а» — см.
  `ROLE_MODEL.md` §5.
- Появились `DISPATCHER.md` и `MAILBOX/` — единый канал
  cross-role-коммуникации.
- Появилась эта политика — как единый источник правды по модели.
