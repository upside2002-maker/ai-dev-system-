# Working With Typed Core

Этот playbook нужен для проектов, где Haskell или другой typed FP-язык используется как ядро.

## Когда typed core оправдан

- деньги и валюты;
- статусы жизненного цикла;
- ценообразование;
- инварианты, которые нельзя терять при росте проекта;
- зоны, где ошибка дорого стоит бизнесу.

## Базовые правила

1. Инварианты должны жить в типах или чистых функциях.
2. Доменные ADT не должны растворяться на границе БД или JSON без необходимости.
3. Бизнес-логика должна быть чистой и тестируемой без IO.
4. IO — только на краях: handlers, DB, HTTP, queues.

## Что особенно важно для AI-агента

Агенту на typed core нельзя давать расплывчатые задачи вроде:

- "добавь еще логику котировок";
- "расширь сделки";
- "сделай как правильно".

Нужно давать:

- конкретный модуль;
- конкретный новый инвариант;
- явную форму API или нового типа;
- пример входа/выхода.

## Красные флаги

- raw `Double` или аналог для денег;
- ручные `Text <-> ADT` конверсии без исчерпывающего matching;
- дублирование `fetch-or-404` логики;
- состояние, которое можно положить в state machine, но оно разъехалось по слоям;
- попытка засунуть orchestration в core.

## Рекомендованный цикл

1. Начинать с типов.
2. Потом state transitions.
3. Потом persistence boundary.
4. Потом API handler.
5. Потом тесты на happy path и forbidden transitions.

## Для Sitka-подобных проектов

Core должен владеть:

- сделкой;
- quote;
- transitions;
- pricing settings;
- risk flags;
- финальными статусами и аудитом.
