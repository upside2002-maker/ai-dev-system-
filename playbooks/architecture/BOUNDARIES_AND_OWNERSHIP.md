# Boundaries And Ownership

## Базовая схема для сложного продукта

```text
typed core -> integration/services -> frontend/operator UI
```

## Кто чем владеет

### Core

Владеет:

- типами;
- правилами;
- переходами состояний;
- расчетом денег;
- аудитом;
- каноническими записями.

### Services

Владеет:

- запросами к внешним сайтам;
- browser automation;
- парсингом и scraping;
- очередями;
- glue-code между внешними системами и core.

### Frontend

Владеет:

- представлением;
- последовательностью действий менеджера;
- локальными UI-состояниями;
- понятностью и скоростью операционной работы.

## Анти-паттерны границ

- frontend сам решает, какой статус "на самом деле";
- services считают бизнес-маржу вместо core;
- core начинает знать про браузерный парсер;
- pricing или transitions расползаются по трем слоям.

## Приоритет при споре о границах

1. Данные и инварианты идут в core.
2. Шум и внешняя нестабильность идут в services.
3. Скорость и ясность работы человека идут в frontend.
