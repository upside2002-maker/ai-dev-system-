# Service Integration Playbook

Этот playbook нужен для Python/FastAPI слоя, который соединяет core с внешним миром.

## Ответственность слоя

Services-слой:

- ходит наружу;
- приводит шумные внешние данные к аккуратному виду;
- выдерживает ошибки внешних систем;
- вызывает core по контракту;
- не принимает доменные решения вместо core.

## Хорошие паттерны

- Pydantic для transport schemas;
- нормализация строк, URL и store names;
- dedupe на уровне services;
- явные fallback-ветки;
- перевод ошибок lower-level слоя в понятные HTTP-ошибки;
- health endpoint с зависимостями, а не просто `"ok"`.

## Плохие паттерны

- писать final business rules в Python;
- silently swallow critical errors без trace;
- смешивать реальные и mock-результаты без маркировки;
- импортировать в core все подряд без фильтрации и dedupe.

## Для sourcing-потока

Правильный flow:

1. собрать результаты по магазинам;
2. нормализовать и дедуплицировать;
3. отдельно собрать Avito или mock Avito;
4. вернуть агрегированный pre-deal result;
5. только потом создавать сделку из выбранных результатов.

## Проверка готовности

Перед приемкой services-задачи нужно убедиться:

- endpoint отвечает быстро и предсказуемо;
- parser failure не валит весь сервис;
- core failure превращается в понятный 502/4xx, а не в stack trace наружу;
- mock/fallback помечены явно.
