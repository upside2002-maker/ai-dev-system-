# Agent Operating Model

## Роли

### 1. Core Agent

Отвечает за:

- доменные типы (Domain/*.hs);
- state machine (Domain/Deal/StateMachine.hs);
- pricing/risk logic (Engine/Pricing.hs);
- API-контракты (Api/*.hs);
- DB schema и миграции (Db/Schema.hs);
- PersistField instances и type safety на границах.

Не отвечает за:

- парсинг сайтов;
- браузерную автоматизацию;
- UI-поведение.

### 2. Services Agent

Отвечает за:

- FastAPI routes и middleware;
- внешние парсеры (Avito, магазины);
- orchestration и wiring;
- нормализацию и dedupe внешних данных;
- graceful degradation и fallback;
- Telegram bot, внешние API.

Не отвечает за:

- финальные бизнес-решения по деньгам и состояниям;
- хранение истины о lifecycle сделки.

### 3. Frontend Agent

Отвечает за:

- операторские сценарии и workflow;
- минимизацию кликов и когнитивной нагрузки;
- API-клиент (src/api/client.ts, types.ts);
- UX вокруг уже существующих контрактов.

Не отвечает за:

- расчёт цен;
- бизнес-логику статусов;
- обход серверных ограничений через локальную логику.

## Цикл работы агента

```
1. Прочитать CLAUDE.md              ← правила
2. Прочитать .claude/corrections.md ← ошибки предшественников
3. Прочитать overlay CURRENT_STATE  ← где мы сейчас
4. Определить слой ответственности
5. Взять playbook для слоя
6. Выполнить минимально достаточное изменение
7. Проверить build/test/lint в своём слое
8. Зафиксировать correction если ошибка повторяемая
9. Обновить overlay если состояние проекта изменилось
```

## Декомпозиция задач

Правильная декомпозиция — один слой, конкретные файлы:

- "добавь PersistField instance для USD в Domain/Types.hs и обнови Db/Schema.hs"
- "подключи Telegram Bot API в app/integrations/telegram.py"
- "переделай DealWorkspace чтобы показывал один next action"

Плохая декомпозиция — размытые цели, несколько слоёв:

- "сделай весь flow Telegram уведомлений"
- "переделай архитектуру"
- "исправь все баги"

## Формат задачи для агента

```
Слой: [core | services | web]
Файлы: [конкретные пути]
Задача: [что сделать]
Не трогать: [что не менять]
Проверка: [stack build | pytest | tsc]
Контекст: [ссылка на correction/issue если есть]
```

## Мультиагентная координация

Когда несколько агентов работают параллельно:

1. **Один модуль — один агент.** Нельзя одновременно менять один файл.
2. **Контракты фиксируются до параллельной работы.** Если Core Agent меняет API type — Frontend Agent ждёт.
3. **Overlay — общий source of truth.** Каждый агент обновляет свою часть CURRENT_STATE после работы.
4. **Corrections — общие.** Любой агент может добавить запись.

## Правило роста сложности

Чем сложнее проект, тем важнее:

- уменьшать поверхность изменений за одну задачу;
- усиливать проверяемость на границах между слоями;
- хранить опыт в corrections, а не в памяти чата;
- не давать агенту задачу "сделай feature целиком" — дробить по ownership.
