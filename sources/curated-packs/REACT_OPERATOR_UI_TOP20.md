# React Operator UI Top 20

Этот файл описывает 20 паттернов для операторского интерфейса, а не для "демо-фронтенда".

Базовые источники для отбора:

- `react.dev`
- TypeScript Handbook
- `TanStack/query`

## 1. UI должен отвечать на вопрос "что делать сейчас"

Когда использовать:

- в любом CRM/workspace UI.

Хорошая форма:

- один primary action;
- явный current stage;
- next step понятен без чтения всех блоков.

Избегать:

- UI как витрина внутренних модулей.

## 2. Разделять pre-deal, active deal и post-quote

Когда использовать:

- sourcing + sales workflow.

Хорошая форма:

- разные режимы интерфейса;
- разный визуальный приоритет;
- разная терминология.

Избегать:

- сваливать все стадии в одну бесконечную колонку.

## 3. State placement по смыслу

Когда использовать:

- React state design.

Хорошая форма:

- локальное UI состояние локально;
- серверные данные отдельно;
- derived state считать, а не хранить лишний раз.

Избегать:

- держать всё в одном гигантском `useState`.

## 4. Server state != local UI state

Когда использовать:

- lists, detail, health, async actions.

Хорошая форма:

- данные с сервера и их refresh logic отделены от баннеров, модалок и выделения.

Избегать:

- смешения данных и мелкого UI-состояния в один объект.

## 5. Banner/feedback как отдельный короткий канал

Когда использовать:

- успехи и ошибки после действий.

Хорошая форма:

- transient banner;
- понятный текст;
- легко закрыть.

Избегать:

- молчаливых фейлов;
- error dump на полэкрана.

## 6. Одна сущность — один фокусный workspace

Когда использовать:

- когда выбрана сделка.

Хорошая форма:

- sidebar для выбора;
- workspace для работы;
- context panel для дополнительной информации.

Избегать:

- дублировать данные о сделке в трех местах одновременно.

## 7. Context спрятан, action наверху

Когда использовать:

- офферы;
- история;
- Avito references;
- pricing details.

Хорошая форма:

- сначала current step;
- потом supporting context.

Избегать:

- заставлять менеджера прокапываться через историю ради основного действия.

## 8. Технические слова переводить в операторский язык

Когда использовать:

- `quoted`, `sourcing`, `in_stock`, `risk_flags`.

Хорошая форма:

- "КП готово";
- "Ждем решение клиента";
- "В наличии";
- "Есть риск".

Избегать:

- внутренней серверной терминологии на основном экране.

## 9. Primary / secondary / destructive action hierarchy

Когда использовать:

- кнопки в карточке сделки.

Хорошая форма:

- один primary button;
- secondary рядом;
- destructive визуально отдельно.

Избегать:

- пять равносильных кнопок подряд.

## 10. Выбор до создания

Когда использовать:

- create deal from sourcing;
- build quote from offers.

Хорошая форма:

- сначала выбрать пригодные данные;
- потом нажать commit action.

Избегать:

- необратимых действий без явного отбора.

## 11. Async flows должны быть видны

Когда использовать:

- поиск;
- создание сделки;
- transition;
- quote calculation.

Хорошая форма:

- loading state рядом с action;
- заблокированная кнопка;
- понятный текст ожидания.

Избегать:

- "ничего не происходит, наверное сломалось".

## 12. Не делать screen-length форму без смысловых секций

Когда использовать:

- длинные карточки CRM.

Хорошая форма:

- смысловые секции;
- sticky or high-visibility current action zone;
- collapsible secondary blocks.

Избегать:

- бесконечных полотен вниз.

## 13. Selected entity должна быть очевидна

Когда использовать:

- list/detail patterns.

Хорошая форма:

- активный deal/client виден сразу;
- нет сомнений, с чем сейчас работает оператор.

Избегать:

- "нажал где-то слева, а справа непонятно что обновилось".

## 14. Нельзя заставлять помнить правила системы

Когда использовать:

- transitions;
- quote readiness;
- sourcing readiness.

Хорошая форма:

- UI сам подсказывает, можно ли действие;
- объясняет, почему нельзя.

Избегать:

- скрытых правил в голове пользователя.

## 15. Derived summaries above detailed data

Когда использовать:

- offers;
- quotes;
- history.

Хорошая форма:

- сначала summary: "4 оффера, 2 в наличии, 1 КП".

Избегать:

- сразу длинный список без агрегированной картины.

## 16. Данные для решения клиента отдельно от технических деталей

Когда использовать:

- quote stage.

Хорошая форма:

- цена, маржа, текущий stage и decision buttons вместе;
- сырой offer context ниже.

Избегать:

- прятать главное коммерческое решение среди техданных.

## 17. New request flow не должен конкурировать с active deal flow

Когда использовать:

- CRM, где есть входящий поток и уже активные сделки.

Хорошая форма:

- отдельный режим "Новый запрос";
- clear exit back to deal mode.

Избегать:

- когда создание новой заявки визуально спорит с текущей сделкой.

## 18. Фронтенд не должен "угадывать" truth

Когда использовать:

- status badges;
- quote availability;
- next action hints.

Хорошая форма:

- UI опирается на contracts и derived facts;
- не додумывает бизнес-смысл в обход сервера.

Избегать:

- hardcoded business rules scattered across components.

## 19. Композиция, а не один monstrous component

Когда использовать:

- growing workspace UI.

Хорошая форма:

- `InboxSidebar`
- `DealWorkspace`
- `ContextPanel`

Избегать:

- один `App.tsx` на тысячу строк логики и верстки.

## 20. UI для менеджера измеряется временем до действия

Когда использовать:

- всегда.

Хорошая форма:

- меньше кликов;
- меньше чтения;
- меньше переключений внимания;
- меньше сомнений "что делать дальше".

Избегать:

- красивой, но медленной операционки.

## Как использовать этот top-20

Перед задачей на frontend агенту стоит давать:

1. project overlay
2. `CRM_WORKFLOW_UI.md`
3. этот `Top 20`
4. конкретный сценарий менеджера в одну фразу

Например:

"Клиент написал, менеджеру нужно быстро собрать варианты, создать сделку и отправить КП".
