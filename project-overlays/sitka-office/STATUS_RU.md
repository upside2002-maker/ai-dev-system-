## Сейчас

Продакшен живой на сервере `94.72.112.106:8088`. Калькулятор себестоимости и цены клиенту во вкладке «Парсер»: 7 полей с дефолтами из настроек (добавилось «Буфер курса, %» — по умолчанию 5%), breakdown в 7 строк, эффективный курс конвертации считается как банковский × (1 + буфер/100) и используется во всех пересчётах USD→рубли. Текст для копирования клиенту буфер не раскрывает — внутренняя кухня.

Сегодня починена давняя засада парсера: теперь если ищешь «Blizzard parka», система реально находит и «Blizzard AeroLite Parka» $538.30, и «Sitka Blizzard Pro Parka» — раньше она капризничала, требовала чтобы оба слова стояли в названии магазина точно подряд (а они там стоят с разделителем), и выкидывала валидные товары. Запросы из одного слова работали по-прежнему. Теперь и из двух работают как ожидаешь — два слова, оба должны встретиться в названии, без требования к порядку. Параллельно я провёл архитектурный обзор парсера в целом — отчёт лежит в `research/`, в нём есть несколько других хвостов (Amazon почти не работает из-за тайминга, eurooptic просит прокси, для запросов из 3+ слов тот же баг ещё может проявляться). По итогам обзора рекомендация — забрать парсер себе и не зависеть от внешнего автора; это решение твоё, я его не принимаю сам.

В ночь на сегодня я провёл общий аудит обоих проектов и сервера. Главное:

- **Резервные копии базы наконец-то работают.** До сегодняшнего дня их не было вообще — служба была описана в конфигурации, но контейнер ни разу не запускался. Сейчас контейнер поднят, первый дамп лёг (база у нас лёгкая, 19 МБ → файл ~520 КБ), хранится последние 7 дней.
- **База данных закрыта снаружи на постоянной основе.** До сегодняшнего был временный обходной патч (соседи по серверу его поставили в свой аудит), он не пережил бы перезагрузку. Теперь защита встроена в нашу конфигурацию и переживёт что угодно.
- **Добавлен резервный «карман памяти»** на сервере (2 ГБ). Раньше при пиковой нагрузке сервер мог упасть целиком; теперь у него есть страховка.
- **Шаблон серверной конфигурации теперь в коде** (PR #88): если придётся поднимать новый сервер, всё что специфично для боевой среды можно скопировать одной командой, не воссоздавая руками.

В сухом остатке: сайт работает как работал, под капотом стало надёжнее. Ничего из этого пользователь сайта не заметит — это всё инженерные подушки безопасности.

## На очереди

Мелкие задачи, не срочные, копятся в `OPERATING/backlog.md`. Самые ближайшие:

- Исправить значение по умолчанию `localhost:8080` в фронтенд-сборке (3-5 строк, класс C).
- Гонка курсора Avito-опросника при холодном старте — пропускает окно ~60 секунд один раз (10-15 строк, класс A на стороне ядра).
- Зеркалирование серверных правок `docker-compose.override.yml` обратно в репозиторий (~30 строк, класс C), чтобы следующий деплой не повторял их вручную.
- Чёрновая задача `TASKS/2026-04-29-dm7-c-backend-widget-prereq.md` — расширение `Api.TransactionResp` для будущей разбивки кассы по типам расходов. Не срочна, нужна только когда оператор реально захочет разбивку. **Отложенный платёж по новой системе режимов:** задача помечена как класс A, но в шапке нет поля `Mode:`. Перед возвращением её в работу нужно будет дописать `Mode: strict` (или явно понизить класс с обоснованием в разделе «Контекст»). До этого — задача стоит черновиком; если попытаться её принять сейчас, проверка на стороне скрипта `accept-task` откажет с понятным сообщением.
