## 1. TL;DR

**ФАКТ.** В каталоге Avito 23 публичных API. Для Sitka релевантны ~12. Покрывают: мессенджер, объявления, спенд/аналитику, отзывы, заказы, автозагрузку, продвижение.

**ФАКТ.** Аутентификация: OAuth 2.0. Два flow — `authorization_code` (от имени пользователя) и `client_credentials` (под свой аккаунт). Endpoint токена: `https://api.avito.ru/token`.

**ФАКТ.** Messenger API требует подписку «Расширенный» или «Максимальный» (явно написано в «Общей информации» раздела Мессенджер). Остальные API такого явного ограничения не декларируют — но по практике доступ к части методов требует профи-тарифа.

**ВЫВОД.** API закрывает 80% архитектурных задач из `ARCHITECTURE_V2.md` (Phase 2 Inbox, Phase 5 Spend Sync, частично Phase 1 Marketing Analytics). Создание объявлений через обычный CRUD НЕ поддерживается — только через Autoload (XML-фид).

## 3. Детализация по релевантным API

### 3.1 Messenger API — приоритет ⭐⭐⭐ (Phase 2)

**ФАКТ (из раздела «Общая информация»).** Требует подписку «Расширенный» или «Максимальный».

**ФАКТ.** Scopes: `messenger:read`, `messenger:write`.

**ФАКТ. Методы** (из списка H2 секции Messenger API):
- Отправка текстового сообщения — `POST /messenger/v1/accounts/{user_id}/chats/{chat_id}/messages`
- Отправка сообщения с изображением
- Удаление сообщения
- Прочитать чат (mark as read)
- Получение голосовых сообщений
- Загрузка изображений
- Получение информации по чатам (список чатов)
- Получение информации по чату
- Получение списка сообщений **V3**
- Включение уведомлений **V3 (webhooks)**
- Получение подписок (webhooks)
- Отключение уведомлений (webhooks)
- Добавление пользователя в blacklist

**ФАКТ.** Есть поддержка чат-ботов Avito через поле `type: "system"` + `flow_id`. Известные боты:
- `seller_audience_discount` — персональное предложение от продавца
- `sbc_seller_notification` — уведомление о принятии предложения
- `flower_161071` — автоответы на частые вопросы

**ФАКТ.** На уровне прав: если ключ компании — работают все методы; если ключ сотрудника — список чатов возвращает только свои объявления, а чтение/отправка/блэклист/изображения по чужим объявлениям = ошибка.

**ГИПОТЕЗА.** Webhooks V3 дают push-модель вместо polling. Значит sync-job каждые 15 минут (план из `ARCHITECTURE_V2.md`) можно заменить на real-time webhook в services → создание `ConversationThread` и `conversation_message` сразу.

**ВЫВОД.** Phase 2 Inbox реализуем как webhook-first (fallback на 15-мин polling при падении вебхука). Messenger API 1-в-1 ложится на `ConversationProvider` абстракцию из overlay. Но прежде нужен платный тариф — это уже в NEXT_ACTIONS пункт #8.

### 3.2 Item API — приоритет ⭐⭐⭐ (Phase 1)

**ФАКТ. Методы:**
- Получение информации по объявлению (и по списку)
- Получение статистики по звонкам
- Получение информации о стоимости услуг продвижения и доступных значках
- Применение дополнительных услуг / пакета услуг
- Применение услуг продвижения
- Получение статистики по списку объявлений
- **Получение статистических показателей по профилю**
- **Получение статистики расходов профиля**
- Обновление цены объявления

**ФАКТ.** Scopes: `items:info`, `items:apply_vas`, `stats:read`.

**ФАКТ.** Создания или редактирования объявления (кроме цены) в Item API нет.

**ВЫВОД.** Для Phase 1 Marketing Analytics MVP:
- `статистика расходов профиля` → источник `SpendEntry` с `SpendSource.AvitoApiSync` (закрывает Phase 5 почти сразу).
- `статистика по списку объявлений` → метрики на уровне `Listing` (просмотры, избранное, контакты) → вход в `Engine.Marketing.funnelByChannel` на уровне листинга.
- `статистика по звонкам` → альтернативный канал контакта для attribution.

**ВЫВОД.** Обновление цены через API закрывает пункт плана действий #4 из `MY_AVITO_AUDIT.md` («Заменить условные цены в названиях») — частично, только price field, не title. Title менять придётся через Autoload.

### 3.2.1 Детализация трёх ключевых stat-endpoint'ов Item API

Отдельно разобрал три метода — это основа Phase 1 Marketing Analytics.

#### A) `Получение статистики расходов профиля` (агрегированная сумма)

**ФАКТ.**
- Scope: `stats:read`
- Глубина: **270 дней** назад
- Rate limit: **1 запрос в минуту** (жёсткий)
- Body: `dateFrom`, `dateTo` (YYYY-MM-DD), `grouping` (`day`/`week`/`month`), `spendingTypes` (массив), `filter` (nullable, структура в Swagger)
- Типы трат (spendingTypes):
  - `all` — всё в копейках
  - `promotion` — расходы на продвижение (VAS, визуальные услуги)
  - `presence` — размещение + целевые действия (просмотры / чаты / звонки / отклики)
  - `commission` — комиссия Авито Доставки + бронирование жилья
  - `rest` — чат-боты + неопознанное

**ВЫВОД.** Это **главный источник** для `SpendEntry` в `sitka-core`. Один вызов в сутки = закрывает Phase 5 полностью. Разбивка `promotion / presence / commission / rest` ≈ наш `SpendType` ADT (нужно будет расширить: сейчас у нас `Promotion | Boost | Subscription | OneTime | Other Text`).

#### B) `Получение статистики по списку объявлений` (счётчики событий)

**ФАКТ.**
- До **200 objectIds** за один запрос
- Глубина **270 дней**
- Счётчики: `uniqViews`, `uniqContacts`, `uniqFavorites` (старые `views`, `contacts`, `favorites` deprecated с апреля 2021)
- Примечание: «контакт» = запрос телефона ИЛИ начатый чат ИЛИ отклик на резюме
- `periodGrouping`: `day` (без агрегации) / `week` / `month`
- Уникальность считается в пределах одного дня

**ВЫВОД.** Это feed для `Listing.stats` и для атрибуции: «у листинга X за такой-то день было Y уникальных контактов». Нужно при реализации — не суммировать уникалы между днями (это разные люди технически, но Avito уже считает их уникальными в пределах дня, склейка по неделям/месяцам — их встроенная).

#### C) `Получение статистических показателей по профилю` — **лучший endpoint**

**ФАКТ.** Этот метод фактически **один покрывает весь funnel + spend на уровне профиля и объявления**. Группировки: `totals` / `item` / `day` / `week` / `month` (комбинируются).

**ФАКТ. Показатели (основные):**
- `views` / `contacts` / `favorites`
- Декомпозиция контактов: `contactsShowPhone`, `contactsMessenger`, `contactsShowPhoneAndMessenger`, `contactsSbcDiscount` (отклики на рассылку скидок!)
- **`viewsToContactsConversion`** — конверсия просмотр → контакт, в процентах
- **`averageViewCost`** — средняя цена просмотра (расходы / просмотры)
- **`averageContactCost`** — средняя цена контакта (CPL!)
- `impressions`, `impressionsToViewsConversion` (показ → просмотр)

**ФАКТ. Целевые действия (CPA/CPx):**
- `clickPackages` — целевые просмотры, оплаченные из тарифа
- `jobContacts` — отклики на вакансии (нам неактуально)

**ФАКТ. Расходы (в копейках):**
- `presenceSpending`, `promoSpending`, `restSpending`, `commission`
- `spendingBonus` — списанные бонусы

**ФАКТ. Заказы через Авито Доставку:**
- `orderedItems` / `orderedItemsPrice` — заказанные товары + их сумма
- `deliveredItems` / `deliveredItemsPrice` — принятые заказы

**ФАКТ. Количество объявлений:** `activeItems`, `newActiveItems`, `oldActiveItems`.

**ВЫВОД.** Этот endpoint даёт Phase 1 Marketing Dashboard «бесплатно» — большая часть наших расчётных метрик (CPL, CPD, funnel по stage, ROI по объявлению) уже посчитана Avito. Наш `Engine.Marketing` в этом случае деградирует в простой mapper: Avito response → `MarketingDashboardResp`. Это меняет архитектуру Phase 1 — часть `Engine.Marketing.funnelByChannel` можно не писать совсем для канала Avito, для остальных каналов (TG / WA когда придут) — оставить как есть.

**ГИПОТЕЗА.** Rate limit на этом endpoint не назван явно — нужно проверить на Swagger файле. По аналогии с «статистикой расходов» — возможно тоже 1 RPM.

### 3.3 CPA API — приоритет ⭐⭐⭐ (Phase 1 + Phase 5)

**ФАКТ. Методы:**
- `balanceInfo` — баланс и внесённый за текущий период аванс
- `callsByTime` — список звонков с параметрами по времени (целевые и не-целевые)
- `chatsByTime` — список чатов с параметрами по времени
- `phonesInfoFromChats` — номера телефонов из целевых чатов
- `callById` / `call/{call_id}` — модель/запись звонка
- `chatByActionId/{actionId}` — модель чата по id
- `createComplaint` / `createComplaintByActionId` — опротестовать звонок/чат

**ФАКТ.** Часть методов помечена `deprecated` (старые baseline + запись звонка + `chatsByTime` (deprecated)). Есть свежие аналоги без deprecated-пометки.

**ГИПОТЕЗА.** Прямая связка: каждый «целевой чат» в CPA API → отдельная запись в нашем `ConversationThread` с атрибуцией на конкретное объявление. Номер телефона из `phonesInfoFromChats` — это мостик к существующему `Client.phone` и дедупликация (тот же клиент, пришедший в разные сделки).

**ВЫВОД.**
- CPA API — **лучший** источник spend по CPA-модели: `balanceInfo` даёт реальный аванс, а не прогноз.
- `callsByTime` + `chatsByTime` идеально ложатся на `SourceTouch` (мы знаем: пришёл с конкретного объявления, с таким-то временем).
- `phonesInfoFromChats` закрывает сценарий «клиент ушёл в WhatsApp» — если у нас есть телефон из чата, можем связать WhatsApp-тред с тем же клиентом в `Client`.

### 3.4 Ratings API — приоритет ⭐⭐ (Phase 4)

**ФАКТ. Методы:**
- `/ratings/v1/answers` — ответить на отзыв
- `/ratings/v1/answers/{answer_id}` — удалить ответ
- `/ratings/v1/info` — информация о рейтинге продавца
- `/ratings/v1/reviews` — список опубликованных отзывов с пагинацией

**ФАКТ.** Поддерживает и `AuthorizationCode`, и `ClientCredentials`.

**ГИПОТЕЗА.** Пагинация = можем вытянуть все 101 отзыв за один раз (история разовая, дальше инкрементально).

**ВЫВОД.** Для Phase 4 Knowledge Hub — это готовый источник контент-базы. Схема:
- Services: раз в сутки pull `/ratings/v1/reviews` → таблица `avito_reviews` в Postgres services
- Core: `NoteAdded` event для новых отзывов + ссылка на `avito_review_id`
- Web: новый раздел «Отзывы» в Knowledge с поиском/тегами

Это закрывает пункт 7 из `MY_AVITO_AUDIT.md` («Использовать 101 отзыв как контент-маркетинг»).

### 3.5 Autoload API — приоритет ⭐⭐ (операционная работа)

**ФАКТ. Методы:**
- `getProfile` / `createOrUpdateProfile` — настройки автозагрузки (URL фида, расписание, почта для отчётов)
- Загрузка файла по ссылке
- Получение полей категории / дерева категорий
- ID объявлений из файла / ID объявлений на Авито
- Список отчётов, объявления из конкретной выгрузки, списания за объявления
- Статистика по последней / конкретной выгрузке

**ФАКТ.** Scope: `autoload:reports`.

**ГИПОТЕЗА.** Создание объявлений возможно **только** через XML-фид автозагрузки. Это объясняет, почему у «Импорт маркет SITKA» 310 активных объявлений — они ведут фид в своей системе.

**ВЫВОД.**
- Для Sitka путь к 30-50 активных позициям (Приоритет 3 в аудите) — через собственный XML-фид (генерим в `sitka-services`) + Autoload API для мониторинга статуса и ошибок.
- Это **новая фаза** (назовём Phase 6 — «Avito Autoload pipeline»), пока не приоритет, но закладываем в roadmap.
- 15 неопубликованных сейчас — быстрее опубликовать руками через интерфейс, API тут не ускоряет (нужен готовый фид).

### 3.6.1 Promotion API (новая версия VAS)

**ФАКТ.** Это новая версия API для управления услугами продвижения (старая версия — в разделе Item под «Применение VAS»).

**ФАКТ. Методы:**
- `Словарь типов услуг продвижения` — справочник (scope `items:info`, 100 RPM)
- `Список услуг продвижения` — активные услуги по объявлениям (до 100 itemIds, scope `items:info`, 100 RPM)
- `Список заявок` — заявки на подключение BBIP (scope `items:apply_bbip`)
- `Статус заявки` — статус конкретной заявки
- `BBIP. Прогноз продвижения` — прогноз результатов при заданном бюджете
- `BBIP. Подключение услуги продвижения` — запустить BBIP
- `BBIP. Варианты бюджета продвижения` — какие бюджеты доступны

**ФАКТ.** BBIP = Budget-Based Item Promotion — продвижение с заданным бюджетом (альтернатива разовому VAS).

**ФАКТ. Из Item API — справочник slug'ов VAS:**
- `highlight` — «Выделить»
- `xl` — «XL-объявление»
- `stickerpack_x1/x2/x3` — значки на XL
- `x2_1`, `x2_7`, `x5_1`, `x5_7`, `x10_1`, `x10_7`, `x15_1`, `x15_7`, `x20_1`, `x20_7` — пакеты «до N раз больше просмотров на 1/7 дней»

**ВЫВОД.** Связка «продвижение»:
- Справочник типов (`Словарь`) + варианты бюджетов → в UI как dropdowns
- `Список услуг продвижения` → `Listing.activeVas[]` в core (какие VAS сейчас применены + срок)
- Траты на VAS уже учтены в `promoSpending` из «статистики показателей по профилю» — отдельно считать не надо

### 3.6.2 CPxPromo API — цены за целевые действия

**ФАКТ. Методы:**
- `Получение детализированной информации о действующих и доступных ценах за целевые действия и бюджетах` (scope `cpxpromo:read`, 20 RPM, по itemId)
  - Возвращает: действующую цену целевого действия (в копейках), доступные цены, бюджеты, **преимущества перед конкурентами**
- `Получение текущих цен за целевое действие и бюджетов по нескольким объявлениям`
- `Остановка продвижения`
- `Применение автоматической настройки`
- `Применение ручной настройки`

**ВЫВОД.** Это то, через что управляется **ставка на активных объявлениях** (пункт #3 из NEXT_ACTIONS — «Поставить ставку 7-10 ₽»). «Преимущества перед конкурентами» — готовая метрика, которую сейчас владелец оценивает руками по аудиту.

**ВЫВОД.** Для Sitka с 3 активными — делать руками через UI. API полезно когда будет 30+.

### 3.7 Auction (CPA-аукцион) API — приоритет ⭐

**ФАКТ. Методы:**
- Получение информации о действующих и доступных ставках
- Сохранение новых ставок

**ФАКТ.** Требует персонального менеджера Avito для подключения.

**ГИПОТЕЗА.** CPA-аукцион и CPxPromo — это два разных механизма или один и тот же под разными названиями? В документации «CPA-аукцион» идёт отдельным разделом, «CPxPromo» — «Настройка цены целевого действия». Нужно при интеграции уточнять у Avito-менеджера что именно включено для нашего аккаунта.

**ВЫВОД.** Закрывает пункт #3 из NEXT_ACTIONS («Поставить ставку 7-10 ₽ на активные»). Но на старте можно руками — API нужен когда объявлений десятки.

### 3.9 Три источника spend на Avito — как не посчитать дважды

**ФАКТ.** По итогам обхода нашёл **три** источника данных о расходах:

| Источник | Endpoint | Уровень данных | Глубина |
|----------|----------|----------------|---------|
| Item API | `Получение статистики расходов профиля` | Агрегировано, разбивка `promotion/presence/commission/rest` | 270 дней |
| Item API | `Получение статистических показателей по профилю` | Агрегировано + метрики воронки, разбивка по item | (в явном виде не указано, по аналогии ~270 дней) |
| CPA API | `balanceInfo` | Текущий баланс + аванс периода | — (моментальный снимок) |
| User API | `Получение истории операций пользователя` | **Сырые транзакции** кошелька (деньги + бонусы) | 1 год, окно 1 неделя/вызов |
| Autoload API | `Списания за объявления в конкретной выгрузке` | Списания в рамках выгрузки | ограничено выгрузками |

**ГИПОТЕЗА.** Иерархия источников (от общего к детальному):
1. **Показатели по профилю** — для дашборда Phase 1 (основной)
2. **Статистика расходов профиля** — для контроля по типам (promotion vs presence)
3. **История операций** — для audit-trail и reconciliation (если CRM-цифра не совпадает с кошельком)
4. **CPA balanceInfo** — для алерта «аванс заканчивается» (фикс для проблемы #1 из NEXT_ACTIONS)

**ВЫВОД.** Правило: одну и ту же цифру нельзя тянуть из двух источников. Для `SpendEntry` в core — один источник (показатели по профилю). Остальные — для отдельных use-case'ов (алерты, reconciliation, detailed drill-down).

**ВЫВОД.** Бонусы Avito (spendingBonus / списанные бонусы в истории) — это отдельная сущность, не деньги. Нужно либо завести в core `SpendType.Bonus`, либо игнорировать. По-хорошему — учитывать как «не-денежный spend» с нулевым влиянием на ROI, но с влиянием на funnel (бонус даёт просмотры).

## 4. Что API НЕ даёт

**ФАКТ.**
- **Нет прямого CRUD по объявлениям.** Create/update title/description/photos — только через Autoload XML.
- **Нет истории переписки глубже чем API начнёт её сохранять.** Старые чаты через API доступны, но сам Avito ограничивает на backend возвратом только разумного окна (точный лимит в доке не указан, надо тестировать).
- **Нет доступа к полному профилю клиента** (паспортные данные, возраст, регион регистрации). Только то, что клиент сам засветил в чате + phone из `phonesInfoFromChats` если чат целевой.
- **Нет webhook на новые отзывы.** Только polling через `/ratings/v1/reviews`.
- **Нет API для отключения/блокировки конкурентов в рекомендациях** — это UI-only feature платного тарифа («Скрытие конкурентов» из аудита).
- **Нет API для видео-обзоров товаров** — только через приложение/веб.

**ВЫВОД.** Операционная часть (редактирование 15 неопубликованных, переименование профиля, цены в названиях) останется ручной — либо через интерфейс, либо через Autoload-фид. API — это backend sync, не средство массового редактирования контента.

## 6. Ограничения и риски

**ФАКТ.** Messenger API требует подписку «Расширенный» или «Максимальный». Это блокер для Phase 2.

**ФАКТ.** У пользователя сейчас CPA-тариф без платных инструментов (см. `MY_AVITO_AUDIT.md` раздел 4). Значит `messenger:*` scope скорее всего не выдадут до апгрейда.

**ГИПОТЕЗА.** Autoload, Autostrategy, Ratings доступны на базовом тарифе — прямого ограничения в документации не видел. Но это требует ручной проверки попыткой получить токен + вызвать метод.

**ФАКТ.** Rate limits в публичной документации не указаны. Из `ARCHITECTURE_V2.md` уже заложен риск #2 — «нужно заложить exponential backoff».

**ГИПОТЕЗА.** Webhooks нестабильны (V3 только недавно добавлены, V2 ещё присутствует как «Получение подписок»). Нужен fallback на polling.

**ВЫВОД.**
- Перед стартом Phase 2 — обязательно апгрейд тарифа (пункт #8 в NEXT_ACTIONS).
- Для Phase 1 (чистая аналитика без мессенджера) — можно пробовать на текущем тарифе, но с готовностью, что часть методов вернёт 403.
- Заложить в services: retry с jitter, circuit breaker, webhook failure → fallback polling.
