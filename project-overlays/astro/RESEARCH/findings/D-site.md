# Блок D — Сайт и внешний контур

**Дата сбора:** 2026-04-17.

Для эксперта уровня Марины сайт — не «визитка», а **единственный single source of truth**, работающий одинаково для РФ и диаспоры (в отличие от любого мессенджера/соцсети — см. блок C).

---

## D.1 — Сайт эксперта-астролога (B2C)

### Роль сайта

**ВЫВОД (корневой критерий).** Сайт Марины решает **четыре функции**, которые не закрывает ни один мессенджер:

1. **Формальное юр.присутствие.** Реквизиты, политика ПДн, договор-оферта, согласие — всё это обязано быть на зарегистрированном сайте (см. блок B).
2. **Точка записи и оплаты.** Канал (Дзен/Telegram) выводит трафик → сайт принимает заявку и деньги → календарь фиксирует сессию.
3. **Single URL для всех каналов.** `marina-arhipova.ru/записаться` работает из Дзен-поста в РФ, из Telegram в Берлине, из Instagram в Токио.
4. **Актив, на который сохраняется равное право контроля.** Канал в Дзене/Telegram может быть заблокирован/деанонимизирован/утрачен; домен и хостинг — собственность эксперта.

### Что должен содержать — минимальный MVP

Обязательные блоки:

1. **Главная / Offer.** Экспертная позиция («Профессиональная западная астрология»), 5 лет практики, специализации.
2. **Услуги.** С ценами или диапазоном. Минимум: натальная консультация, соляр, синастрия, ректификация. Доп: кармическая, элективная, талисманы.
3. **Описание методики работы.** Как проходит консультация: предварительный опросник, сбор данных рождения, созвон 1-2 часа, аудиозапись + письменное резюме.
4. **Trust-блок.** Счётчик проведённых консультаций, ссылки на Дзен/YouTube/Telegram как публичный контент. Сертификаты обучения (если есть — у ВШКА, LabLife, у Марины — запросить).
5. **Отзывы.** Минимум 5-10 верифицированных, с именем/городом (с согласия).
6. **FAQ.** Подготовка к консультации, точность времени рождения, что если время рождения неизвестно (ректификация), возвраты, этика.
7. **Контакты + форма заявки / онлайн-запись.**
8. **Юр.блок в подвале.** Оферта-договор, политика обработки ПДн, реквизиты самозанятого/ИП (без лишних идентификаторов — только ФИО и ИНН как требует закон).
9. **Мобильная версия критична** — 70% трафика Дзена с мобильных ([статистика Дзена](https://tass.ru/ekonomika/26897149)).

Опционально:

- Блог / статьи (дублируется с Дзеном; если дублируется — добавляет SEO).
- Калькулятор натальной карты (встраиваемый виджет — см. F) — как lead-magnet для сбора email / Telegram-контактов.
- Free-тема «5 минут до начала ретрограда Меркурия» — для подписки на рассылку.

### Онлайн-запись

**ФАКТ.** Yclients — стандарт онлайн-записи в РФ для индустрии услуг (салоны красоты, фитнес, медицина, мастер-классы) ([yclients.com/online-booking](https://www.yclients.com/online-booking)).

**ФАКТ (контекст).** Основная ЦА Yclients — офлайн-услуги (салоны). Для чисто онлайн-консультации система избыточна.

**ВЫВОД.** Альтернативы для Марины:
- **Tilda + встроенная форма + календарь через Google Calendar / Calendly / RusCalendar** — минималистичный и достаточный стек.
- **Собственный TG-бот записи** — типовая модель у многих астрологов; но с блокировкой TG в РФ становится менее универсальной.
- **Yclients с онлайн-услугой** — рабочий вариант, но переплата за функционал офлайн-записи который не нужен.

Выбор конкретного стека — задача architect-фазы, но с ориентиром «минимальный и независимый от одного мессенджера».

### Предоплата для снижения no-show

**ФАКТ.** Типовая модель на маркетплейсах и у практиков: консультация с полной предоплатой ([mylablife.ru, astro-academia.ru](https://mylablife.ru/consultations/)). После оплаты — опросник, фиксация времени, созвон.

**ВЫВОД.** Для Марины:
- Предоплата 100% через СБП/карту — отрезает no-show почти полностью.
- Если клиент не готов к 100% — возможна предоплата 30-50% как резерв слота.
- Политика возврата при переносе / отмене — в оферте (см. блок B).

### Trust-блоки — конкретика для астрологии

**ФАКТ.** Типичные trust-блоки у топ-практиков:
- Мария Штерн — «диплом ВШКА + 1900 консультаций» ([astrotarget.ru](https://astrotarget.ru/)).
- Павел Андреев — «10+ лет опыта, школа LabLife» ([mylablife.ru](https://mylablife.ru/)).
- Лиза Васина — активный YouTube + Telegram как live-доказательства ([shkoly-astrologii.ru](https://shkoly-astrologii.ru/liza-vasina-astrolog-otzyvy/)).

**ВЫВОД для Марины.** Специфические её активы:
- 5 лет публичных материалов на Дзене — сильный верифицируемый актив.
- Узкие специализации (соляр, элективная, кармическая) — сильнее позиционирует чем «делаю всё».
- Возможно — ссылка на академическую принадлежность (если выпускница ВШКА или другой школы — уточнить).

### Мобильная версия

**ФАКТ.** 70% Дзена с мобильных; 2.9% средняя конверсия Insta-миграции в VK ([tass.ru 26897149](https://tass.ru/ekonomika/26897149), [habr](https://habr.com/ru/news/t/662003/)). Для всей русскоязычной ЦА — мобильный трафик выше десктопа.

**ВЫВОД.** Mobile-first — must. Адаптив на Tilda/Taplink встроен; на кастомном фронте — явная mobile-first разработка.

### Живые примеры сайтов

**ФАКТ.** Пример сайта астролога на Tilda: [astrologem.tilda.ws](http://astrologem.tilda.ws/) — астропсихолог Молдованова.

**ФАКТ.** Taplink как одностраничный агрегатор ссылок: [taplink.cc/mariia_stern](https://taplink.cc/mariia_stern) — Мария Штерн использует Taplink.

**ВЫВОД.** Наблюдаемая практика:
- Некоторые эксперты используют Taplink / Tilda как single-page landing — низкая планка входа, быстро.
- Более продвинутые (Мария Штерн, Лиза Васина, Павел Андреев) имеют полные сайты на своих доменах с отдельными разделами.

### Конструктор vs кастомная разработка

**ВЫВОД (корневой критерий).** Выбор зависит от **требуемого уровня интерактивности**:

- **Если сайт = лендинг + форма + блог** → Tilda достаточна (средняя подписка ~750 ₽/мес). Запуск за 1-2 недели при наличии текстов.
- **Если сайт включает калькулятор натальной карты / личный кабинет клиента / интеграцию с SaaS в будущем** → кастомный фронт (React/Vue/Svelte) с backend.

**ДЛЯ MARINA ФАЗЫ B2C.** Tilda / Taplink / WordPress достаточны. Не переинвестировать в кастом на старте. Это совпадает с принципом research → architecture: **MVP-каркас не должен диктовать продуктовые решения**.

**Стек astro репозитория (Haskell core + Python services + React web)** — гипотеза клиента (см. `README.md`). Для чистого landing-сайта Марины **этот стек избыточен**. Он имеет смысл только если в архитектуру сразу закладывается SaaS (блок D.2), что требует подтверждения клиентом.

## D.2 — Сайт SaaS для астрологов (B2B, перспектива)

### Landing page

**ФАКТ.** Стандартные секции B2B SaaS landing: features, pricing, demo, customer logos, docs link, signup/trial CTA ([обобщение по рынку]).

**Наблюдение (Chronos как референс).** [chronos.mg](https://chronos.mg/) — пример работающего landing российского астропроцессора: ясный список функций, демо без регистрации (частичное), тарифы (550/2000/10001 ₽), блог/новости, ссылки на мобильные приложения.

### Публичное demo

**ВЫВОД.** Для SaaS расчётов **публичное demo критично**:
- Астролог хочет потрогать интерфейс до подписки.
- Барьер «регистрация → оплата → попробовать» — слишком высокий в микронише с высокой лояльностью к Chronos.
- Без demo придётся компенсировать бесплатным триалом (что Chronos и делает — неделя бесплатно).

Минимальный demo-scope: ввод данных рождения + рендер натальной карты + базовая интерпретация аспектов/домов. Без сохранения данных, без PDF-экспорта.

### Документация и API

**ГИПОТЕЗА.** Документация нужна на двух уровнях:
- **Пользовательская** (как астрологу использовать сервис) — must have.
- **API-спецификация** (интеграция с сайтами астрологов / ботами) — надо решать по спросу; на MVP — не нужно.

**ВЫВОД.** На MVP — только пользовательская документация. API откладывается. Chronos API публично не позиционирует — значит спрос в сегменте низкий или закрытый.

### Один бренд vs два

**ВЫВОД (важный для стратегии G).** Два пути:

**Путь 1 — один бренд Марины.**
- `marina-arhipova.ru` / `marinaarhipova.ru` — консультации.
- `marina-arhipova.ru/software/` или поддомен — SaaS.
- Плюс: единый маркетинг, кросс-продажа (клиент консультации → видит SaaS).
- Минус: B2B-покупатели SaaS могут быть конкурентами Марины как эксперта — бренд эксперта мешает B2B-продажам.

**Путь 2 — два бренда.**
- Марина-эксперт — личный бренд.
- SaaS — отдельный бренд (например «AstroCalc» или что-то нейтральное).
- Плюс: SaaS не ассоциируется с конкретным экспертом; другие астрологи используют без репутационных вопросов.
- Минус: два маркетинга, два сайта, два сообщества.

**Живой контрпример.** Chronos — отдельный бренд, не «PavelAndreev.soft». LabLife — отдельный бренд школы. Павел Андреев — персональный бренд эксперта. **Три** бренда в одной экосистеме, не два. Это работает, но требует серьёзного маркетингового ресурса.

**ВЫВОД.** Для МVP B2B выбор между двумя моделями — вопрос не research, а архитектурный. Но с учётом ограниченного ресурса «небольшой команды» (из README), возможно компромисс:
- Первый год SaaS запускается **под брендом Марины как подсайт/поддомен**, с пометкой «софт, которым пользуется Марина».
- При росте и появлении других B2B-клиентов — ребрендинг и разделение.

### Разные ЦА — разные UX

**ВЫВОД.** B2C-сайт (консультации) и B2B-сайт (SaaS) требуют разных подходов:
- B2C: тёплый эмоциональный тон, личная фотография, истории клиентов.
- B2B: функциональный тон, скриншоты интерфейса, метрики продукта (точность расчётов, поддержка ОС).

Если совмещать на одном сайте — легко получить плохо работающий микс. См. выше путь 1 vs 2.

## D.3 — Конструктор vs кастомный: итоговый ориентир

| Фаза | Сайт | Стек ориентир | Срок запуска |
|------|------|---------------|--------------|
| B2C MVP сейчас | Лендинг Марины + онлайн-запись + оферта | Tilda / WordPress / Taplink | 1-2 недели (без калькулятора карт) |
| B2C расширенный | + личный кабинет клиента + калькулятор как lead-magnet | Tilda + встраиваемые виджеты или переход на кастомный | 1-2 месяца |
| B2B SaaS MVP (6-12 мес) | Landing + demo + signup + web-приложение | Кастомный фронт (React/Vue/Svelte) + backend | ~3-6 месяцев (без готовой архитектуры) |
| B2B SaaS + курсы | Выше + LMS-часть (уроки, прогресс) | Либо интеграция с готовой LMS (GetCourse, Antitreningi) либо кастомная | Зависит от выбора |

**ВЫВОД (для architecture-фазы).** Стек из `README.md` (Haskell core + Python services + React web) применим только к фазе B2B SaaS, **не к B2C лендингу**. Для B2C лендинга такой стек — overkill и замедление старта. См. G-strategy для корректной последовательности.
