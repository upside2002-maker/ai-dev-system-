# Блок F — Операционная модель

**Дата сбора:** 2026-04-17.

Как реально работает продажа и оказание услуги от первого касания до закрытия. Для астрологии есть специфические этапы, не встречающиеся в «обычном» консультировании.

---

## F.1 — Путь клиента для консультации (B2C)

### Типичная воронка

```
1. Контент в канале (Дзен-статья / Telegram-пост / Insta-reel / YouTube-видео)
      ↓
2. Интерес клиента ("хочу такое для себя")
      ↓
3. Переход по ссылке → на сайт
      ↓
4. Выбор услуги + слота + первичная заявка (имя, email, краткий запрос)
      ↓
5. Оплата (см. блок E)
      ↓
6. Получение опросника:
   - ФИО
   - Дата рождения (обязательно)
   - Время рождения (критично, точность до минут)
   - Место рождения (город/страна — для часового пояса)
   - Запрос клиента (что хочет узнать)
      ↓
7. Заполнение опросника клиентом (1-3 дня до консультации)
      ↓
8. Подготовка Марины: построение натальной карты, соляра, транзитов (30-60 мин)
      ↓
9. Консультация 1-2 часа (Zoom/Meet/Телемост) + запись
      ↓
10. После консультации: аудио/видео + письменное резюме клиенту
      ↓
11. Follow-up через 1-3 месяца (опционально):
    - Работают ли выявленные паттерны
    - Апсел доп.консультации / соляра / синастрии
```

### Специфические боли этапов

**ФАКТ (проблема сбора данных рождения).** По данным опросов практиков (упоминания на vc.ru, mylablife):
- Не все клиенты знают **точное время рождения**. Вариативность от «точно до минуты» (из справки роддома) до «примерно утром» (по рассказу родителей).
- При отсутствии точного времени натальная карта становится недостоверной (асцендент и дома сдвигаются на 1° за 4 минуты — за «примерно утром» асцендент может перескочить 2 знака).
- Решение: **услуга ректификации** (9 000-12 000 ₽ в рынке) — уточнение времени по событиям жизни клиента.

**ВЫВОД (операционный).** В опросник должно быть встроено:
- Чёткий вопрос «Знаете ли Вы точное время рождения?» с подсказками (справка роддома / слова родителей / не знаю).
- Автоматический роут: если «не знаю» → предложение ректификации как отдельной услуги до основной консультации.

**ФАКТ (квалификация запроса).** На маркетплейсах типа Lunaro — форма запроса клиента часто общая («хочу узнать своё будущее»). Эксперт тратит первые 15 мин консультации на уточнение.

**ВЫВОД.** На сайте Марины в опроснике должно быть:
- Категория запроса: отношения / карьера / здоровье / эмиграция / профориентация / духовный путь / общий.
- Свободное поле для конкретизации (3-5 предложений).
- Возможно — подсказки-примеры по категориям.

Это экономит 15-20 мин на консультации, которые идут на фактическую работу, а не прояснение.

**ФАКТ (no-show и переносы).** Онлайн-консультации — классическая проблема no-show. Решение индустрии: 100% предоплата (см. блок E).

**ВЫВОД.** Предоплата решает no-show почти на 100% по индустрии. Оставшиеся ~1-3% случаев (клиент заплатил, но не пришёл) — обрабатываются вручную по оферте.

### SLA критичные

**ВЫВОД (корневой критерий для MVP).** Критичных SLA три:
1. **Время ответа на первое сообщение** (заявка с сайта / вопрос в директе) — оптимально в течение 24 часов. Это не уникально для астрологии, но это первое касание, где конверсия ломается.
2. **Срок до проведения консультации** — от оплаты до сессии. В индустрии: 1-14 дней. Для Марины с её графиком (один эксперт, не поток) — 3-14 дней реалистично.
3. **Срок присылки пост-материалов** (аудио/резюме) — 1-3 дня после сессии. Критично для доверия — если резюме приходит через 3 недели, клиент забыл что обсуждали.

### Инструменты (текущая практика индустрии)

**ФАКТ.** Типичный стек практика-астролога:
- Онлайн-запись — Tilda-форма / Yclients / собственный TG-бот / ручной приём через DM.
- Опросник — Google Forms / Tilda-форма / Typeform / Notion-форма.
- Календарь — Google Calendar / RusCalendar / Яндекс.Календарь.
- Созвон — Zoom / Google Meet / Яндекс.Телемост / Skype (редко).
- Запись — локально на компьютере Марины + выгрузка в Google Drive / Яндекс.Диск.
- Резюме — обычно Google Doc / Word / письмо в email.
- Учёт клиентов — Excel / Google Sheets / Notion / Bitrix24.
- Астрологические расчёты — Chronos / ZET / Solar Fire / Astro.com.

**ВЫВОД.** У практиков **нет единого CRM-инструмента**. Это разнородный стек из 5-8 инструментов, каждый под одну функцию. Это создаёт **операционный долг** (контекст клиента разбросан по сервисам) — **и это потенциальная ниша для SaaS в будущем** (см. F.2), но для B2C Марины — пока не её задача решать это продуктом.

Для MVP Марины: **собрать устойчивый minimal-viable стек** из готовых инструментов:
- Сайт (Tilda) + Calendly/Яндекс.Календарь интеграция.
- Google Forms для опросника.
- Google Meet / Яндекс.Телемост для созвонов.
- Локальная запись + Google Drive для аудио/видео.
- Google Sheets / Notion для учёта клиентов и follow-up.

Не пытаться на этой фазе делать собственную CRM — не задача MVP.

## F.2 — Путь пользователя SaaS (B2B, перспектива)

### Typical onboarding

```
1. Landing — увидел Chronos/SaaS Марины где-то (курс, реклама, рекомендация)
      ↓
2. Free tier / Trial sign-up (email + пароль)
      ↓
3. Первый расчёт: ввод данных клиента (или самого астролога)
      ↓
4. Получение натальной карты на экране
      ↓
5. Попытка сохранить карту в профиль / вернуться позже
      ↓
6. Upgrade prompt → подписка
      ↓
7. Регулярное использование:
   - Новые клиенты → расчёты
   - Пометки астролога на карте
   - Соляр / транзиты на дату
   - Экспорт в PDF
   - Интеграция с заметками
```

### Отличие от «обычного» SaaS

**ВЫВОД.** Астролог как пользователь SaaS **не похож** на типового b2b-пользователя (например, пользователя Notion или Figma):

1. **Длительность сессии.** Астролог работает с картой 1-3 часа перед консультацией клиента. UI должен поддерживать глубокое разглядывание (панели, тулы, multiple charts открыты одновременно).
2. **Персонализация обозначений.** Каждый астролог имеет **свою школу** (классическая / эволюционная / ведическая / хорарная). UI должен позволять:
   - Настройку системы домов (Placidus / Koch / Equal / Campanus и т.д.).
   - Включение/выключение определённых точек (Лилит, Хирон, фиктивные планеты).
   - Настройку аспектов (орбисы, какие аспекты показывать).
3. **Заметки и приватность.** Астролог пишет заметки по картам — это его интеллектуальная собственность. **Утечка чужих заметок** — серьёзный риск для доверия к SaaS.
4. **Offline-доступ.** Иногда консультация проходит в офлайн-месте со слабым интернетом. Нужна хотя бы view-функция без сети (обычно это PWA или локальное приложение).

**Живой контрпример.** Chronos реализует кастомизацию домов, точек, аспектов. Это **стандарт** в нише, не отличительный признак.

### Хранение данных клиентов астролога

**ФАКТ (связь с блоком B).** 152-ФЗ: дата/место рождения — ПДн. Астролог как клиент SaaS становится оператором ПДн для своих конечных клиентов; SaaS — обработчик по поручению.

**ВЫВОД.** Требования к SaaS:
- Серверы в РФ (требование 152-ФЗ для ПДн граждан РФ).
- Шифрование при хранении.
- Возможность удаления данных по запросу конечного клиента (через астролога-оператора).
- Журналирование доступа.

Это не hygiene — это legal must для SaaS с юридической чистотой.

## F.3 — Астрологические расчёты (техническая специфика)

### Минимальный MVP расчётов для консультации

**ФАКТ (наблюдение из канала Марины и других).** Базовый кейс западной классической астрологии включает:

1. **Натальная карта** — планеты в знаках и домах, аспекты между планетами, фиктивные точки (узлы, Лилит, селена).
2. **Соляр** (solar return) — карта на день точного возврата Солнца в натальную позицию (ежегодная карта).
3. **Транзиты** — движение реальных планет по натальной карте на заданную дату.
4. **Синастрия** — наложение двух натальных карт для анализа отношений.
5. **Директные дуги / прогрессии** — методы прогноза.
6. **Дома** — различные системы (Плацидус — доминирует в западной классике).

**Опционально для некоторых субниш:**
- Хорарная астрология (ответ на конкретный вопрос по карте момента).
- Элективная астрология (выбор благоприятного времени для события).
- Астрокартография (перенос карты на географическую сетку).

### Библиотеки расчётов (open source)

**ФАКТ.** **Swiss Ephemeris** — эталон. Double license: AGPL / коммерческий CHF 750-1550. Лежит в основе большинства современных астрологических программ ([astro.com/swisseph](https://www.astro.com/swisseph/sweph_e.htm), [GitHub aloistr/swisseph](https://github.com/aloistr/swisseph), [roxyapi.com](https://roxyapi.com/blogs/swiss-ephemeris-explained-developers)).

**ФАКТ.** **pyswisseph** — Python-обёртка, распространяется под AGPL ([PyPI](https://pypi.org/project/pyswisseph/), [GitHub astrorigin/pyswisseph](https://github.com/astrorigin/pyswisseph)).

**ФАКТ.** **sweph** — Node.js биндинг ([GitHub timotejroiko/sweph](https://github.com/timotejroiko/sweph)).

**ФАКТ.** **Flatlib** — Python, основан на Swiss Ephemeris, с более высокоуровневым API (GitHub flatangle/flatlib).

**ФАКТ.** **Astrolog 5.40** — 100% бесплатный open source, доступен для множества платформ ([deluxeastrology.com](https://deluxeastrology.com/blog/top-astrology-software-programs-in-2024/)).

### Альтернативы Swiss Ephemeris

**ФАКТ.** **Skyfield** (Python, BSD лицензия) — астрономическая библиотека на основе NASA JPL ephemeris. Открытая лицензия позволяет использовать в closed-source SaaS без ограничений.

**ГИПОТЕЗА.** Skyfield даёт эквивалентную точность планетарных позиций (оба используют JPL DE). Но:
- У Skyfield нет готовых астрологических абстракций (дома, аспекты, управители).
- Они программируются поверх.
- Timezone-handling и историческая точность могут отличаться.

**ВЫВОД (для architect-фазы).** Выбор **эфемерид** — архитектурное решение, не research. Варианты:
1. **Swiss Ephemeris AGPL** — 0 ₽ лицензия, но SaaS становится open source.
2. **Swiss Ephemeris коммерческая** — ~CHF 1550 (150 k ₽) разовая; SaaS остаётся closed source. Экономически — приемлемо с первых клиентов.
3. **Skyfield BSD** — 0 ₽ лицензия, SaaS может быть closed source. Но придётся строить астрологические абстракции самим — больше разработки, риск ошибок в интерпретации домов.

Это decision для architect + обсуждения с клиентом, не для research.

### Точность

**ФАКТ.** Профессиональные астрологи требуют:
- Точность планетарных позиций до секунд дуги.
- Учёт прецессии, нутации, относительной позиции от Земли (геоцентрические координаты).
- Корректный handling timezone, DST (летнее время), исторических изменений временных зон.
- Поддержка расширенных точек (Лилит — истинная и средняя, селена и др.).

**Swiss Ephemeris** — соответствует всем этим требованиям. Skyfield — для базовых планет и дат (после 1900) тоже подходит, для специфических точек потребуется расширение.

### Визуализация

**ФАКТ.** Open-source SVG-библиотеки для рендера:
- **@astrodraw/astrochart** — TypeScript, современный, без зависимостей, активный репозиторий ([npm](https://www.npmjs.com/package/@astrodraw/astrochart), [GitHub AstroDraw/AstroChart](https://github.com/AstroDraw/AstroChart)).
- **Kibo/AstroChart** — оригинальная JS-версия ([GitHub Kibo/AstroChart](https://github.com/Kibo/AstroChart)).
- **astrochartjs** — для ведических карт ([GitHub erajasekar/astrochartjs](https://github.com/erajasekar/astrochartjs)).

**ВЫВОД.** Базовый рендер — не задача R&D, есть открытые решения. Но:
- Эстетика карт — важный фактор для pro-астрологов (они смотрят карты часами).
- Настройки цветов/стилей/символов планет — ожидают.
- Возможно потребуется доработка существующих библиотек или собственный рендер для достижения нужного уровня.

## Критическое замечание про стек из README

### Заявленный стек mini-MVP

**ФАКТ.** В `README.md` проекта указан стек `Haskell core + Python services + React web` как гипотеза клиента (не подтверждённое решение).

### Почему это важно здесь

**ВЫВОД (для architect).** Астрологические расчёты — Python-домена:
- `pyswisseph` (Python, наиболее зрелая обёртка).
- `Flatlib` (Python).
- `Skyfield` (Python).

Экосистема Python в астрорасчётах **зрелая**. Haskell эквивалентов Swiss Ephemeris **нет** в public open source (проверено grep в Hackage).

**Вариант 1 (переиспользование).** Python-сервис для расчётов + любой backend для бизнес-логики/API. Это максимизирует переиспользование зрелых библиотек.

**Вариант 2 (Haskell-реализация эфемерид).** Разработка эфемерид в Haskell с нуля — это **месяцы работы** на квалифицированном разработчике с опытом в астрономии. Это не exploratory задача — это R&D-проект объёмом ~1-2 человеко-года.

**ВЫВОД (для Phase 1 Architecture).** Research **не принимает** архитектурное решение, но фиксирует:
- Python для расчётов — естественно и экономично.
- Haskell для расчётов — возможно, но дорого и не даёт бизнес-ценности для клиента.
- Разделение «Python services (расчёты) + любой core-язык (бизнес)» — reasonable.
- Использование заявленного стека sitka (Haskell core + Python services + React web) — **частично подходит**, но слой «core» требует бизнес-обоснования, не копирования из другой ниши.

Этот вопрос переходит в architect-фазу как открытый вопрос для обсуждения с клиентом.

### MVP vs SaaS — разный стек

**ВЫВОД.** Для **B2C сайта Марины (фаза сейчас)** стек из sitka **избыточен**. Лендинг + форма + онлайн-запись — Tilda. Нет компилированной business-логики, нет многослойного backend.

Для **B2B SaaS (6-12 мес)** — стек становится релевантен, если есть необходимость в «core» слое с инвариантами (подписки, биллинг, ПДн-границы). Даже тогда — Python/TypeScript достаточны для MVP; Haskell добавляется при доказанной необходимости type-safety для финансовых операций, что не характерно для SaaS такого масштаба.

## Что ушло в OPEN_QUESTIONS

- Точная комбинация услуг Марины (соляр, ректификация, кармическая и т.д.) и их соотношение по выручке — нужны бизнес-данные.
- Выбор эфемерид (Swiss Ephemeris AGPL vs комм. vs Skyfield BSD) — архитектурное решение.
- Выбор СRM для B2C фазы (Notion / Bitrix24 / custom) — задача архитектора с учётом бюджета.
- Работа с оффлайн-режимом SaaS (PWA vs native app vs web-only) — архитектурное решение, зависит от поведения целевых астрологов.
- Глубина кастомизации SaaS под разные астрологические школы (классическая / ведическая / эволюционная) — продуктовое решение, зависит от выбранного сегмента B2B.
