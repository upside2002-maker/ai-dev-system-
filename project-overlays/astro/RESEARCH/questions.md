# Astro — Research Questions

Вопросы которые нужно закрыть на Phase 0 перед переходом к архитектуре.

**Правила работы:**
- Каждый ответ помечать `ФАКТ` (со ссылкой на источник) / `ГИПОТЕЗА` / `ВЫВОД`
- Живой контрпример сильнее абстрактной статистики
- Не выбирать "лучший" из нескольких рабочих — описывать trade-offs
- Корневой критерий вместо списка симптомов
- AI-разбор не заменяет живого специалиста (юриста/бухгалтера/отраслевого консультанта)

**Специфика этого проекта — два продукта с разными ЦА:**
- Продукт 1: индивидуальные консультации эксперта конечным клиентам (B2C). Это сейчас.
- Продукт 2: SaaS-сервис автоматизации расчётов для других астрологов (B2B). Это перспектива.

Research на каждый блок должен разбирать **отдельно для каждого продукта**, не смешивая. В некоторых блоках (например, юр/оплаты) это будут **разные** модели.

**Результат:** на каждый блок ниже создать файл в `findings/` — например `findings/A-market.md`, `findings/B-legal.md` и т.д.

## Блок A — Ниша и рынок

Цель: понять контуры рынка, игроков, тренды. Разобрать для **двух** продуктов.

### A.1 — B2C (консультации эксперта)

1. Какой размер рынка астрологических консультаций в РФ (оценка порядка)?
2. Кто прямые конкуренты-эксперты (топ-10, их каналы, цены, специализация)?
3. Средний чек на консультацию в рынке (от-до, медианное значение)?
4. Каналы продаж типичные для практикующих астрологов (Instagram был основным — что после блокировки)?
5. Какие субниши внутри астрологии: натальная / прогнозы / совместимость / бизнес-астрология / предсказательная?
6. Какой уровень скептицизма ЦА и как конкуренты с ним работают?
7. Есть ли сезонность (Новый год, ретрограды, знаковые транзиты)?

### A.2 — B2B (SaaS для астрологов)

1. Какие существуют сервисы автоматизации астрологических расчётов? Международные (AstroSeek, Astro.com, Solar Fire) и российские.
2. Какой уровень насыщенности этой ниши — есть ли живой коммерческий SaaS для русскоязычных астрологов?
3. Что астрологи сейчас используют для расчётов (программы, сайты, ручные расчёты)?
4. Какие есть боли у практикующих астрологов в расчётах и организации работы с клиентами?
5. Какая типичная монетизация таких сервисов (подписка / оплата за расчёт / freemium)?
6. Swiss Ephemeris и другие расчётные библиотеки — open source, готовые. Это снижает барьер входа для конкурентов.

## Блок B — Юридическая и финансовая модель

Цель: выбрать применимую конструкцию для приёма денег. Разобрать для **двух** продуктов отдельно.

### B.1 — Консультации эксперта (B2C)

1. Применимые правовые конструкции:
   - Возмездное оказание услуг (классика)
   - Самозанятость (НПД 4%/6%) — самое простое для эксперта-одиночки
   - ИП УСН 6% — если планируется найм или оборот >2.4M/год
2. Специфика астрологии в юр.контексте:
   - ОКВЭД — какой подходит (93.29 "прочая развлекательная деятельность" / 96.09 "прочие услуги" / другой)?
   - ФЗ "О защите прав потребителей" — применим ли к астрологическим услугам? Как формулируют оферту действующие астрологи-ИП?
   - Возвраты — как обрабатывают претензии "не сбылось"?
   - Статус лицензирования — нужна ли лицензия / аккредитация (предположительно нет, проверить)?
3. Первичка: договор-оферта на консультацию, чек через онлайн-кассу (для самозанятого — в приложении "Мой налог"), способ фиксации результата (запись / резюме / отчёт)?
4. Живые примеры: найти 3-5 действующих астрологов-ИП / самозанятых с открытой юр.информацией на сайте. Их ОКВЭДы и структуру.

### B.2 — SaaS для астрологов (B2B, перспектива)

1. Модель — подписка или оплата за использование. Какая конструкция договора-оферты для SaaS подходит?
2. Самозанятый не может принимать от юрлиц >2.4M/год → для SaaS скорее всего ИП УСН или ООО.
3. Лицензионный договор на ПО vs услуги — что правильнее для SaaS расчётов?
4. Данные пользователей (даты рождения, места — чувствительные) — применимость 152-ФЗ "О персональных данных". Нужна ли регистрация оператора ПДн?
5. Международные клиенты (если будут) — НДС для иностранных плательщиков, валютный контроль.
6. Интеллектуальная собственность — Swiss Ephemeris и другие внешние библиотеки, совместимость лицензий с коммерческим использованием.

**Обязательно:** результат этого блока — основа для живого юриста. AI-ассистент формулирует варианты.

## Блок C — Каналы дистрибуции и доверия

Цель: выбрать 2-3 канала для старта. Для астрологии характерны свои каналы — не копировать из других ниш.

### C.1 — Общие вопросы

1. Где физически находится ЦА астрологических услуг в РФ сейчас (после миграций 2024-2026)?
2. Какие каналы работают в РФ **сейчас** с учётом блокировок YouTube, замедления Telegram, блокировки Instagram? Проверить свежие источники.
3. Специфика астрологической ниши: ЦА преимущественно женская (25-55), мобильный трафик, визуальный контент. Как это влияет на выбор каналов?

### C.2 — Каналы специфичные для астрологии

1. Instagram (заблокирован в РФ) — был исторически основным для астрологов. Как конкуренты адаптировались?
2. Telegram канал эксперта — типичная модель. Монетизация (закрытый канал / консультации / курсы)?
3. TikTok / короткие вертикальные видео — астрологический контент популярен. Реальная применимость в РФ?
4. Yclients / Bookform / другие booking-сервисы для консультаций — какие используют астрологи?
5. Специализированные астрологические платформы (AstroChart, Astrolog и т.д.) — как канал привлечения для эксперта?
6. Маркетплейсы услуг (Profi.ru, Яндекс.Услуги) — применимость для астрологических консультаций?

### C.3 — Каналы доверия в астрологии

1. Отзывы — где их собирают (на личном сайте, в Telegram-канале, на Profi.ru)?
2. Экспертный контент (статьи, разборы знаковых событий, прогнозы по транзитам) — какой формат работает?
3. Как астрологи строят доверие в условиях высокого скептицизма ЦА (в отличие от, например, услуг косметолога)?
4. Какова роль личного бренда vs безличного сервиса в монетизации?

### C.4 — Каналы для SaaS (B2B, перспектива)

1. Где сидят практикующие астрологи которых можно привлечь в SaaS? (Telegram-каналы, закрытые сообщества, конференции.)
2. Каналы для B2B продвижения: партнёрства с астрологическими школами, конференции, реферальные программы среди коллег.
3. Специфика продаж в микронише: малый TAM, но высокая LTV при удержании.

**Важно:** приоритеты каналов для astro **не копировать из sitka** (байерский бизнес имеет совершенно другую ЦА и модель).

## Блок D — Сайт и внешний контур

Цель: определить минимальный сайт который даёт доверие + заявки. Разобрать отдельно для двух продуктов.

### D.1 — Сайт эксперта-астролога (B2C)

1. Какая роль сайта: лендинг с онлайн-записью / блог с экспертным контентом / личный бренд + портфолио?
2. Обязательные trust-блоки для астрологических услуг: сертификаты обучения, опыт в годах, кейсы, отзывы, методика работы?
3. Онлайн-запись на консультацию (Yclients / Tilda forms / самописная) — что выбирают коллеги?
4. Интеграция с онлайн-оплатой (как получить оплату до консультации чтобы снизить no-show)?
5. Мобильная версия критична (большая часть трафика в нише — мобильный).
6. Примеры сайтов топ-10 русскоязычных астрологов — разобрать что у них работает.

### D.2 — SaaS для астрологов (B2B, перспектива)

1. Landing page SaaS: какие секции обязательны для B2B-тула (features, pricing, demo, docs)?
2. Публичная demo-версия — критична для SaaS расчётов (дать попробовать без регистрации)?
3. Документация и API-спецификация — уровень детализации для привлечения технических астрологов?
4. Разделение личного сайта эксперта и сайта SaaS — один бренд с двумя продуктами или разные бренды?

### D.3 — Конструктор vs кастомный

1. Для фазы эксперта-одиночки достаточен конструктор (Tilda / Taplink / WordPress) — это факт или нужна кастомная разработка?
2. Для SaaS потребуется кастомный фронт (React — в стеке sitka, но это гипотеза клиента, не подтверждено).
3. Какой минимальный путь от "старт" до "работающего приёма консультаций онлайн" по времени и деньгам?

## Блок E — Оплаты

Цель: выбрать способ приёма денег согласованный с юр.моделью из блока B. Разобрать для двух продуктов.

### E.1 — Оплата консультации (B2C)

1. Типичные способы для этой ЦА: СБП-QR / банковская карта / перевод на карту / система booking-сервисов.
2. Если модель — самозанятый: приём через приложение "Мой налог" + банковский перевод / СБП. Что теряется / приобретается vs ИП с р/с и онлайн-кассой?
3. Предоплата vs постоплата — что снижает no-show для онлайн-консультаций (типичная для астрологии боль)?
4. Возврат — политика при отмене клиентом / переносе / неудовлетворённости. Как формулируют коллеги?
5. Международные клиенты ("немного мир") — как принимать от нерезидентов (PayPal / Wise / Revolut — применимость из РФ)?
6. Что видит клиент в момент оплаты — влияет ли платёжный UX на конверсию?

### E.2 — Оплата SaaS (B2B, перспектива)

1. Подписка (месяц / год) или pay-per-use (per расчёт)? Что подходит для нишевого инструмента астрологов?
2. Рекуррентные списания — требования ЦБ РФ, использование ЮKassa / Robokassa / CloudPayments.
3. Международные подписки (если будут) — Stripe недоступен из РФ, альтернативы.
4. Free tier / trial — нужен ли для SaaS астрологов (низкая готовность платить "на пробу" в нише).
5. Модель возврата для SaaS — типовая (pro-rated refund при отмене).

### E.3 — Money flow

Для каждого продукта пошаговая схема с документами:
```
Клиент → заявка → оплата → фиксация сделки → оказание услуги → документы → закрытие
```

**Перекрёстная проверка:** выводы этого блока согласовать с блоком B. Если противоречат — пересмотреть один из них.

## Блок F — Операционная модель

Цель: как реально работает продажа от первого касания до закрытия сделки. Для астрологии характерны специфические этапы.

### F.1 — Путь клиента для консультации

1. Типичный путь: контент в канале → интерес → заявка → уточнение запроса → сбор исходных данных (дата/время/место рождения) → оплата → консультация → follow-up.
2. Боли этапов:
   - Сбор точных данных рождения (не все знают время с точностью)
   - Квалификация запроса (какой тип консультации нужен)
   - Переносы и no-show
3. SLA критичные: время ответа на первое сообщение, срок до проведения консультации, формат и срок follow-up.
4. Какие инструменты используют астрологи: Telegram-бот для записи / Yclients / Google Forms / ручной учёт в Notion?

### F.2 — Путь пользователя SaaS (перспектива)

1. Typical onboarding: регистрация → ввод данных клиента → расчёт → экспорт / сохранение в профиль.
2. Что отличает "клиент SaaS астролога" от "клиент обычного SaaS" — нужна ли глубокая персонализация UI под профессиональные пометки астролога?
3. Хранение данных клиентов астролога (натальные карты, заметки) — требования к безопасности и приватности.

### F.3 — Астрологические расчёты (техническая специфика)

1. Какие расчёты входят в минимальный MVP для экспертной консультации (натальная карта / соляр / транзиты / асцендент / дома / аспекты)?
2. Какие open-source библиотеки для расчётов существуют (Swiss Ephemeris, Flatlib, pyswisseph)? Лицензии и ограничения.
3. Точность расчётов — что критично для астрологов-профессионалов?
4. Визуализация карт — какие готовые решения (SVG chart libs, JS-библиотеки)?

**Важно для architect-фазы:** заявленный стек `Haskell core + Python services + React web` (как в sitka) — **гипотеза клиента**. Для астрологических расчётов более естественен Python (pyswisseph готовый), а Haskell-реализация эфемерид будет разработкой с нуля на месяцы. Этот вопрос переходит в architect-этап как открытый.

## Дополнительный вопрос — только для этого проекта

### G — Разделение продуктов и стратегия развития

1. Продукт 1 (консультации эксперта) и продукт 2 (SaaS для астрологов) — это **разные бизнесы**. Можно ли развивать их параллельно одной небольшой командой или нужно приоритизировать?
2. Какая последовательность имеет экономический смысл:
   - Сначала монетизировать экспертные консультации, потом параллельно строить SaaS?
   - Сразу закладывать SaaS-каркас, используя эксперта как первого пользователя?
   - Другой сценарий?
3. Риски каждого сценария — финансовые, операционные, отвлечение фокуса.
4. Какой сигнал / метрика будет триггером перехода от "эксперт-бизнес" к "SaaS-бизнесу"?

## Выход из Phase 0

Когда все 7 блоков закрыты (A-F + G):

1. Каждый `findings/[блок].md` содержит ответы с пометками `ФАКТ`/`ГИПОТЕЗА`/`ВЫВОД` и источниками
2. Создать `findings/SUMMARY.md` — сводка выводов и решений по каждому блоку, с явным разделением "решение для B2C-продукта" vs "решение для B2B-SaaS"
3. Создать `findings/OPEN_QUESTIONS.md` — что осталось неясным и требует живого специалиста (юриста / бухгалтера / астролога-практика / разработчика)
4. Передать на review пользователю

**Не начинать Phase 1 (Architecture) пока:**
- пользователь не подтвердил ключевые выводы research (особенно разделение B2C и B2B)
- не получена консультация живого специалиста по юр.блокам (особенно ОКВЭД, статус астрологии как услуги, применимость ФЗЗПП)
- SUMMARY и OPEN_QUESTIONS не созданы

**Открытый вопрос переходящий в Phase 1:** применимость стека sitka (Haskell core + Python services + React web) к астрологической специфике. Research фиксирует аргументы "за" и "против", architect принимает решение с учётом существующего mini-MVP.
