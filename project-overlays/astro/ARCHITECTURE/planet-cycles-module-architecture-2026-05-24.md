## 0.1 Методология возвратов — locked parameters (источник: Дараган «Транзиты», прислано Мариной 2026-05-24)

Verbatim-источник используется ТОЛЬКО для вывода инженерных параметров. Гард «no Daragan verbatim» в силе для клиентского текста (трактовки — волна 4, парафраз).

| Параметр | Значение | Цитата-обоснование |
|---|---|---|
| Орб | **1°** | «Для уверенности сужайте орбисы до 1 градуса» |
| Фаза | **только сходящийся + точный** (applying→exact); расходящийся не считаем | «учитывать необходимо только сходящийся и точный аспект» |
| Быстрые (Солнце/Луна/Меркурий/Венера/Марс) | один точный контакт, окно **часы–дни** | «транзит таких планет по своему натальному месту — дело краткосрочное, от часов до нескольких дней» |
| Медленные (Юпитер..Плутон) | возврат = **серия точных проходов** (ретро-петля), окно месяцы; даты каждого прохода | «с высшими планетами… время контакта значительно больше… отслеживать… событийный ряд во время образования точного аспекта» |
| Частота внутренних | Меркурий/Венера/Солнце — **раз в год**; Луна — месяц; Марс — 2 года | «возвращения бывают регулярно: раз в год, для Луны — раз в месяц, а для Марса — раз в два года» |

**Закрывает Q1/Q2/Mercury-флаг:**
- **Q1** (какой проход): быстрые — один контакт; медленные — **серия всех точных проходов** (headline = ближайший проход, + полная серия дат).
- **Q2** (окно): орб 1°, applying→exact; быстрые часы-дни, медленные месяцы.
- **Mercury-флаг СНЯТ:** Дараган сам — «раз в год». Геоцентрика подтверждена, Phase A корректна; таблица «3-4×/год» = орбитальная механика, не предиктивный возврат.

**Отложено (НЕ в returns-MVP):**
- **Фильтр чистоты возврата** (возврат «чистый», если натальная И транзитная планета не образуют других аспектов в этот момент) — «для чистоты информации… никаких других аспектов». Требует одновременного aspect-движка → **волна 2+**.
- **Трактовочная рамка** (быстрые = психологический фон; медленные = событийный ряд на точном аспекте) → **волна 4** (парафраз, не verbatim).

**Приоритет презентации (Марина 2026-05-24):** сильнее всего — **Луна, Солнце, Венера, Марс, Юпитер, Сатурн** (6, реально в жизни). Меркурий вторично. Уран/Нептун/Плутон — с пометкой. (Уран-оппозиция в 42 в её «важных возвратах» = самоаспект → волна 2, не MVP.)

## 0.2 Daragan passage 2 — per-planet директивы (2026-05-24)

**Контрольные времена = орб 1° во времени (ВАЛИДАЦИЯ locked-решения):**

| Планета | Контрольное время (Дараган) | = орб |
|---|---|---|
| Солнце | ±сутки | ~±1° (Солнце ~1°/день) |
| Луна | ±2 часа | ~±1° (Луна ~0.5°/час) |
| Меркурий/Венера/Марс | часы–дни | ~±1° |
| Лилит | ~2 недели | ~±1° (Лилит ~0.11°/день → ±9 дн) |
| Медленные | месяцы | 1° в ретро-петле |

Один закон **орб 1°** сам порождает все контрольные времена Дарагана. Параметр подтверждён первоисточником.

**Scope-директивы:**
- **Только ИСТИННЫЕ возвраты** (точная натальная долгота). «Символические» 30°-возвраты — НЕ использовать («лучше не использовать вовсе»).
- **Юпитер и выше:** возвраты редки → НЕ отдельная self-control техника. MVP **показывает даты** (ask Марины «все 10»), но рамка = milestone-событие (волна 4), не техника.
- **Лунные Узлы:** возвраты НЕ использовать (Дараган) → Nodes out из всех волн возвратов.
- **⚠️ Лилит ВОЗВЫШЕНА:** «обращение Лилит — аспект, который необходимо проверять и прорабатывать», контроль ~2 недели. Дараган ставит Лилит ВЫШЕ внешних планет (must-check vs «не-использовать-для-Юпитер+»). **Противоречит решению «Лилит = волна 5».** PENDING decision (§ ниже): добавить в returns-MVP (нужно расширить `data Planet` + выбрать `lilith_mode` mean/true) ИЛИ сделать немедленным инкрементом после 10-классических. Не блокирует Phase B (движок per-planet generic; Лилит — аддитивно).

**Wave-4 контент (трактовки per planet, парафраз — НЕ verbatim):** Солнце = витальность/оптимизм/лёгкость самореализации vs упадок тонуса/невостребованность/здоровье (+ «месяц до ДР = Солнце в 12 солярном доме»); Луна = здоровье (хилег) + адекватность эмоций; Меркурий = поток информации (кол-во=сила, качество=скука/утомление плохо); Венера = мироощущение/привлекательность vs сомнения; Марс = активность/работоспособность vs вспыльчивость/конфликты, бытовые травмы=очень плохо. Источник в транскрипте 2026-05-24. Гард no-Daragan-verbatim в силе.

## 3. Целевая архитектура (returns-MVP)

### 3.1 Core (Haskell) — вся математика

**Рефактор (Phase B шаг 1):** вынести retro-aware «найти пересечение долготы + окно» из `TransitCalendar.hs` в `Domain.TransitMath` (поведение `analyzeAnnualCalendar` — бит-в-бит, golden cases не меняются: инвариант).

**Новый `Domain.Returns`:**

```haskell
module Domain.Returns
  ( PlanetReturn(..)
  , ReturnsResult(..)
  , findNearestReturns
  ) where

data PlanetReturn = PlanetReturn
  { prPlanet        :: !Planet
  , prNatalLong     :: !Longitude360
  , prExactJd       :: !Double          -- ближайший возврат строго после reference_jd
  , prOrbEnterJd    :: !(Maybe Double)  -- окно действия (~2 недели у медленных)
  , prOrbExitJd     :: !(Maybe Double)
  , prMotion        :: !TransitPhase    -- direct | retrograde | direct_return
  , prPeriodYears   :: !Double          -- период обращения (справочно)
  , prBeyondLifespan:: !Bool            -- True для Нептун/Плутон; пометка (Уран = once)
  } deriving (Eq, Show)

newtype ReturnsResult = ReturnsResult { rrReturns :: [PlanetReturn] }  -- ровно 10, порядок Sun..Pluto

findNearestReturns
  :: [PlanetPosition]                       -- натальные позиции (цели)
  -> Double                                 -- reference_jd
  -> Map Planet [(Double,Double,Double)]    -- per-planet sample'ы от reference вперёд
  -> ReturnsResult
```

Логика: для каждой натальной планеты — взять её sample-стрим, найти **первое** пересечение натальной долготы строго после `reference_jd` (retro-aware: если планета у возврата ретроградит, может быть петля — берём корректный «возврат», нумеруя проходы как в TransitCalendar), бисекцией уточнить до минуты, посчитать окно по орбу. Пометка `prBeyondLifespan` для Нептун/Плутон.

### 3.2 Services (Python) — сырые эфемериды + endpoint + PDF

- **Sampling** (`app/ephemeris/`): новый сэмплер. Для каждой планеты семплит вперёд от `reference_date` до первого пересечения, окно ограничено периодом (§ 4). **Никакой aspect-математики.**
- **Endpoint**: `GET /persons/{id}/returns?as_of=<iso>` (default as_of = now). Один вызов returns-workflow. Один snapshot (это отдельный workflow, не нарезка соляра — bright line #6 соблюдён).
- **PDF**: секция в `solar.html.j2` (reference_date = дата рендера) + хелпер в `transit_themes.py`. Таблица «планета → ближайший возврат → окно → пометка».

### 3.3 Frontend (React)

- `types.ts`: `PlanetReturn`, `ReturnsResult`.
- UI-панель «Ближайшие возвраты»: натал (из person) → таблица 10 дат, live от сегодня.
