## Сборщик и деградация

`platform/cpds_platform/collectors/liquidations.py`:
- `aggregate_events` — сворачивает stream-сообщения forceOrder в дневные строки
  `{day_ms, side, usd_cents, events, source}` по (день × сторона). Соглашение
  стороны: S=SELL → вынесло ЛОНГ, S=BUY → вынесло ШОРТ. Разбор отделён от сети,
  тестируется на фикстуре `binance_force_orders.json` (в т.ч. обёртка
  комбинированного stream `{"stream":..,"data":..}`). Кривые/чужие события
  пропускаются построчно (это не сбой источника), источник из-за одного кривого
  события не падает.
- Честная деградация ЕДИНЫМ механизмом (как farside/onchain, урок Т-1, Б2):
  `collect_stream` возвращает `StreamRead(degraded=причина)` и НЕ молчит, когда
  stream не подключён / фидер упал / актив не объявлен. stdlib websocket нет,
  сторонних сетевых пакетов проект не тянет (как http.py) → подключение stream
  подаётся извне как `feeder`; без фидера — деградация по дизайну.
- Числа сырья ЦЕЛЫЕ (центы, int) — без float (мандат Т-7b и §10); перевод в
  каноническую строку — только на границе снимка через numfmt.

Живая проба (CLI без фидера) — честная деградация, не молчание:
```
BTC ликвидации: источник недоступен: stream BTCUSDT@forceOrder не подключён
(REST истории ликвидаций нет: Binance закрыт, Bybit без публичного REST)
```

## Накопление + идемпотентность

`store.upsert_rows(path, rows, ["day_ms","side"])` — идемпотентное накопление по
СОСТАВНОМУ ключу (день×сторона):
- повтор тех же данных → файл не меняется (0 изменений);
- тот же день с подросшим объёмом (накопление вперёд из stream) → строка дня
  ОБНОВЛЯЕТСЯ, не задваивается;
- прошлые дни из старых прогонов сохраняются.

Раскладка: `data/liquidations/<SYMBOL>.parquet` (в репо не коммитится, как вся
история). Тест `test_upsert_idempotent_and_accumulates_tail` фиксирует все три
свойства. Демонстрация на фикстуре: первый upsert = 3 строки, повтор = 0.

## Сырьё в снимке (НЕ в сигнале)

`snapshot.liquidations_raw` + `build_snapshot(..., liquidations=…)`:
- `leverage.raw.liq_imbalance_usd` — дисбаланс (лонги−шорты) в USD, ЗНАКОВАЯ
  строка (+ перевес лонгов, − перевес шортов);
- `leverage.raw.liq_spike_pct` — перцентиль суммарного объёма дня в собственной
  истории ([0..1]) — «всплеск».
- Кладётся В семью «плечо рынка» рядом с фандингом/oi (если семья жива); если

## Оговорки качества

- **Абсолют занижен в разы** (Binance режет stream ~1/сек с 2021) — как абсолют не
  пригоден; только дисбаланс/всплеск. Зафиксировано в KNOWN_LIMITATIONS (Д-О6),
  в снимке (`liquidations:undercount`), в декларации vendors и в README.
- **Истории нет** → метрики молодые; `liq_spike_pct` на старте малоинформативен
  (короткий ряд), доверие низкое до накопления.
- **Одна биржа** (Binance) — не весь рынок; Bybit можно добавить рядом тем же
  контрактом (декларация + второй feeder), агрегировать по источникам.
- **Сеть отсутствует в тестах** — разбор на фикстурах; живой сбор требует внешнего
  feeder (websocket-обвязка/Claude), которого в CLI по умолчанию нет → штатная
  деградация.

## Риски для ревью

- **Семантика стороны.** S=SELL→лонг, S=BUY→шорт — стандартное соглашение Binance
  forceOrder. Проверить, что ревьюер согласен; перепутать стороны = перевернуть
  знак дисбаланса.
- **Источника stream нет в CLI по умолчанию** (нет stdlib websocket / без сторонних
  пакетов) → живого «накопил день» из сети сейчас НЕ показать, только синтетика на
  фикстуре. Это осознанно: накопление пойдёт, когда обвязка/Claude подаст feeder.
  Решить, достаточно ли этого для приёмки «начали накопление», или нужен реальный
  websocket-адаптер отдельной мелкой задачей.
- **Метка `liquidations:undercount` живёт в `data_quality.missing`** (как и
  `source:*`). Семантически это не «missing», а «неполнота» — выбрано ради
  неизменности схемы (новое поле = смена схемы запрещена). Ревью: ок ли смысловой
  оверлоад `missing`, или завести отдельное соглашение/префикс.
- **upsert vs append.** Для накопления вперёд нужен upsert (день докапливается),
  а не append-skip как у цены/фандинга (там ключ неизменен). Это сознательное
  отличие; проверить, что не ломает ожиданий «идемпотентной докачки хвоста».
- **Зона leverage остаётся зоной фандинга** — ликвидации не влияют на зону/сигнал
  (по мандату). Убедиться, что ревью это устраивает как «сырьё рядом».
