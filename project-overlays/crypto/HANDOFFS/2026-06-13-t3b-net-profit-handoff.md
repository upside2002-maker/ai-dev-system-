## Зафиксированные правила (главное для ревью)

### Правило округления произведения (ЕДИНОЕ)

Произведение двух фикс.-точных величин (масштаб 10^8 × 10^8 = 10^16)
приводится к 10^8 делением на 10^8 с округлением **полу-вверх по модулю**
(round half away from zero): дробная часть ≥ 1/2 округляется ОТ нуля.
Реализация целочисленная `divRoundHalfUp n d = (n + d`div`2) `div` d` для
n≥0, симметрично для n<0. Все величины здесь неотрицательны (qty/price/fees ≥ 0
по схемам), так что «от нуля» = «вверх»; знаковая ветка задана на будущее.
Применяется ровно в двух вычислениях — `proceeds` (через `mulFixed8`) и
`cost_basis` — оба через `divRoundHalfUp`, одно правило.

**`cost_basis` считается ОДНОЙ дробью с ЕДИНСТВЕННЫМ округлением**, без
промежуточного округления средней цены (чтобы не копить ошибку):
`cost_basis = divRoundHalfUp ((Σ qty_i·price_i)·qtySell) ((Σ qty_i)·10^8)`.
Масштабы: числитель 10^16·10^8 = 10^24, делитель 10^8·10^8 = 10^16, частное
10^8.

### Правило остатка распределения

Целочисленный остаток `r = net − Σ floor(net·доля/10^8)` (0 ≤ r < число корзин)
отдаётся ЦЕЛИКОМ ОДНОЙ корзине — с **наибольшей долей**; при равенстве долей —
**первой по фиксированному порядку K1 < K1_5 < K2 < K3**. Реализовано
`argMaxFirstByShare` (foldl1, строгое `>` сохраняет первый при ничьей).
Поскольку каждая база ≥ 0 и r ≥ 0, ни одна сумма не отрицательна, а
Σ = Σбаз + r = net точно. Перед распределением проверяется: доли
неотрицательны и в сумме РОВНО 1.0 (=10^8) — иначе отказ `BadDistributionSum`
(без этой проверки тождество Σ=net не держится); net ≥ 0 — иначе `NegativeNet`
(распределять убыток нельзя, это сигнал «фиксировать нечего» для ворот Т-3d).

## Риски для ревью

- **Округление произведения.** Правило полу-вверх по модулю задано и применено
  в одном месте; знаковая ветка `divRoundHalfUp` (n<0) сейчас не достигается
  (все входы ≥ 0), но симметрична — проверена тестом `−1.5→−2`. Стоит свериться,
  устраивает ли «полу-вверх» как канон (альтернатива — banker's rounding;
  выбран полу-вверх как простой детерминированный и совпадающий с интуицией
  округления денег вверх). `cost_basis` округляется ОДИН раз (одна дробь) — нет
  двойного округления avg→cost_basis.
- **Остаток распределения.** Отдаётся ОДНОЙ корзине (правило ТЗ «крупнейшей
  доле»), не методом наибольших остатков. Детерминизм ничьей — первый по
  K1<K1_5<K2<K3 (порядок `distributionEntries`). Проверить кейс «крупнейшая
  доля НЕ K1» (тест `ничья БЕЗ K1` → остаток в K2). Остаток < числа корзин по
  построению, единичный получатель его вмещает.
- **Переполнение `Integer`.** Тип `Integer` произвольной точности —
  переполнения нет по построению, даже на промежуточном 10^24 в `cost_basis`.
  Это сознательный выбор (как в Signal.hs).
- **Граница «продать больше позиции».** Строгое `>`: ровно вся позиция
  разрешена, `+1e-8` — отказ. Пустая позиция (Σqty=0) при qty>0 → отказ
  `SellExceedsPosition req 0` (не деление на ноль: ветка `totalQty>0` достижима
  только когда продажа разрешена). Проверить, что отсутствие лотов трактуется
  как «продать нечего», а не как успех с нулём.
- **net=0 и net<0.** net=0 → распределение нулей, Σ=0 (фиксировать нечего, но
  не ошибка). net<0 (убыток из `computeNetProfit`) → `allocateProfit`
  отказывает `NegativeNet`: распределять убыток нельзя. Решение «фиксировать ли
  при net≤min» — ворота Т-3d, не здесь.
