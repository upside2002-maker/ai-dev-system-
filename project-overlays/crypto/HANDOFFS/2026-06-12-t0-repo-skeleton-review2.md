### Корень дефекта (доказан байт-в-байт)

`pString` в `core/src/Json.hs` (ветка `'u' -> … chr (hexVal hex)`) разбирает
КАЖДУЮ `\uXXXX` отдельно и НЕ склеивает суррогатную пару в одну кодовую точку.
Поэтому ключ `"𐀀"`:
- **Python** (`json.loads`) → одна кодовая точка U+10000 → сортируется по 0x10000;
- **Haskell** → две сырые кодовые точки [U+D800, U+DC00] → сортируется по 0xD800.

Канонизация сортирует ключи (`sorted(...)` / `sortBy (comparing fst)`), поэтому
при соседнем ключе BMP из диапазона **[U+E000, U+FFFF]** порядок ключей
РАЗЪЕЗЖАЕТСЯ → разные канонические байты → разный SHA-256.

Доказательство (вход `{"豈":1,"𐀀":2}`):
- Haskell-хэш `44f636391a41aabf5749bcd458c066afa7ba177483a9f4bb8c9585a8b55c901a`
  == SHA-256 от `{"𐀀":2,"豈":1}` (non-BMP ключ первым, сорт по 0xD800);
- Python-хэш `b86f1155689ffaa02303407488a98fc2dc278b8691d434cf0b6693c49d00e1eb`
  == SHA-256 от `{"豈":1,"𐀀":2}` (F900 первым, сорт по 0x10000).

Граница подтверждена свипом: BMP-ключ U+0041/U+00E9/U+4E2D/U+D7FF против non-BMP
ключа → MATCH; U+E000/U+F900/U+FFFD/U+FFFF → MISMATCH. Ровно порог 0xD800 vs
0x10000.

### Насколько это бьёт «по-настоящему» (честная рамка серьёзности)

- Срабатывает ТОЛЬКО когда non-BMP символ в ключе приходит ЭКРАНИРОВАННОЙ парой
  `\uXXXX\uXXXX`. Если тот же символ записан ЛИТЕРАЛЬНЫМИ UTF-8 байтами, оба
  языка дают одну кодовую точку и хэши СОВПАДАЮТ (проверено: `{"豈":1,"𐀀":2}`
  литералом → MATCH `b86f1155…`). То есть боевые снимки-на-диске с обычными
  UTF-8 байтами сегодня безопасны.
- Снимки пишет Python-обвязка; если она сериализует `ensure_ascii=False` —
  литералы, дыра молчит. Но контракт §10 обещает «байт в байт из ОБОИХ языков»
  на ЛЮБОМ валидном входе, а `\u`-эскейпы — валидный JSON; Т-1/Т-3 потянут данные
  из обоих языков и от внешних поставщиков, которые вправе писать `\u`. Это та же
  природа, что и закрытая КРИТИЧЕСКАЯ дублей: латентный «раскол мозга» на одном
  логическом входе. Поэтому — блокер для фундамента денежного учёта, а не косметика.
- Лечение (НЕ вносил): в `pString` после сбора `\uXXXX` склеивать high+low
  суррогат (0xD800–0xDBFF за которым 0xDC00–0xDFFF) в `chr ((hi-0xD800)*0x400 +
  (lo-0xDC00) + 0x10000)`, как делает Python. Тогда ключ — одна кодовая точка,
  сортировка совпадёт. Заодно стоит добавить в `golden/evil/` ДВУХКЛЮЧЕВОЙ
  образец non-BMP+BMP[E000..FFFF], чтобы корпус ловил регресс.
