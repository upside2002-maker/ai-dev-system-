2. **Инвариант 4 (§9.5, главный).** `canReenter asset price locks` запрещает
   перезаход, если есть АКТИВНАЯ блокировка этого актива с `slSalePrice < price`
   (покупаем дороже, чем продавали); по цене ≤ цены продажи — разрешено.
   Property-тест перебирает набор цен продажи (1e-8, 0.5, 1.0, 100, 50000,
   12345.00000067) × пробы цены перезахода (−1e-8, ровно, +1e-8, −1.0, +1.0) и
   сверяет с ожиданием. Граница `≤` закрыта табличными кейсами (ровно цена
   продажи → разрешено; +1e-8 → запрет; −1e-8 → разрешено).

3. **Сброс снимает запрет.** `evaluateResets` переводит сработавшую активную
   блокировку в `Reset`; после этого `canReenter` ею не ограничивается. Тест:
   блокировка по 50000 при Active блокирует перезаход по 60000; снимок через 7
   месяцев → timeout(6) → `Reset` → перезаход по 60000 разрешён. Плюс кейс
   «Reset не ограничивает даже по 1000000» и идемпотентность (Reset остаётся
   Reset).

4. **Дизъюнкция сброса.** `resetTriggered` = `drawdown || timeout ||
   (critical_ok && (capitulation || valuation_low_zone))`. Табличные кейсы: каждое
   из 4 условий ПО ОТДЕЛЬНОСТИ сбрасывает (остальные выключены `noResetRules`);
   кейс «ни одно не выполнено при ВСЕХ включённых правилах» → блокировка остаётся
   `Active`; кейс «любое одно из всех включённых даёт Reset» (4 проверки).
   Выключенные условия (`Nothing`/`False` в `ResetRules`) в дизъюнкцию не входят.

5. **Критичность.** При `critical_ok = false`: `capitulation` и
   `valuation_low_zone` НЕ срабатывают (явный гейт `criticalOk &&` в
   `resetTriggered`); `drawdown` и `timeout` срабатывают (цена и календарь
   он-чейн не требуют). Тесты на каждое из 4 условий при `critical_ok=false`,
   обратный кейс при `critical_ok=true` (cap/val снова работают), и кейс
   дизъюнкции: при `critical_ok=false` cap/val заглушены, но drawdown в той же
   блокировке всё равно сбрасывает.

6. **Несколько блокировок на актив / чужой актив.** Запрет, если ХОТЬ ОДНА
   активная блокировка актива имеет `sale_price < price`; разрешено, если все
   активные актива ≥ цены. Сброшенные не ограничивают. Блокировки ДРУГОГО актива
   игнорируются (фильтр `slAsset == asset`). Property-тест сверяет `canReenter`
   с ссылочным определением «запрет ⟺ ∃ active BTC c sale_price < price» на
   декартовом произведении цен × 7 наборов блокировок (смесь активов/статусов).

7. **timeout календарный (год/месяц), без float и без «30 дней».**
   `monthIndex ym = year*12 + (month−1)`; сброс при
   `monthIndex(now) − monthIndex(sale) ≥ timeout_months`. ISO-дата разбирается
   `parseYearMonth` посимвольно (префикс `YYYY-MM-DD`, время после дня
   игнорируется; месяц обязан быть 01..12). Границы: ровно `timeout_months` →
   Reset (нестрогое ≥); на месяц меньше → Active; на месяц больше → Reset;
   переход через год (2025-11 → 2026-02 = 3 мес); день месяца не влияет
   (2026-01-31 → 2026-02-01 = 1 месяц, хотя прошёл 1 день — подтверждает
   «месяцы, не дни»); тот же месяц → 0. Property: разница индексов = 0..23 по
   24 последовательным месяцам.

8. **Нет Double/Float; детерминизм.** `grep -E "Double|Float"` по `Reentry.hs` и
   `ReentrySpec.hs` — только в комментариях «никаких Double/Float», ни одного
   типа. Все числа проходят через `parseFixed8` (как в проде). Детерминизм —
   отдельный раздел тестов: повторный вызов `canReenter`/`evaluateResets`/
   `createSaleLock` на тех же входах даёт тот же результат.

## v1-упрощение drawdown (что зафиксировал, где)

Условие `drawdown_pct` в §9.3 описано как «просадка ≥ X% от ПИКА ПОСЛЕ ПРОДАЖИ».
Пик после продажи требует ИСТОРИИ цен; чистая функция от одного снимка её не
имеет, а схема `market_snapshot.schema.json` не несёт `peak_since_sale` и менять
её нельзя (Do not touch). Поэтому в v1 просадка считается **от ЦЕНЫ ПРОДАЖИ**:

```
reset(drawdown) ⟺ текущая_цена ≤ цена_продажи × (1 − drawdown_pct)
```

Сравнение целочисленное в масштабе 10^16 (`price·scale ≤ sale_price·(scale−dd)`),
без деления и без float. Доля `dd` вне [0..1] → условие не считаем (не
срабатывает).

Упрощение **консервативное**: пик после продажи ≥ цены продажи, значит порог «от
пика» = пик·(1−dd) ≥ цена_продажи·(1−dd) = порог «от продажи». Цена пробивает
более высокий порог «от пика» раньше или одновременно — то есть сброс «от
продажи» наступает НЕ РАНЬШЕ, чем наступил бы «от пика». Блокировка держится
дольше, а не короче → в сторону осторожности (запрет ФОМО не ослабляется).

## Diversion-проверка (тесты реально ловят нарушения)

Прогнал мысленные мутации логики; для каждой указан тест, который её ловит
(в `ReentrySpec.hs` есть раздел «Diversion-проверка» с явными анти-кейсами):

| Мутация в Reentry.hs | Какой тест падает |
|---|---|
| `slSalePrice <= price` вместо `<` в `canReenter` (запрет на равенстве) | «инв.4: ровно цена продажи → разрешено» и анти-кейс «< вместо ≤» |
| `>` вместо `<` (запрет на дешёвом перезаходе) | «инв.4: на 1e-8 дешевле → разрешено», property инв.4 |
| дизъюнкция → конъюнкция (`&&` между условиями) | каждый «ТОЛЬКО X → Reset» (одиночное условие перестало бы сбрасывать) |
| гейт `criticalOk` поставлен на drawdown/timeout | «critical_ok=false → drawdown/timeout всё равно сбрасывает», анти-кейс гейта |
| гейт `criticalOk` снят с cap/valuation | «critical_ok=false → capitulation/valuation НЕ сбрасывает» |
| убрать фильтр `rcAsset /= slAsset` в `resetTriggered` | анти-кейс «чужой актив не сбрасывает» |
| убрать фильтр `slAsset == asset` в `canReenter` | «чужой актив: ETH не влияет на BTC», property чужого актива |
| `>` вместо `>=` в timeout (строгое сравнение) | «timeout: ровно timeout_months → Reset» |
| timeout «30 дней»/дни вместо календаря | «день месяца не влияет» (1 день = 1 месяц), анти-кейс календаря |
| `evaluateResets` сбрасывает Reset-блокировки заново / трогает не Active | «сброс идемпотентен», «НИ ОДНО → остаётся active» |
| `createSaleLock` ставит статус ≠ Active | «createSaleLock: статус Active» |
| drawdown `<` вместо `≤` на границе | «drawdown граница — цена РОВНО порог → Reset» |

Все мутации детектируются хотя бы одним падающим тестом — зелёный прогон не
маскирует подмену логики.

## Отклонения

1. **drawdown от цены продажи, а не от пика после продажи** (см. раздел
   «v1-упрощение» выше). Санкционировано ТЗ как явное v1-упрощение; зафиксировано
   в KNOWN_LIMITATIONS (Я-1), HANDOFF и haddock модуля. Поведение
   консервативное.

2. **`ResetContext` как явный вход вместо «снимок целиком».** Модуль не разбирает
   `market_snapshot.json` сам (это обвязка/Т-3d) — `evaluateResets` принимает
   пары `(ResetContext, SaleLock)`, где контекст несёт уже извлечённые из снимка
   величины на КОНКРЕТНЫЙ актив (цена, год-месяц снимка, флаг нижней зоны
   valuation, вход `classifySignal`). Это держит функцию чистой и узкой и даёт
   обвязке подставить нужный актив-срез на каждую блокировку. Флаг
   `rcValuationLowZone` принимается готовым (зона ∈ {cheap, capitulation}
   вычисляется вызывающим), чтобы не дублировать разбор зон снимка — `Signal`
   уже владеет зонами, второй разбор плодить нельзя.

3. **`YearMonth` — новый локальный тип для тайм-аута.** В принятом ядре нет
   календарного типа (даты живут строками в обвязке/каноне). Ввёл минимальный
   `YearMonth` + `parseYearMonth` ровно под месячную гранулярность тайм-аута;
   арифметика — целая (`monthIndex`), без float и без «30 дней». Не дублирует
   ничего существующего.

## Риски для ревью (на что смотреть особенно)

1. **Инвариант 4 на МНОЖЕСТВЕ блокировок.** `canReenter` запрещает, если
   `any blocks locks` (хоть одна активная актива дешевле цены покупки). Стоит
   перепроверить логику «достаточно одной»: ошибка тут (например `all` вместо
   `any`, или потеря фильтра по активу/статусу) — прямая дыра в главном
   инварианте. Property-тест со ссылочным определением это покрывает, но это
   денежное сердце — взгляд глазами не лишний.

2. **Критичность для capitulation/valuation-сброса.** Гейт `critical_ok` стоит
   ИМЕННО на cap/valuation (нужен он-чейн) и НЕ стоит на drawdown/timeout. Для
   capitulation гейт фактически двойной: `classifySignal` при `critical_ok=false`
   и так вернёт `Unconfirmed` (≠ `Capitulation`). Явный `criticalOk &&` оставлен
   ради читаемости и ради valuation, у которого своего гейта в `Signal` нет.
   Стоит подтвердить, что двойной гейт на capitulation — не баг, а осознанная
   подстраховка, и что valuation без явного гейта был бы дырой.

3. **Календарный timeout на границе.** Нестрогое `≥ timeout_months`, индекс
   `year*12 + (month−1)`, день/время игнорируются. Граница «ровно
   timeout_months» = Reset; «на месяц меньше» = Active. Стоит проверить
   формулу индекса (особенно month−1 и переход через год) и нестрогость на
   границе — это влияет на момент снятия запрета.

4. **v1-упрощение drawdown** (риск осознанный, см. выше) — ревью на согласие с
   формулировкой консервативности и с записью в KNOWN_LIMITATIONS.
