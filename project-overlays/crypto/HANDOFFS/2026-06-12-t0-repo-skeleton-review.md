## Находки по серьёзности

### КРИТИЧЕСКАЯ — кросс-хэш расходится на дубле ключей (ломает §10 / §9.5 №5)

Один логический JSON-объект с повторяющимся ключом даёт РАЗНЫЙ хэш из двух
языков. Доказано холодно:

- Вход `{"a":1,"a":2}`:
  - Python `json.loads` оставляет ПОСЛЕДНИЙ дубль → канон `{"a":2}` →
    `7e8059f495589fcd981232cc11d00b00da3802c01d688fa1cf1f6bed6e5bb33c`
  - Haskell `parseJson` (`pObject` копит обе пары, `reverse`) хранит ОБА →
    канон `{"a":1,"a":2}` →
    `1c53ee0df7b12fd4d65b976120c7fa6b847dc41dffd7f0331c3237a1ceab1756`
  - Доказательство схлопывания: SHA-256 от строки `{"a":1,"a":2}` (две пары) ==
    в точности Haskell-хэш `1c53ee…`; от `{"a":2}` == Python-хэш.
- То же на тройном дубле `{"a":1,"a":2,"a":3}` — расходятся.
- **Бьёт и боевой путь Gate, не только подкоманду hash.** DecisionInput с дублем
  ключа: Haskell `input_hash=22a78288…`, а Python того же входа насчитал бы
  `d00588…`. Воспроизводимость «байт в байт» (§9.5 №5, §10) сломана.
- Дыра не закрыта НИГДЕ: ни `json.loads`, ни jsonschema дубль не ловят (Python
  схлопывает дубли ещё ДО схемы — схема видит уже один ключ); в репо нет ни
  `object_pairs_hook`, ни проверки на дубли. `grep` по `duplicate/дубл/
  object_pairs_hook` пуст.

Почему это критично сейчас, а не «потом»: канонизация — единственный кит Т-0,
ради которого ТЗ потребовало независимого ревьюера. Снимки пишет Python-обвязка,
а `input_hash`/`config_hash` считает Haskell-ядро; любой кривой или подсунутый
снимок с дублем ключа разводит хэши молча. На этом фундаменте строится денежный
журнал (§10: «решение воспроизводимо байт в байт»). Контракт обязан либо
ОТВЕРГАТЬ дубль ключей с обеих сторон, либо одинаково его трактовать — сейчас ни
то, ни другое.

Минимальный патч (НЕ внесён, описываю): на ОБЕИХ сторонах отвергать дубль
ключей в объекте при парсинге канона.
- Python: `loads_strict` → `json.loads(text, object_pairs_hook=...)`, который
  бросает `CanonicalError` при повторе ключа.
- Haskell: в `pObject`/`Json` после сбора пар проверять уникальность ключей,
  иначе `Left`. (Заодно `lookupKey` перестанет зависеть от «первый против
  последнего».)

### СРЕДНЯЯ — расхождение «принял/отверг» на нескольких краевых входах

Не разъезд хэша, а разное ПОВЕДЕНИЕ (одна сторона приняла, другая отвергла).
Сегодня не вредит (в боевом пути такой вход → `Left` → observe_only), но это
трещина в заявлении «канонизация байт-в-байт» и мина для Т-1/Т-3, где данные
пойдут из обоих языков:

- **Целое с ведущими нулями** `{"n":007}`: Python `json.loads` ОТВЕРГАЕТ (JSON не
  разрешает ведущие нули), Haskell принимает и `normalizeInt` делает `7`.
- **Управляющий символ-литерал в строке** `{"k":"x\<TAB>y"}` (сырой 0x09 внутри
  строки): Python `json.loads` ОТВЕРГАЕТ («Invalid control character»), Haskell
  спокойно принимает и хэширует. Разные классы «что вообще является валидным
  входом».
- Первопричина: Haskell-`pNumber` берёт `span` по `"+-0123456789.eE"` и
  пропускает строковые литералы с любыми сырыми байтами; Python опирается на
  строгий `json`. Парсеры расходятся в том, ЧТО считать валидным JSON.

(`+5`, `NaN`, `Infinity`, `1e10`, `--5`, `1.2.3`, BOM, битый UTF-8, float —
проверено: ОБА отвергают. Эмодзи/вне-BMP, суррогатные пары через `\u`, не-ASCII
ключи, NFC vs NFD, пустые ключи/строки, регистрозависимые ключи, огромные целые,
деньги-строки с ведущими нулями/`-0.00` — совпадают. См. ниже.)

### СРЕДНЯЯ — схема `decision_output` не требует `estimated_net_profit` у денежных атомов

ТЗ (чек-лист ревью) прямо спрашивает: отклоняет ли схема `TakeProfit` без
`estimated_net_profit`? **Нет.** В `$defs/action` `required: ["atom"]`, а
`estimated_net_profit` опционален для ВСЕХ атомов. Проверено: атом
`{"atom":"TakeProfit"}` без прогноза прибыли схему ПРОХОДИТ; `DistributeProfit`
без `distribution_plan` тоже. По §9.1/§9.3 денежные атомы несут
`estimated_net_profit`; схема этого не гарантирует. Для Т-0 (заглушка денег не
выражает) вреда нет, но это фундамент схем для Т-3 — деньги без прогноза тогда
проскочат контракт. Лечится `if/then` (atom ∈ денежные → required
`estimated_net_profit`) или `allOf`-ветками.

### СРЕДНЯЯ — money-паттерн не фиксирует точность (вопреки §10 «фиксированной точности»)

Паттерн `^-?[0-9]+\.[0-9]+$` требует точку и хотя бы одну цифру с каждой стороны,
но НЕ фиксирует число знаков после неё. `"1.5"` и `"1.50"` оба валидны — это
разные строки → разные канонические байты → разный хэш для арифметически равных
сумм. §10 говорит «строки фиксированной точности». Канонизатор money-строки
намеренно НЕ нормализует (трактует как непрозрачные), значит точность обязана
фиксировать схема — а она не фиксирует. Удар по воспроизводимости денежных хэшей,
когда два поставщика напишут сумму с разным числом нулей.

### МЕЛОЧЬ — `lookupKey` берёт первый дубль, `json.loads` — последний

Следствие той же дыры дублей: на `{"critical_ok":true,"critical_ok":false}`
Haskell-`lookupKey` читает ПЕРВЫЙ (`true`→full), Python читал бы ПОСЛЕДНИЙ
(`false`→observe_only). Сегодня деньги не текут (логика ворот живёт только в
Haskell, заглушка всё равно молчит), но это латентный «раскол мозга» на одном
байтовом входе. Уйдёт вместе с запретом дублей.

### МЕЛОЧЬ — `decision_output` допускает денежные атомы при `observe_only`

Схема не запрещает `TakeProfit` при `data_rights=observe_only`. Worker раскрыл
это осознанно (HANDOFF риск №3): запрет — инвариант ворот (Gate.hs), не схемы.
Защита проверяется в Gate и тестах. Принимаю как обоснованный выбор роли схемы;
фиксирую, что ревью это видело. Если Admin хочет жёстче — `if/then` в схеме.
