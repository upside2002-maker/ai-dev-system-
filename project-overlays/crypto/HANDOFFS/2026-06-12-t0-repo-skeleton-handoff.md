## Отклонения от ТЗ и почему

1. **Имя Python-пакета — `cpds_platform`, а не `platform`.** Каталог
   `platform/` как требует ТЗ, но импортируемый пакет внутри переименован: имя
   `platform` затеняет стандартный модуль `platform`, который тянет зависимость
   `jsonschema` (через `attrs`) — без переименования валидация падала на импорте.
   Каталог и роль не изменились, только внутреннее имя пакета.
2. **Добавлена 8-я схема `decision_input.schema.json`.** ТЗ перечисляет 7 схем;
   вход ядра (§9.1) — композиция снимка + 4 журналов + профиля, нужен для CLI и
   golden-образцов DecisionInput. Это не подмена 7 основных, а вспомогательная
   схема-композиция. Все 7 на месте.
3. **SHA-256 и JSON в Haskell написаны вручную (свои модули).** Не aeson/crypton:
   ради офлайн-сборки `make check` с чистого клона ядро зависит только от пакетов
   из поставки GHC. Это убирает сетевой шаг и хрупкость 60-дневного индекса
   hackage при сборке. Корректность SHA-256 закреплена тестами на известных
   векторах FIPS 180-4.
4. **Сверх ТЗ: цель `cli-accept`** в `make check` (структурная валидация выхода
   заглушки по схеме + проверка observe_only/без-денег). ТЗ требует это как

## Доработка №2 (суррогатные пары) — контрольное ревью №2 (вердикт REJECT → закрыто)

Контрольное ревью №2 (`HANDOFFS/2026-06-12-t0-repo-skeleton-review2.md`)
подтвердило закрытие всех прежних находок, но вскрыло **НОВУЮ критическую** того
же класса, что и дубли ключей: расхождение кросс-языкового хэша на ВАЛИДНОМ входе.

### Суть дыры (доказана ревью байт-в-байт)

`pString` в `core/src/Json.hs` (ветка `\u`) разбирал каждый `\uXXXX` отдельно и НЕ
склеивал экранированную суррогатную пару `𐀀` в одну кодовую точку. Ключ
вне базовой плоскости, поданный парой, хранился двумя сырыми код-юнитами и
сортировался по `0xD800`, а Python (`json.loads`) — по склеенной точке `0x10000`.
При СОСЕДНЕМ BMP-ключе из верхней части базовой плоскости `[U+E000..U+FFFF]`
порядок ключей разъезжался → разный SHA-256 из двух языков на одном логическом
входе. Нарушение §10/§9.5 №5 («один вход → один хэш»), фундамент денежного учёта.
Воспроизведено до фикса: `{"￿":1,"𐀀":2}` → Haskell `5f5e6c84…` ≠
Python `cc317afd…`.

### Что сделано

1. **Склейка пар в `pString`** (`core/src/Json.hs`): после чтения `\uXXXX`, если
   это высокий суррогат `0xD800–0xDBFF`, а следом сразу `\uXXXX` низкого
   `0xDC00–0xDFFF`, склеиваю в одну кодовую точку
   `((hi-0xD800)·0x400)+(lo-0xDC00)+0x10000` — ровно как `json.loads`. Помощники
   `isHighSurrogate`/`isLowSurrogate`/`tryLowSurrogate`/`combineSurrogates`. Точка
   ≥ U+10000 в каноне снова выводится парой (`escapeString` уже это делал —
   расхождения в выводе не было, только в парсинге/сортировке). `pString` один на
   ключи и значения — фикс покрывает оба места.

2. **Краевые случаи выровнены с Python (сперва проверил фактическое поведение
   Python в репо, потом зеркалил)**. `json.loads` НЕ отвергает одиночные суррогаты
   и НЕ склеивает их — хранит сырыми код-юнитами. Зеркалю точно:
   - одиночный высокий `\uD800` без пары (в т.ч. за ним ASCII / другой escape /
     ещё один высокий) — остаётся сырым `0xD800`, сортируется ниже `U+E000`;
   - одиночный низкий `\uDC00` — сырой `0xDC00`;
   - пара в неверном порядке `\uDC00\uD800` (низкий перед высоким) — НЕ пара, две
     отдельные точки (склейка только high→low);
   - пара в ЗНАЧЕНИИ строки (не только в ключе) — склеивается так же;
   - смесь: литеральный UTF-8 non-BMP символ и его `\u`-эскейп-форма в одном
     документе → один хэш в обоих языках.

   **Решение по одиночным суррогатам: ПРИНИМАЕМ оба** (не отвергаем) — так делает
   Python, а правило контракта «класс валидного входа совпадает в обоих языках».
   Отвергать их в Haskell означало бы разойтись с Python в обратную сторону.

3. **Злой корпус `golden/evil/` дополнен** (24 валидных, было 19; 17 невалидных):
   - `surrogate_key_pair_vs_bmp.json` — ровно кейс ревью (non-BMP ключ `\u`-парой +
     соседний `U+FFFF`); до фикса разъезжался, теперь кросс-хэш совпал;
   - `surrogate_pair_in_value.json` — пара в значении;
   - `surrogate_mix_literal_escape.json` — литеральный non-BMP ключ + `\u`-форма в
     значении (один документ);
   - `lone_low_surrogate.json`, `surrogate_reversed_pair.json` — одиночный низкий
     суррогат и пара в неверном порядке (оба ПРИНИМАЮТ оба языка, как Python).
   Суммы закреплены в `manifest.json`. `test_evil_corpus_covers_all_samples`
   гарантирует, что манифест покрывает все файлы на диске.

4. **`docs/CONTRACTS.md`**: раздел «Суррогатные пары и одиночные суррогаты» —
   доказательство дыры, формула склейки, и явно зафиксированная семантика
   одиночных суррогатов (ПРИНИМАЕМ — зеркало `json.loads`).
