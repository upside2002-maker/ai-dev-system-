## Отклонения от ТЗ и почему

- **Полная история с 2019 на диске уже была** (от предыдущего, оборванного
  Worker'а; артефакты `data/` с меткой 19:55). Поэтому прогон 1 `collect`
  показал `+0` (всё уже скачано). Чтобы доказать живой сетевой путь и реальную
  докачку хвоста, эта сессия сделала контрольный прогон 2: обрезала хвост
  BTC-свечей на 5 точек и убедилась, что `collect` докачал ровно `+5`. Это
  честная демонстрация идемпотентности; полная история (2019-01-01 …
  2026-06-13, 2721 день) присутствует и подтверждена чтением Parquet.
- **Farside в Т-1 не отдаёт данные по HTTP** (анти-бот / Cloudflare → 403 на
  простой запрос). Это ровно тот случай, ради которого §3.1 объявила Farside
  `access: browser`, а браузерное чтение отнесла к Т-2. Коллектор честно
  деградирует «источник молчит» (None, поля нет), снимок собирается без потоков
  ETF. Разбор HTML-таблицы реализован и протестирован на фикстуре реальной
  вёрстки + анти-бот-заглушке + мусоре. На живом снимке `etf_flow` отсутствует —
  это деградация по дизайну, а не сбой.
- **Золото (FRED) в снимок не попало** — в окружении нет `FRED_API_KEY`
  (ключи в репо запрещены, платных/секретных не заводили). `fetch_gold()` →
  None без ключа, поле `gold` отсутствует. Это в рамках ТЗ (золото опционально,
  ключ только из окружения).
- **PAXG/токен-золото вынесено из активов в вспомогательное сырьё** —
  FRED даёт спот-цену золота, не токен; активы снимка — BTC/ETH. В рамках ТЗ.
- **Семьи скоров считаются по ведущему активу (BTC).** Оценка состояния рынка
  в целом; в v1 — по BTC. `assets` наполняются по каждому. Калибровка/корзина —
  Г-0 (Т-7), сюда не тащили (ТЗ).
- **Сеть стандартной библиотекой (`urllib`), не `requests`** — чтобы не тянуть
  лишний сетевой пакет в продукт.
- **`timeout(1)` на macOS отсутствует** — для живых прогонов команды запускались
  без внешнего таймаута (у HTTP-клиента свои таймауты); на результат не влияет.

## Риски для ревью (куда смотреть ломателю)

1. **Хрупкость сетевых сборщиков.** `fetch_*` ходят в публичные точки без
   ключей; вёрстка/лимиты источников могут меняться. Проверить, что падение
   ЛЮБОГО источника на середине `collect`/`snapshot` не валит процесс (в `cli.py`
   каждый источник в `try/except`, печатает строку и идёт дальше) и не портит уже
   сохранённое (Parquet пишется заменой файла целиком). Farside уже сейчас
   молчит (403) — это эталон деградации, не баг.
2. **Кросс-чек цен.** Порог `price_cross_check_max_rel_diff` (1%) в конфиге.
   Внимание: если Coinbase МОЛЧИТ, кросс-чек не проводится (отсутствие
   подтверждения, не `stale`). Проверить, что молчание Coinbase не маскирует
   реальное расхождение, когда цена ОТДАНА. Тесты на фикстурах: +5% (ловит),
   +0.1% (молчит).
3. **Канонизация снимка — главный фундамент §10.** `snapshot_hash` считается по
   документу БЕЗ `meta.snapshot_hash`; кросс-хэш файла Python (`hash_cli`) ==
   Haskell (`cpds-core hash`) подтверждён на ЖИВОМ снимке (`d32042…`). Стоит
   ткнуть снимок с не-ASCII в источниках/сырье и граничные `numfmt.fixed8`
   (очень большие mcap `316387048346.60790000`, отрицательные ema_dev/ath_drawdown,
   ноль) — Money-паттерн схемы строгий, `numfmt` обязан попадать ровно.
4. **Идемпотентность Parquet.** `store.append_rows` дедуплицирует по ключу
   (свечи — `open_time` мс, фандинг — `funding_time` мс). Ткнуть: повтор
   пересекающегося диапазона, пустой вход, докачка хвоста после ручного
   обрезания (эта сессия проверила обрезание на 5 точек → `+5`).
5. **Зоны по квантилям.** Зона = перцентиль текущей метрики в собственной
   истории + пороги конфига. Проверить направление (высокий фандинг/RSI/растяжка
   → горячо; низкий → капитуляция) и поведение на короткой истории (уверенность
   падает). Пороги — стартовое приближение, калибровка Г-0 (Т-7), не ворота этого
   TASK.
6. **Сеть в тестах.** Тесты гоняют только `parse_*` на фикстурах. Убедиться, что
   ни один тест не зовёт `fetch_*`/`urlopen` (грепом и блокировкой сокетов).
   Особый случай — `test_fred_no_key_means_silent`/`monkeypatch.delenv`: даже
   если у ревьюера задан `FRED_API_KEY`, тест его удаляет.
7. **Многоагентный контекст.** Эту сдачу делала сессия финализации после обрыва;
   код предыдущего Worker'а не переписан. Зелёный `make check` + три живых
   прогона подтверждены этой сессией заново (не пересказ).
