# Ревью Т-4a (виртуальный исполнитель) — сведённый вердикт Admin'а

**Дата:** 2026-06-13
**Метод:** многолинзовое адверсариальное ревью (4 линзы + верификация), tier B.
**Сырой итог:** REJECT, 1 критическая.
**Вердикт Admin'а: ДОРАБОТКА с расширением периметра — дыра в КОНТРАКТЕ,
не только в исполнителе.**

## Находка (подтверждена, реальна)

Исполнитель читает `action['position_pct']` напрямую в TakeProfit и
BuyDipFromBuffer (executor.py:222, :528). Схема `decision_output` делает
`position_pct` опциональным → схемно-валидное решение без доли роняет
исполнитель с KeyError.

## Разбор Admin'а: где чинить

Проверил код ядра (Cpds.hs) — ядро НА ПРАКТИКЕ всегда кладёт `position_pct`
там, где исполнитель его читает:
- TakeProfit → position_pct = транш ✔
- CreateSaleLock → position_pct = транш ✔
- BuyDipFromBuffer → position_pct = buyback ✔
- DistributeProfit → position_pct = Nothing (правильно: у него
  distribution_plan, не доля позиции)
- Observe / ManualReview → нет

Значит на реальном конвейере (единственный источник решений — наше ядро)
падение НЕ стреляет. Но контракт нечестен: денежное действие без доли по
схеме валидно, хотя по смыслу неопределено. Исполнитель не должен
полагаться на «ядро добрый». **Чиним в корне — в контракте**, симметрично
уже существующей условной обязательности `estimated_net_profit`.

## Решение (доработка, расширенный периметр, риск поднят до A)

Три части в одной доработке:
1. **Схема** `decision_output.schema.json`: `position_pct` условно-обязателен
   для атомов, двигающих позицию/буфер — **TakeProfit, CreateSaleLock,
   BuyDipFromBuffer, MoveBufferToTerminal**. НЕ для DistributeProfit
   (у него distribution_plan) и НЕ для Observe/ManualReview. Множество
   ОТЛИЧАЕТСЯ от множества estimated_net_profit (там DistributeProfit есть,
   CreateSaleLock нет) — прописать аккуратно.
2. **Ядро**: после ужесточения схемы прогнать всё (make check) — выход ядра
   обязан остаться валиден (ядро уже кладёт position_pct у нужных атомов);
   golden decision_output обновить если потребуется; кросс-хэш и злой корпус
   целы. Логику ядра не менять.
3. **Исполнитель**: оборонительно — отсутствие position_pct у атома, где он
   обязан быть, → явная ошибка/отказ исполнения, НЕ KeyError (страховка
   поверх контракта).

Периметр расширен по решению архитектора (схема — принятый контракт Т-0):
риск поднят B→A, режим strict.
