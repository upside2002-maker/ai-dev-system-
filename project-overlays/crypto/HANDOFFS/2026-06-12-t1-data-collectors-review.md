# Ревью Т-1 (сборщики данных + снимок дня) — сведённый вердикт Admin'а

**Дата:** 2026-06-12 (ночь)
**Метод:** многолинзовое адверсариальное ревью (6 линз → верификация каждой
находки → свод), плюс ручная проверка Admin'а спорных мест.

**Вердикт Admin'а после разбора: ДОРАБОТКА (узкий список блокеров), не

## Почему не принимаю сырой вердикт как есть

Многоагентное ревью склонно раздувать серьёзность и дублировать. Разобрал
каждую «критическую»:

1. **buildOk=false у 4 линз — ЛОЖНОЕ срабатывание.** Проверил сам: честно
   чистый клон (`git clone` без `.venv`) + `make check` → зелёно, цель сама
   поднимает окружение. Падение у линз — их сетевое/окруженческое (установка
   зависимостей требует сеть; параллельные сборки), не дефект продукта.
   Acceptance #1 выполнен.
2. **Гонка при записи Parquet** — две линзы, РАЗНЫЕ вердикты об одном:
   одна «критическая», другая честно «MEDIUM, Т-1 однопоточный ручной, не
   блокер, риск к операционной фазе». Беру вторую: это операционный долг
   (Т-4+), не блокер каркаса.
3. **Потеря накопленного при обрыве пагинации** — помечено критическим, но
   верификатор сам пишет: на диск ничего не теряется, идемпотентная докачка
   восстанавливает. Реальный вред — молчаливый возврат 0 (бар-except).
   Это про наблюдаемость, не про потерю данных. Понижаю.
4. **Утечка ключа FRED в трейсбек** (2 находки = 1 дыра) — РЕАЛЬНО.
   `fred.fetch_gold()` не обёрнут как другие источники, а `http.py`
   включает полный URL (с `api_key`) в текст исключения → ключ в stderr/логи.
   Настоящий дефект, нарушает «нет ключей». БЛОКЕР.

## Блокеры Т-1 (узко, чинить до приёмки)

- **Б1. Ключ FRED не должен утекать.** Обернуть `fred.fetch_gold()` в
  try/except как остальные источники; в `http.py` НЕ включать query-строку
  (или маскировать значения параметров) в текст `FetchError`. Тест:
  смоделировать сбой FRED при заданном ключе → ключа нет ни в выводе, ни
  в исключении.
- **Б2. Наблюдаемость деградации единообразна.** Любой источник при сбое
  (включая defillama, farside, обрыв пагинации binance/coinbase) печатает
  строку «источник X недоступен: …» и помечается в `data_quality`
  (missing/stale) — ровно как обещал сам HANDOFF. Молчаливый бар-except
  и молчаливый возврат 0 запрещены. Тест на каждый источник.
- **Б3. Свежесть данных (max_staleness_days) — подключить или убрать.**
  Параметр объявлен в конфиге, но не используется; архитектура §3.3
  требует «протухло → stale». Подключить минимально: возраст последней
  точки больше порога → метка stale для семьи. Тест на просроченные данные.

## Операционный долг → файл известных ограничений, НЕ блокер Т-1

Записан в `project-overlays/crypto/KNOWN_LIMITATIONS.md`, привязан к этапу,
где станет реальным (Т-4 «бумажный полигон» / операционная непрерывность):

- ретраи с паузой для временных сбоев сети (сейчас одна попытка);
- защита от гонки при параллельной записи Parquet (замок/temp+rename) —
  Т-1 однопоточный ручной, гонки в текущем пути нет;
- атомарная запись снимка (temp+rename) против обрыва между двумя файлами;
- эволюция схемы в store: сейчас молча роняет новые колонки → хотя бы
  громкая ошибка;
- частичное сохранение скачанного при обрыве пагинации (идемпотентность
  уже спасает целостность; вопрос эффективности).

Обоснование разделения: ТЗ Т-1 — «ручной запуск, сеть только при ручном»;
непрерывный автоматический сбор (где эти риски реальны) — отдельная частота
контура (архитектура §2), приходит в операционную фазу. Раздувать каркас
данных до промышленного сборщика сейчас — дрейф.
