## Summary

Сделан модуль `core/src/Signal.hs` — чистая детерминированная функция
`classifySignal :: Families -> DataQuality -> ProfileConfig -> SignalState`.
По зонам пяти семей + флагу `critical_ok` + профилю (веса семей, порог
перегрева) даёт `SignalState`: `OverheatStrong | OverheatMild | Capitulation
| Neutral | Unconfirmed`. Маппинг зон в вклад — строго по таблице ТЗ. Вся
арифметика на ЦЕЛЫХ из строк фиксированной точности (×10^8, тип `Integer`);
Double/Float нет ни в одном виде. Денег, журналов, ворот, изменения выхода
CLI здесь нет — только классификация-фундамент (Т-3b/c/d отдельно).

Модуль самодостаточен: `Types.hs` НЕ тронут (Signal.hs определяет свои `Zone`,
`Families`, `DataQuality`, `ProfileConfig`). `Gate.hs`, `CpdsStub.hs`,
`Canonical.hs`, `Json.hs`, `Sha256.hs`, `schemas/`, `platform/` — не тронуты.
Выход `cpds-core` прежний (всё ещё Observe-заглушка): подключение
классификации к выходу — Т-3d, не здесь.

## Сделано (по каждому инварианту — чем закрыт)

Тесты: `core/test/SignalSpec.hs`, отдельный test-suite `cpds-signal-test`
(на одном `base`, как и `Spec.hs` — офлайн-сборка без QuickCheck). Property
проверяются ПЕРЕБОРОМ по конечному пространству зон (6^5 = 7776 снимков) ×
набор профилей (стартовые §9.4 + детерминированный LCG-генератор весов/порога)
+ табличные граничные кейсы. Всего 44 проверки, все зелёные.

- **Инвариант 1 (критичность).** `critical_ok=false` → первый гард
  `classifySignal` отдаёт `Unconfirmed`, перебивая всё. Тесты: перебор всех
  7776 снимков × 43 профиля при `DataQuality False` → всегда `Unconfirmed`;
  отдельно — никогда не `Overheat*`/`Capitulation`.
- **Инвариант 2 (детерминизм).** Функция чистая и тотальная — равный вход даёт
  равный выход по построению. Тест: перебор снимков × оба значения critical_ok
  × профили, два вызова подряд равны.
- **Инвариант 3 (источник параметров).** Порог берётся только из
  `pcOverheatThreshold` профиля, веса — только из `pcWeights`, зоны — только из
  `Families`. Зашитых числовых порогов в коде нет (единственные константы —
  масштаб 10^8, множители зон из таблицы ТЗ, множитель «сильного» 3/2). Тесты:
  смена порога на том же снимке меняет вердикт (`OverheatStrong` ↔ `Neutral`);
  смена весов меняет вердикт; нулевые веса → никогда не Overheat.
- **Инвариант 4 (монотонность).** Порядок конструктора `Zone` и множители зон
  монотонны по шкале neutral→warm→hot→extreme (и neutral→cheap→capitulation).
  Тесты: для каждой семьи, базового снимка и соседней пары шкалы — усиление
  зоны не уменьшает соответствующую взвешенную сумму И не переводит `Overheat*`
  в `Neutral`; зеркально для капитуляции.
- **Инвариант 5 (взаимоисключение).** `classifySignal` возвращает РОВНО одно
  состояние (тип не выражает «оба сразу»). Конфликт (обе суммы ≥ порога,
  достижим только на искусственных весах — на зеркальных зонах реального снимка
  одна сумма всегда 0) разрешается в `Neutral`. Тесты: перебор × профили — на
  конфликте всегда `Neutral`, не Overheat/Cap; искусственный конфликт
  (extreme+capitulation вперемешку, крошечный порог) → `Neutral`.
- **Инвариант 6 (строгий порог).** Сравнение нестрогое `сумма ≥ порог`:
  «строго меньше порога → НЕ Overheat». Граничные табличные кейсы: сумма =
  порог−ε (×10^-8) → Overheat; сумма = порог → Overheat; сумма = порог+ε →
  Neutral; зеркально для капитуляции. Плюс property: на любом входе
  `перегревная сумма < порога ⇒ не Overheat*`.
- **Целые из строк фикс. точности.** `parseFixed8` разбирает "ddd.dddddddd"
  посимвольно в `Integer` ×10^8 (не через Double). В тесте веса/порог всегда
  проходят через `parseFixed8 . renderFixed8` — тот же путь, что из реального
  профиля. grep: реального использования типов `Double`/`Float` в Signal.hs
  нет (только упоминания запрета в комментариях).

## Выбранный критерий «сильного перегрева» и обработка конфликта

**Сильный перегрев** (`isStrongOverheat`) — при подтверждённом перегреве
(сумма ≥ порог) силён, если выполнено ХОТЯ БЫ ОДНО:
- (а) хотя бы одна семья в зоне `extreme`, ИЛИ
- (б) перегревная сумма ≥ 1.5 × порог.

То есть взят критерий «extreme ИЛИ ≥1.5×порог» (оба варианта из ТЗ объединены
через ИЛИ — это самый осторожный вариант: сильным считаем при любом из двух
признаков перегрева с запасом). Множитель 1.5 выражен целочисленно как
`2·сумма ≥ 3·порог` — без деления и без float (деление теряло бы точность на
нечётном пороге). Граница 1.5× ровно → сильный (нестрогое ≥), покрыто
табличным тестом.

**Конфликт Overheat ∧ Capitulation.** Зафиксирован как `Neutral` (выбран
вариант «Neutral» из двух предложенных ТЗ — самый безопасный: при
противоречивом сигнале ядро ничего не утверждает). Отдельного конструктора-
конфликта не вводил — инвариант 5 держится тем, что `classifySignal` отдаёт
ровно одно состояние и конфликтная ветка идёт первой после `Unconfirmed`.
На реальном снимке зоны зеркальны (семья не может быть одновременно «горячей»
и «холодной»), поэтому одна из сумм всегда 0 и конфликт недостижим;
разрешение оставлено на случай произвольных весов/входов property-теста.

## Отклонения

- Тесты на `base` без QuickCheck (как и существующий `Spec.hs`): «property»
  реализованы как ПОЛНЫЙ перебор конечного пространства зон + варьирование
  весов/порога детерминированным LCG. Для зон это исчерпывающе (сильнее
  случайного QuickCheck); для весов/порога — широкая детерминированная выборка.
  Это сознательный выбор ради офлайн-сборки проекта, не упрощение охвата.
- Отдельный test-suite `cpds-signal-test` вместо расширения `Spec.hs` — чтобы
  не трогать золотые хэш-тесты ядра и держать сигнальные тесты изолированно.
- Профиль в `Signal.hs` (`ProfileConfig`) узкий: только `family_weights` и
  `overheat_threshold` (то, что нужно классификации). Остальные поля схемы
  профиля (транши, корзины, распределение) сюда не тащил — это Т-3b/d. Парсинг
  JSON-профиля целиком в этот тип — тоже не здесь (классификатор принимает уже
  разобранный `ProfileConfig`; подключение к снимку/профилю JSON — Т-3d).

## Риски для ревью (куда смотреть ломателю)

1. **Граница порога (инв.6).** Сравнение нестрогое `сумма ≥ порог`. Проверить
   намерение: «ровно на пороге» = перегрев (так в ТЗ: «строго меньше → не
   Overheat»). Капитуляция использует ТОТ ЖЕ порог — подтвердить, что это
   соответствует «зеркально по тому же порогу» из ТЗ.
2. **Критерий «сильного» = extreme ИЛИ ≥1.5×порог.** Если ревью считает, что
   ТЗ требует строго ОДИН из двух (а не объединение) — это место правки.
   Целочисленное `2·сумма ≥ 3·порог` для 1.5× — проверить, что округления нет
   (его и нет: чистые целые).
3. **Масштабы.** Веса/порог — 10^8; произведение вес×множитель — 10^16;
   взвешенная сумма и поднятый порог сравниваются в 10^16 (`thresholdE16 =
   порог × 10^8`). Перепроверить, что суммы и порог точно в одном масштабе
   (тест «valuation=extreme + price=hot = 2.0» фиксирует ожидаемую величину
   в 10^16).
4. **Целочисленность без переполнения.** Тип `Integer` (произвольная точность),
   не `Int` — переполнения нет по построению даже на абсурдных весах. Если
   ревью предпочитает `Int64` с явной проверкой границ — обсудить (сейчас
   намеренно `Integer` ради безопасности денежного сердца).
5. **Множители зон — контракт, не настройка.** `overheatMultiplier` /
   `capitulationMultiplier` зашиты по таблице ТЗ (extreme=1.0/hot=0.5/warm=0.25;
   capitulation=1.0/cheap=0.5; остальное 0). Это намеренно НЕ в профиле — если
   ревью хочет их в конфиг, это смена контракта (не в этой задаче).
6. **Конфликт → Neutral.** На реальном зеркальном снимке недостижим; проверить,
   что разрешение в Neutral (а не отдельный ManualReview-конструктор) приемлемо
   для Т-3d, который будет читать `SignalState`.
