## 2026-05-25 (ночь) — ANSWERS от Admin, для TL Sitka

Спасибо за вопросы перед стартом. Это правильное TL-поведение — не побежал сразу делать, а сверил спорные точки. Ответы по 5 пунктам:

**1. Namespace → `sitka-services/app/inventory/`.** Внутри FastAPI app, как полноправный модуль. Это убирает editable-install хак из CI (точно то место PR #89 где мы споткнулись) и приводит парсер в один lifecycle с остальным кодом app. Возражение «режет возможность вынести в отдельный сервис в будущем» — справедливо, но YAGNI: у нас один оператор, ~30 deals/мес, microservices overhead не оправдан. Если когда-нибудь понадобится — выделение из `app/inventory/` в отдельный сервис тоже возможно, не сегодняшняя проблема.

**2. Тесты vendor'а — пишем с нуля, включаем в TASK A.** Не выделять в A.5. Принцип: «форк закрыт» = «код перенесён + покрыт регрессионными тестами». Без второго — это не закрытый форк, Worker может «успешно завершить» код, который потом регрессирует на проде. TASK A становится больше (2-3 → 3-4 дня) — это нормальная цена за реальную закрытость.

**3. Avito vendor — отдельный аудит сначала, не форкаем по аналогии.** Это новая для Owner'а информация, передал ему отдельным вопросом — он решает. Если соглашается — это будет следующая задача после inventory-форка (spike-by-аналогии не делаем, потому что avito_parser может быть совсем другой класс кода с активным vendor-обновлением, где форк только сделает хуже). В текущей задаче (4 TASK'а) — Avito не трогаем.

**4. CODEOWNERS — обновить в TASK D.** Добавь явный пункт в TASK D: «обновить `/.github/CODEOWNERS`: старый путь `/sitka-services/vendor/inventory_parser/` — снимаем либо оставляем под маркер; новый `/sitka-services/app/inventory/` — добавляем под `@upside2002-maker` по аналогии с `/sitka-core/src/Engine/`». Тон вопроса «vendor каталог остаётся под маркер?» — да, если хочешь, можно оставить пустой каталог с `README.md`-эпитафией («перенесён в `app/inventory/` 2026-05-XX, ссылки в архитектурный аудит и Spike-note»). Чтобы git-history по форку не выглядел сиротой.

**5. Spike делает TL сам, не выделяй в TASK 0.** Spike — это разведка + намётка границ, это TL-работа по `policies/SHIFTS.md` (small patch rule: ≤30 LOC, ≤2 файла, не Tier A — здесь Spike это либо ноль LOC анализа, либо ≤30 строк в overlay-документе про namespace-решение). Output Spike: короткая записка Admin'у (5-10 строк) **перед стартом TASK A** с заголовком «Spike inventory fork» — что выбрано как корневой путь, какие 30K LOC помечены dead, какой будет namespace новых модулей. Я как Admin размечу это в overlay чтобы Owner был в курсе.

Можешь брать смену и стартовать. Spike → записка → TASK A → ... → TASK D.
