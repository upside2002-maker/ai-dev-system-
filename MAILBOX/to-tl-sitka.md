## 2026-05-21 — от Admin, для TL Sitka

**Что:** провести архитектурный аудит парсера `sitka-services/vendor/inventory_parser/`. Код написан сторонним разработчиком и был vendored в наш репо PR #67 (2026-04-27). Внимательно прочитать весь код, понять как он вписан в нашу архитектуру, найти несоответствия инвариантам и потенциальные проблемы. Это не «правка одного бага», а полноценный архитектурный обзор.

**Почему:** парсер — критичный integration layer (4 поставщика: Rogers, 1shot и др.), оператор работает с ним каждый день, на нём построен живой business flow «поиск → quote → лид». Поведенческий баг уже всплывал (двусловные запросы `Blizzard parka` 2026-05-13). При следующем vendor-обновлении или добавлении нового магазина нам нужно глубокое понимание архитектуры — иначе будем менять «вслепую». Если есть фундаментальные проблемы — лучше выявить сейчас, до того как они проявятся в проде.

**Срочность:** не горит, но и не откладывай больше чем на неделю. Делай параллельно со своим основным backlog'ом — не вместо.

**Что вернуть в `to-admin.md`:** один markdown-документ с архитектурным аудитом, путь — `project-overlays/sitka-office/research/inventory-parser-architecture-audit-2026-05-21.md` (создай каталог `research/` если его нет). Структура:

1. **Краткое резюме (5-7 строк):** зелёный / жёлтый / красный + 3 главных вывода.
2. **Карта парсера:** какие модули за что отвечают, точки входа, точки выхода, как парсер взаимодействует с `core` и `services`.
3. **Соответствие нашей архитектуре:**
   - Correction 001 (domain decisions в integration layer) — нет ли утечки бизнес-логики в парсер?
   - граница integration vs domain;
   - инварианты из `/Users/ilya/Projects/sitka-office/.claude/architecture-invariants.md`.
4. **Потенциальные проблемы — по категориям:**
   - производительность (timeouts, sequential vs parallel запросы, retry-логика, кэширование);
   - надёжность (что если поставщик меняет формат / API мёртв / network flap);
   - безопасность (как обращаются с credentials, потенциальная injection в построении query);
   - поддерживаемость (как добавить новый магазин — насколько адаптеры изолированы, нет ли дублирования);
   - тесты (что покрыто, чего не хватает, fixture-based vs real network).
5. **Конкретные находки** с координатами (файл:строка), не общие фразы.
6. **Приоритизированный backlog:** P0 / P1 / P2 / Nice-to-have.
7. **Рекомендация по работе дальше:** оставлять как vendored, fork'нуть и владеть, переписать частично, или ничего не трогать пока работает.

**Глубина:** вдумчивый прочёт всего кода в `sitka-services/vendor/inventory_parser/` (включая `inventory/v2/`, `adapters/`, `tests/` если они там есть), а не только README или верхушка одного файла.
