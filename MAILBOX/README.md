# MAILBOX — почтовые ящики между постоянными ролями

Модель работы — `policies/SESSIONS.md`. Живой dashboard ролей —
`DISPATCHER.md`.

## Файлы

- `to-admin.md` — TL Sitka и TL Astro пишут сюда отчёты, эскалации,
  вопросы для Admin'а / Owner'а.
- `to-tl-sitka.md` — Admin пишет сюда задачи для TL Sitka.
- `to-tl-astro.md` — Admin пишет сюда задачи для TL Astro.
- `archive/` — обработанные сообщения переезжают сюда в файл
  `<YYYY-MM>.md` той ролью которая обработала.

## Формат записки

Append-only. Каждое сообщение — отдельный блок, отделённый `---`:

```markdown
---

## 2026-05-20 04:30 — от Admin, для TL Sitka

**Что:** запусти задачу про парсер двусловных запросов.

**Почему:** оператор поймал баг 2026-05-13 (`Blizzard parka` → 0
hits), в backlog зафиксировано как Tier C ~1-3 LOC. Текущий
приоритет среди мелких хвостов — самый высокий по DX.

**Срочность:** не горит. Делать когда дойдут руки.

**Что вернуть в `to-admin.md`:** PR-номер + что было сделано + ссылка
на тест который ловит регрессию.

---
```

Поля:
- **Заголовок** — `## <дата+время> — от <автор>, для <адресат>`.
- **Что** — одна-две фразы по существу.
- **Почему** — короткий контекст (откуда взялась задача).
- **Срочность** — горит / не горит / fix-к-дате.
- **Что вернуть** — ожидаемая форма отчёта в ответном ящике
  (опционально).

Полей не больше пяти. Если задача требует больше — это уже полноценный
TASK в `project-overlays/<slug>/TASKS/`, а записка ссылается на него.

## Жизненный цикл записки

1. Автор append'ит в нужный `to-*.md`.
2. Адресат при `обнови контекст` читает свой ящик.
3. Когда задача в работе — адресат отвечает или просто берёт в
   работу, обновляет `DISPATCHER.md`.
4. Когда задача закрыта — адресат **перемещает свою запись** из
   живого `to-*.md` в `archive/<YYYY-MM>.md` (вырезать-вставить, не
   копировать). Это сигнал «обработано».

Удалять записки **нельзя** — только архивировать. Audit trail
остаётся в git.

## Что **не** идёт в MAILBOX

- Полноценные TASKs с lifecycle — это `project-overlays/<slug>/TASKS/`.
- Reviewer findings — это `HANDOFFS/`.
- Журнал событий — это `OPERATING/journal/`.
- Backlog мелочей — это `OPERATING/backlog.md`.

MAILBOX — это **только cross-role короткая переписка**: «возьми
задачу X / сделал X / эскалирую X на Owner'а». Долгоживущие
артефакты остаются на своих местах.

## Связь с DISPATCHER.md

После записки в почтовый ящик автор **также** обновляет
соответствующую строку в `DISPATCHER.md` — раздел «Активные роли»
(если статус роли изменился) или «Открытые блоки» (если задача
требует cross-role внимания).

`MAILBOX/` — это поток сообщений.
`DISPATCHER.md` — это срез текущего состояния.

Они дополняют друг друга, не дублируют.
